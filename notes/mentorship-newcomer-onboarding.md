# 規劃備忘：Mentorship — 新人帶領與認知設計（線十一）

**建立日期**：2026-05-27  
**狀態**：💡 策略規劃中  
**定位**：專注於「資深 QA 如何有尊嚴、有效率、且不崩潰地帶領新人與實習生」，從「認知科學」與「操作頻寬」的視角重構帶人心法。

---

## 🧠 核心主張與痛點

資深工程師在帶人時，最常遇到的挫折是：
*   「他怎麼教不會？」
*   「我明明說得很清楚了，他一動手就漏掉邊界條件？」
*   「我自己做只要 10 分鐘，教他做要花 1 小時，那我不如自己做。」

這條主題線的核心主張是：**「能做」與「能邊做邊教」是兩種完全不同的能力**。
我們不能依賴「在操作複雜功能的當下進行口頭教學」，因為複雜操作本身就已經吃滿了資深 QA 的**認知頻寬**。
真正的解決之道是**「認知空間的非同步重構」**——把教學時間移到「操作前後」，利用**文件防禦**、**觀摩影子**、以及**PDT 提問訓練**，在不榨乾自身頻寬的前提下，培育出具備獨立提問能力的 QA 新星。

## 🚀 規劃中文章地圖 (The Onboarding Cognitive Design Map)

---

### 第一篇：〈我懂了我爸做水電時為什麼沒辦法邊教邊做〉
*   **網址名稱 (`urlname`)**：`cant-teach-while-doing`
*   **主題定位**：共鳴性故事與自我覺察（目前在 `notes/`，需升級至 `_drafts/`）。
*   **SEO 長尾關鍵字**：`如何帶新人工程師`、`資深QA帶人`、`職場師徒制`
*   **核心內容大綱**：
    1.  **原始故事**：帶實習生測試極度複雜的影音直播時，我發現自己說話語無倫次、頻寬全滿，突然理解了小時候父親做水電時被我在旁邊問東問西時的不耐煩。
    2.  **分工型 vs. 協作型帶人**：
        *   分工型（給你個獨立 App 測，我事後 review）＝ 認知成本低，皆大歡喜。
        *   協作型（一起測複雜功能，邊操作邊講邊看他）＝ 認知頻寬當場過載。
    3.  **藍領「師傅模式」的古老智慧**：為什麼木匠、水電、廚師很少「邊做邊教」？因為真實的專業操作需要「全神貫注」。師傅模式的「你站著看，看完我才放手讓你做」其實是有大腦科學根據的。
    4.  **結語**：承認自己「無法邊做邊教」不是你的失敗，而是看清能力的邊界。

---

### 第二篇：〈PDT 帶人法：與其教新人「怎麼測」，不如教他「怎麼問」〉
*   **網址名稱 (`urlname`)**：`pdt-based-newcomer-mentoring`
*   **主題定位**：PDT 方法論在教學場景的具體運用（強化線一 PDT 招牌主張）。
*   **SEO 長尾關鍵字**：`QA新人培訓`、`軟體測試思維養成`、`問題驅動測試帶人`
*   **核心內容大綱**：
    1.  **前言**：教新人測試步驟（「點這裡，輸入123，看有沒有錯誤」），他學到的只是 procedural knowledge。系統一換，他又不會了。
    2.  **為什麼「提問」是唯一可以離開現場教學的武器**：操作步驟必須依附在特定畫面上，但「提問模型」可以在咖啡廳裡對著一張白紙聊。
    3.  **資深 QA 的「退位」提問藝術**：
        *   不要直接指出他的測試漏洞。
        *   當他交來一個測試計畫時，用 PDT 的 3 個黃金問題反問他：
            *   「如果這個金流 API 在第三步 timeout 了，系統現在的狀態是什麼？」
            *   「你怎麼知道你這次的 regression 範圍沒有漏掉 side effect？」
            *   「如果這個 bug 上線了，第一個會來敲門的用戶會是誰？」
    4.  **讓新人學會自己當自己的「提問教練」**：把 PDT 的問診類比移植到新人大腦中。
    5.  **結語**：當新人學會了精準提問，你就不再需要為他做保姆式的 review 了。

---

### 第三篇：〈文件是防禦，PDT 是進攻：如何為新人打造一套認知緩衝區？〉
*   **網址名稱 (`urlname`)**：`documentation-defense-and-pdt-offense`
*   **主題定位**：結合自動化工具與文檔，設計出一套不需要資深 QA 時時刻刻盯著的 Onboarding 系統。
*   **SEO 長尾關鍵字**：`QA Onboarding流程設計`、`測試規格文件維護`、`新進QA培訓計畫`
*   **核心內容大綱**：
    1.  **前言**：你是不是又在新人到職第一天，塞給他一本 50 頁、早已過期三年的「系統操作手冊」，然後跟他說「你先自己看」？
    2.  **文件防禦（Defense）**：
        *   建立「Living Document」（如 `specs.md` 或 API 最小生存集合）。
        *   文件不寫瑣碎步驟，只寫「系統邊界、核心相依性、容易踩的坑」。
        *   這本文件是資深 QA 的防禦武器——它用來回答 80% 的 Procedural 蠢問題，把新人的無效發問擋在門外。
    3.  **PDT 進攻（Offense）**：
        *   利用你開發的 **PDT Self-Review Agent**。
        *   要求新人寫好 bug report 或測試計畫後，**先丟給 AI Review Agent 跑三輪 PDT 自我問答**，把顯而易見的邊界漏洞自己補上。
        *   新人交給資深 QA 的，必須是「已經過 AI Review、只剩下資深業務邏輯漏洞」的半熟品。
    4.  **認知緩衝區的運作成效**：
        *   新人有安全的「AI 沙盒」練習發問，不怕問蠢問題丟臉。
        *   資深 QA 獲得極大的頻寬解放，專注在關鍵設計 review。
    5.  **結語**：好的導師（Mentor）不是提供答案的百科全書，而是設計學習環境的架構師。
