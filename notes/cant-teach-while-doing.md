# 備忘：分工我擅長，協作我不行——我懂了我爸做水電時為什麼沒辦法邊教邊做

**建立日期**：2026-05-20
**狀態**：💡 觀察記下，未來想寫時打開看。**不在 `_drafts/`**——還在生鮮觀察階段。
**起源**：2026-05-20 帶實習生測影音直播時撞到的牆，意外讓我懂了我爸。

---

## 觀察（原始版本，先保留我自己的話）

> 我可能只擅長分工，**願意給實習生一個獨立的 APP 自己測試**——這種情境我做得很好。
>
> 但跟他**一起測試影音直播的複雜功能**時，我**沒有能力邊講邊一起做**。
>
> 我開始懂我爸做水電時沒有辦法邊教邊做了。

---

## 我看到的結構（粗版）

這個觀察藏著一個對比：**兩種帶人模式，認知成本完全不同**。

| 模式 | 場景 | 我做得怎樣 | 為什麼 |
|---|---|---|---|
| **分工型** | 給他一個獨立 APP 自己測 | ✅ 做得來 | 各自跑各自的，我只需要事後 review |
| **協作型** | 跟他一起測複雜功能 | ❌ 做不來 | 操作本身已經吃滿我的認知頻寬，剩不下力氣解釋 |

**核心問題不是「我不會教」，是「**操作 + 解釋 + 監看實習生**」三件事**同時要做時，認知頻寬塞不下**。

---

## 為什麼這個觀察值得寫

### 1. 跨世代 + 跨領域共鳴強

「我懂了我爸做水電時為什麼沒辦法邊教邊做」——這句話自帶情緒重量。

讀者一讀到「水電」「我爸」——立刻有畫面（小時候在旁邊看大人工作、被叫去拿工具、覺得大人「為什麼不教我」）。
**這份「我以為很簡單的事其實不簡單」的領悟**，是這篇文章最有力的核心。

不是 QA 圈才有的問題——**所有需要「操作 + 同時帶人」的職業都有這個牆**：水電、外科醫生、廚師、現場 SA、後端 on-call。

### 2. 是個誠實的自我觀察，不是抱怨

說「我做不來」需要勇氣。**沒有罵實習生「悟性不夠」、沒有罵公司「沒給我訓練時間」、沒有把鍋甩給任何人**——而是承認**這是我這個位置上的能力邊界**。

這份姿態本身就是文章的底色。

### 3. 跟你既有的線會合

| 線 | 怎麼合流 |
|---|---|
| 線一（PDT） | PDT 提問需要的腦力 vs 操作執行的腦力——可能不能同時用 |
| 線六（職涯 / 大人學） | 帶人 / mentorship 的「真實能力曲線」——不是「夠資深就會帶人」這麼簡單 |
| 線七（信仰 / 工作觀） | 對藍領 / 上一代工作的「我懂了」——一份遲到的敬意 |

## 我隱約看到的論述方向（不是現在就要寫，但可以留著）

### 可能的核心主張

> **「能做」跟「能邊做邊教」是兩種不同的能力。資深不等於後者，只等於前者。**
>
> 我們的職場文化默認「資深就該帶新人」——但這個默認是錯的。**帶人是另一份工作，需要另一份頻寬，要付出另一種代價。**
>
> 我爸沒辦法邊做水電邊教我——不是他能力不足，是他在認真做事。**真正會做事的人，做事的時候是『沒辦法分心』的。**

### 可能的延伸

- 為什麼 QA 帶實習生時，**簡單功能（給你自己做）反而比複雜功能（跟你一起做）容易**？答案：簡單功能 = 分工，複雜功能 = 協作，後者要「操作 + 解釋 + 監看」三線並進
- 怎麼設計「能邊做邊教」的工作流？**把『操作』跟『解釋』錯開**：先帶他看我操作完整一輪、再讓他單獨重複一次、最後我們才一起做
- 「**師傅模式**」的智慧：藍領訓練很少「邊做邊教」——多半是「**你站旁邊看，看夠了我才放手讓你做**」。**這種延遲教學其實是有結構性原因的**

## 2026-05-20 補充：我打算採取的兩個應對

承認「我邊做邊教做不來」之後，如果只停在觀察，這篇文章只是感慨。**但如果接著問「那我打算怎麼做」，文章就有了行動力**。

我心裡浮現兩個策略：

### 策略一：補足 QA 文件
**處理「我教不來」的部分——可以文字化的 procedural knowledge**

- 把操作流程、邊界條件、容易踩的坑寫成文件
- 實習生**事後自己讀**，不依賴我「邊做邊講」
- 我下次再帶他做時，省下的口頭頻寬可以放在「**對著文件做提醒**」而不是「從零講解」

### 策略二：引導實習生學會 PDT 這套問問題的方式
**處理「我能教、且最值得教」的部分——思維模式**

- PDT 提問不依賴「正在操作」這個場景——可以離開電腦坐下來慢慢聊
- 它是 transferable 的：他學會了 PDT，自己做任何新功能都能用
- **這正好填補「文件補不到」的空缺**：文件能告訴你「該做什麼」，但 PDT 告訴你「該問什麼」

### 兩個策略合起來的結構意義

把「邊做邊教」這個本來做不到的事情，拆成三個可以分開做的事：

```
觀摩（看我操作一輪）
   +
讀文件（事後自己消化操作細節）
   +
PDT 訓練（離開操作場景，慢慢聊「該問什麼」）
   =
不需要「邊做邊教」也能訓練出能獨立工作的新人
```

**這是「師傅模式」的現代化版本**——藍領師傅靠「你站旁邊看」累積 tacit knowledge；知識工作者多了兩個工具（文件 + PDT 提問訓練），可以**把教學時機從「操作當下」移到「操作前後」**。

### 對 PDT 招牌主張的強化

寫這篇時，可以順便埋一句強化線一招牌主張的話：

> **「PDT 是新人最值得優先教的東西——不是因為它最重要，是因為它是少數可以離開操作場景單獨討論的測試思維。」**

這句話把 PDT 從「QA 自己的方法論」拓寬到「**最該被教學的測試思維**」——理由是其他東西（操作、邊界感、視覺判斷）大都依附在情境上不好離開，唯獨 PDT 提問可以單獨拿出來練習。

→ **發這篇文時，呼應 [PDT 招牌主張](/post/problem-driven-testing-intro/) 那篇形成一個自然連結點**。

## 文章框架（粗草，未來再細）

```
前言：那個下午的兩個場景
  → 上週：給實習生一個獨立 APP，他做得很好、我 review 也順
  → 這週：陪他測影音直播的複雜功能，我語無倫次、節奏全亂
  → 同一個我、同一個實習生——差別在哪？

一、兩種帶人模式：分工 vs 協作
  → 分工：你做你的、我做我的，事後 review
  → 協作：同一份工作裡，操作 + 解釋 + 監看三線並進
  → 認知頻寬不同

二、為什麼複雜功能特別吃這份頻寬？
  → 影音直播這類功能：狀態多、時序敏感、邊界複雜
  → 我自己測都已經要全神貫注——再加上「同步講解」就過載

三、我懂了我爸做水電時沒辦法邊教邊做
  → 跨世代敘事：小時候我以為他「不願意教」
  → 現在才懂：不是不願意，是專業操作本身就吃滿了他的腦力
  → 這份遲到的「我懂了」，是寫這篇的真正動機

四、那怎麼辦？把「教學」從「操作當下」移到「操作前後」
  → 觀摩：實習生站旁邊看我跑一輪（操作當下唯一能做的事）
  → 文件：把可文字化的部分寫成 QA 文件（事後讓他自己讀）
  → PDT 訓練：離開操作場景，慢慢聊「該問什麼」（這是最值得集中教的）
  → 三個合起來，等於「師傅模式」的現代化版本
  → ⭐ 順手埋一句：**PDT 是新人最值得優先教的東西，因為它是少數可以離開操作場景單獨討論的測試思維** → 自然連結到 [PDT 招牌主張](/post/problem-driven-testing-intro/)

結語：資深 ≠ 會帶人
  → 我們的職場文化默認「資深就該帶新人」
  → 但這個默認是錯的——帶人是另一份工作，需要另一份頻寬
  → 給主管：別預設資深的人天然會帶人
  → 給自己：誠實承認「邊做邊教」做不來，**不是失敗，是看清楚邊界**
```

## 紅線提醒

- ❌ 不要寫成「實習生悟性不夠」——主角是我自己的能力邊界，不是他
- ❌ 不要過度浪漫化藍領工作——「我爸做水電」的部分要寫得樸實，避免知識分子的鄉愁感
- ❌ 不要把「資深≠會帶人」寫成主管罵——這是給整個職場文化的觀察，不是 call out
- ❌ 不要把這篇寫得太長——這個觀察的力量在「短而有畫面」。**控制在 2500 字以內**比較好
- ❌ 不要硬塞 PDT 框架——這篇是另一個維度的洞察，跟 PDT 是平行不是從屬

## 一個值得珍惜的小細節（給未來的自己）

寫的時候，**請保留「我懂了」這三個字**。

不是「我發現」（太分析）、不是「我意識到」（太正式）——是「**我懂了**」。

這三個字裡有遲到的歉意、有對父親的敬、有對自己過去傲慢的承認。文章的溫度全靠這三個字撐起來。
