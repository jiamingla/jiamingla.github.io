# 面對「做不完」：從對方角度看為什麼——回歸 vs 冒煙之爭與多產品線驗證（骨架）

**建立**：2026-06-17
**狀態**：🌱 骨架——核心 thesis 已成形、待真實素材沉澱
**起源**：6/17 user surface——工作煩悶（不到焦慮）+ 自覺「**我沒有以對方角度去思考為什麼這麼做比較好**」（上頭壓回歸、user 一直覺得功能+冒煙就好）。延伸到多產品線 bug 驗證（修一個 APP、怎麼驗其他 20 個也好了）
**主題線**：線六（職涯 / 大人學）+ 線五（自動化決策）+ 線三（多 APP 技術）+ 線一（驗證哲學）

## 核心 thesis（這篇的閉環）

> **「做不完」的煩悶，一大半根源不是事情多，是「我覺得不該這樣做、卻被迫做」。一旦能 steelman 對方為什麼這樣要求，就從「被迫」變成「我理解這個 trade-off、我選擇怎麼配置」。**

閉環：
```
做不完的煩悶
  → 真實 stakes：20 個 APP × 回歸 = 驗證成本爆炸
  → 這正是「回歸 vs 冒煙」之爭的核心：回歸可規模化（自動化跑 20 app）、
     冒煙不可規模化（靠資深判斷）
  → 我一直推冒煙、卻沒先承認上頭的本金（交代/免責/規模化/AI 政治）
  → 解不是「誰對」，是「配置」——而 Head 昨天的 4 點就是配置框架
```

## 第一層：我（QA）為什麼覺得功能 + 冒煙就好

- 回歸大多是「機器擅長的確定性驗證」、但很多是低價值重複
- 冒煙 + 功能能更快抓到真問題、把資深判斷用在刀口
- 連 [brand-value 提案](brand-value-and-smoke-test-proposal.md)：AI 產出上千結果沒人讀得完、要一組「高信賴度基準線」（冒煙就是）
- 連 [not-just-make-manual-script-automate](/post/not-just-make-manual-script-automate/)：手動回歸直接自動化是浪費、該分工

→ **我的論點沒錯。問題是我直接推它、沒先 grant 對方的本金。**

## 第二層：上頭為什麼想壓回歸（我一直略過的本金）

| 上頭的本金 | 為什麼回歸對他比冒煙好 |
|---|---|
| **要向上交代** | 回歸數字可量化——「跑了 N 條」。冒煙+功能難交代「做了什麼」（連 [qa-report-workplace-wisdom](/post/qa-report-workplace-wisdom/)：老闆要心安） |
| **AI 時代政治** | 想用「AI 跑大量回歸」回應上層對 AI 的期待；回歸是 AI 自動化最好 demo 的 surface |
| **防禦免責** | 全回歸跑過 = 「我們都測了」。只冒煙、出事就是「為什麼沒測到」的鍋。回歸是 CYA |
| **可規模化** | 回歸案例化、能丟給實習生 / AI 跑；冒煙+探索靠資深判斷、**不可規模化** |

→ 連 [5/27 insights 面向 16「對方建議的本金前提」](coach-sessions/2026-05-27-insights.md)：上頭站在「交代+免責+規模化」的本金、我站在「效率+抓真問題」的本金——**不同本金、不同最優解**。我的 gap 是沒去想他的本金。

### 6/17 user 自己 steelman 出的更精準三本金（為什麼是「現在」）

| 拆出來的本金 | 含義 |
|---|---|
| **規模化是上頭此刻最想看到的** | 不是上頭愛回歸，是「**規模化**」本身是他要向上交代的 KPI；AI 跑回歸正好是規模化最好 demo 的形式 |
| **「給 AI 做手動測試」是被認同的方向** | 不是技術最優解之爭，是**政治上已 align 的方向**——逆它 = 逆整個組織共識 |
| **碰巧來了願意燒 token 的工程總監** | **時機 / 預算窗口現在開著**——總監願意燒 = 資源約束暫時解除 =「不趁現在做更待何時」 |

→ 三個一拼就回答了「為什麼是現在」：**不是「回歸 vs 冒煙誰對」，是「規模化是 KPI + 方向已 align + 預算窗口開著」——三者加起來，上頭壓回歸是完全理性的。** 這就是 user 6/17 把「從對方角度看為什麼」這個動作，**自己用在自己身上**的實證（呼應自覺的那句「我沒以對方角度想為什麼」）。

## 第三層：多產品線把這個 trade-off 放大（user 6/17 延伸）

**「修好一個 APP 的 bug，怎麼驗證其他 20 個 APP 也好了？」**

- 20 個 white-label / 多品牌 APP 共用底層 → 一個 bug 修復，理論上 20 個都受影響
- **驗證成本 × 20**：這正是 [proving-a-bug-is-gone 驗證哲學](proving-a-bug-is-gone-verification-philosophy.md) 的「成本難」（驗證範圍 >> 修復範圍）**再乘以 N**
- **這就是回歸 vs 冒煙之爭的真實 stakes**：
  - 20 app 全回歸 = 唯一可行解是**自動化**（人工不可能）→ 上頭想壓回歸 = 想規模化
  - 但 20 app 全自動化回歸 = 維護債 × 20（連 [cross-app-locator](cross-app-locator-strategy.md) / [white-label 測試策略](white-label-app-testing-strategy.md)）
  - 冒煙 = 每個 app 跑核心 5 路徑、可規模化、又不背全回歸的維護債

→ **多產品線讓「為什麼要規模化」變得可理解**——不是上頭龜毛，是 20 app 真的逼出規模化需求。

## 解：不是「誰對」，是「配置」——Head 的 4 點框架（6/16 下班給的）

Head 昨天給的 4 點，其實就是 trade-off 的配置答案：

| Head 的點 | 對應配置 |
|---|---|
| 偶爾給實習生測一張 | 人力配置——可案例化的丟給新人 |
| 覺得要擴充人力 | 規模化的人力面 |
| **回歸的測像多一些改冒煙** | **往 user 的方向帶**——這是盟友訊號、別讀成「我 vs 上頭」 |
| 哪些給 AI 探索、哪些給自動化 | 工具配置——探索 vs 確定性驗證分流 |

→ 文章的 resolution = 從「我對 / 你錯」升級到「**哪些用冒煙（資深判斷、不可規模化）、哪些用自動化回歸（可規模化到 N app）、哪些給 AI 探索**」。**先 steelman 雙方、才配得出這張表。**

### 時機維度：「願意燒 token 的總監」是有時效的窗口（6/17 戰略反轉）

「願意燒 token 的工程總監」不是常態——**總監會換、預算會收、燒 token 的容忍度會隨成本曲線下降**。這把選擇從「對抗 vs 順從」翻成「借力」：

- ❌ 對抗（堅持「冒煙就夠、別燒 token」）= 逆風 + 錯過窗口
- ✅ **借窗口**：既然有人願意燒 token 規模化，**我的冒煙基準線正好是 AI 規模化的地基**——AI 跑上千結果沒人讀得完、要一組高信賴度基準線錨定（[brand-value 提案](brand-value-and-smoke-test-proposal.md)早論證過）
- 🟢 **而且我已經做出來了**：[smoke 框架](smoke-test-golden-set-spec.md)（7 APP / 4 架構、`pages/` 一行未改、「給 AI 半小時調定位器、人工 review」）**就是「冒煙當 AI 規模化地基」的實體**——不是提案、是在跑的東西。寫這篇時這是最硬的證據：我不是反對規模化、我做了規模化最需要的那塊地基

→ 同一件事、換框架，從「我 vs 規模化」翻成「**我的冒煙是規模化的地基**」、從逆風變順風。連 [5/21 借用主管框架推自己的事](coach-sessions/2026-05-21-insights.md) + [6/3 用他們的語言包裝成互補而非孤立](coach-sessions/2026-06-01-insights.md)。

→ **這讓 resolution 多一個時機維度**：不只「哪些冒煙哪些回歸」，而是「**在這個『有人願意燒 token 規模化』的窗口裡，怎麼把冒煙卡成 AI 規模化的地基、而不是被規模化輾過去**」。

### 最深一層：別把對方定格——他們也在成長（6/17 user 補）

> user：「Head 或 Lead 應該正在做觀念上的調整，說不定他們也開始認同冒煙了，只是我自己需要調適『他們也是有成長的』。」

這把「從對方角度」推到最深：
- **不只理解對方「現在的本金」，還要承認對方的本金「也在演化」**——我心裡有一個**靜態**的「他們要回歸」模型，但人會變
- Head 第 3 點「往冒煙帶」可能不是孤立訊號，是**他們觀念正在移動的證據**
- **我需要調適的不是「怎麼說服一個固執的上頭」，是「更新我對他們的模型」**——把他們從「對手/障礙」重新看成「也在學、也在調整的人」
- 連 [5/27「對齊 vs 說服」姿態校準（把前輩當前輩、不當對手）](coach-sessions/2026-05-27-insights.md)：如果我預設他們不會變，我就會用「對抗」姿態；一旦承認他們在成長，姿態自然變「一起對齊」

→ **這是文章結語最有重量的轉折**：與其執著「證明我的冒煙論對」，不如「**注意到我們其實正在往同一個方向收斂、只是節奏不同**」。煩悶會大幅下降——因為敵人從來不是他們、是我把他們定格的那個靜態模型。

## 候選結構

```
一、破題：今天的待辦長到做不完，煩悶
   - 但煩悶的不是「量」，是「我覺得該用冒煙、卻被要求壓回歸」
   - 核心叩問：做不完的焦慮，有多少其實是「不認同卻被迫做」？

二、我一直以為功能 + 冒煙就好（我的本金）

三、直到我試著想：上頭為什麼想壓回歸？（對方的本金）
   - 交代 / 免責 / AI 政治 / 規模化
   - 關鍵轉折：我不是論點弱，是沒先承認對方的本金就直接推

四、真實 stakes：當產品線有 20 個 APP
   - 修一個 bug、怎麼驗 20 個都好了？驗證成本 × N
   - 回歸（可規模化/自動化）vs 冒煙（靠資深判斷/不可規模化）的 trade-off 被放大

五、解不是「誰對」，是「配置」
   - 哪些冒煙、哪些自動化回歸、哪些給 AI 探索、哪些給新人
   - 先理解雙方本金，才配得出這張表

結語：面對做不完的成熟心態
   - 從「被迫做我不認同的事」→「我理解 trade-off、我參與配置」
   - 連 6/8「少攬不該扛 + 會不會忙到沒成長」
```

## ⚠️ 揭露邊界

- 「上頭壓回歸 vs 我想冒煙」是**現公司進行中的爭議** → 框成「**業界常見的回歸 vs 冒煙張力**」、不寫成「我們公司上頭」
- 「20 個 APP」是**公司產品線指紋** → 抽象成「一個有 N 個 white-label APP 的產品線」（同 [white-label](white-label-app-testing-strategy.md) 去識別化指南）
- Head 的 4 點 → 框成「一個成熟主管會怎麼拆」、不點名
- **矛頭對結構（trade-off）不對人**——這篇的價值是「我學會從對方角度看」，不是「證明我對、上頭錯」
