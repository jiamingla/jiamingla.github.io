# 〈可測試性是「治理問題」，不是「技術問題」〉骨架

> 📂 **定位**：線十（組織/無主之地）+ 線三。**旗艦、最原創**——從真實冒煙框架長出來的「看懂局」文章。
> 🪙 **核心金句**：「**識別碼要放在『擁有它所編碼之決策』的那一層。**」
> 📦 **素材來源**：smoke docs `cross-app-testing-strategy.md` §5（可測試性契約）。真實、已踩過。

---

## 核心論點

設計系統的通用元件是**無語義的插槽**（一個可放圖片、可設點擊去向的殼）；語義（這是搜尋鈕、那是鈴鐺）不在做元件的工程師手上，而是**產品端在組裝畫面時才賦予的**。

→ 所以「叫工程師加 testID」是搞錯層級：他根本沒有語義可加（實際問過，回覆＝「元件太抽象、不知道產品的用途」）。

**同一個 ID 放在不同層級，各自的失敗模式：**

| 放哪層 | 誰蓋 | 失敗模式 |
|---|---|---|
| 不蓋（現況） | 沒人，QA 靠結構 id/位置/文字 | 通用元件的 id 由實作決定、非用途 → 兩 App 同一「搜尋插槽」渲染出不同 id → **維護稅 ∝ App數×版本數** |
| 元件層 | 工程師 | 不知用途、只能給 `imageSlot1` → 等於把 id 換個名字，分歧照舊 |
| QA 測試層（現況 per-app locator） | QA | 可運作，但「語義↔元件」住在 QA 檔案裡、非從真相源傳播 → 每個新 App 人工重推、會漂移 |
| **產品組裝層（理想）** | 產品端 | **唯一同時**具備：①知道用途 ②單一真相源 ③自動傳播到每個用同套系統建的 App |

**代價一句話**：你不是「沒有 ID」，是把「語義→元件」的對應，從一個會自動散佈的單一真相源，**降級成 QA 逐 App、逐版本手動重建的維護稅，且隨 App 數與改版頻率複利成長。**

---

## 結構草

1. 現象：每接一個新 App，都要人工重推一次 locator，為什麼？
2. 因為 ID 蓋錯了層級——識別碼要放在「擁有決策」的那層
3. 三層失敗模式表（上）
4. 正解：ID 從組裝層注入（類比 per-use 的 `aria-label`），對工程師只是渲染引擎一次性改動、對 QA 繞過「不該讀 source」的爭議
5. **真正的難關是治理不是技術**：只有產品端用一致詞彙才跨 App 穩定；先標準化 5–8 個 common chrome（home/search/notification/more/back），別想一次規範全部
