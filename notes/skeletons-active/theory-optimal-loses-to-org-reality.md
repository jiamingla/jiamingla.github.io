# 〈技術上「理論最優」為何敗給「組織現實」〉骨架

> 📂 **定位**：線十（組織）+ 線三。看懂局的技術決策。
> 🪙 **核心金句**：「路線選擇的本質，常是組織問題，不是技術問題。乾淨的方案可能卡在你搬不動的組織關卡上。」
> 📦 **素材來源**：smoke docs `ci-apk-api-vs-crawler.md`。2026-06-15 實測決策紀錄。

---

## 核心論點

要自動下載測試用的 App 安裝檔，有兩條路：
- **官方 REST API**：理論最乾淨（幾個 HTTP 呼叫）。但實測 403——API 在該專案沒啟用、且服務帳號要在**每個子專案逐一被授權**（跨專案、可能跨部門的權限治理）。
- **UI 爬蟲**：不優雅、依賴介面易碎。但它用的是**人類帳號**，本來就看得到所有專案，**零 infra 設定**。

→ 爬蟲「贏」不是因為技術好，是因為**它繞過了組織關卡**。API 要贏，得先有工程師以外的人去動權限治理。

**更上位的教訓**：技術債可以從「難維護的爬蟲」搬到「跨專案權限治理」——但那是一筆**組織成本**，不是改 code 能解的。選路線前先看清：這條路的瓶頸是技術，還是組織？

---

## 結構草

1. 兩條路的技術比較（API 乾淨 vs 爬蟲脆）——如果只看技術，該選 API
2. 為什麼乾淨的敗了：403、跨專案 IAM、要跨部門
3. 爬蟲為何反而穩：人類帳號繞過治理、零 infra
4. 抽象：把「理論最優 vs 能落地」的判準講清楚——**看瓶頸在哪一層**
5. 收：資深的價值不在選出「最乾淨」的方案，在看懂「哪個關卡我搬不動」

## ⚠️ 去識別
- 雲平台專案 id、服務帳號 email、公司、同事名**全去掉**；框成「一個要跨多個雲端專案取檔的情境」。
