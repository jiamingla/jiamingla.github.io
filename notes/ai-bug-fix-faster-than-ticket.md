# 備忘：AI 修 bug 比我發 ticket 還快——QA 還該不該留紀錄？

**建立日期**：2026-05-25
**狀態**：💡 觀察記下，未來想寫時打開看。**不在 `_drafts/`**——還在 idea 階段。
**起源**：2026-05-25 對話——AI 導入後 RD 修 bug 的速度，有時候快到「QA 發 bug report 前就已經修好了」，引發疑問：這時 QA 還該不該開 JIRA ticket 留紀錄？

---

## 原始觀察（保留我的話）

> 隨著 AI 導入，RD 本來修 bug 的速度就不慢，現在快到在我發 bug report 前就修好了。這樣 QA 該不該開 JIRA ticket 留下紀錄？
>
> 我可能會寫**這能夠幫助未來的團隊將 bug 分類、找到團隊比較常出現的 bug 類型**。

---

## 我看到的結構

這個議題表面是行政決定（要不要開 ticket），**但骨子裡是兩個動作被混在一起**：

| 動作 | 被 AI 加速嗎？ | 誰能做？ |
|---|---|---|
| **修 bug**（個別問題解決） | ✅ 大幅加速 | RD + AI |
| **留紀錄 + 找 pattern**（知識累積） | ❌ **完全沒被加速** | 只有 QA 在做（其他角色 KPI 不算這個） |

**關鍵翻轉**：很多人直覺認為「修得快 = 不需要紀錄」。**反過來才對——修得越快，紀錄越重要**。理由：

1. **修了就修了 → bug 知識瞬間蒸發**——沒人記得我們上週修過什麼類別的 bug
2. **沒有 pattern recognition 的基礎** → 沒辦法回答「我們團隊最常出什麼類型 bug？」
3. **沒有那個知識** → 新人 onboarding 全靠口傳、QA 也沒辦法跟主管說「我們的 bug 分布長這樣」
4. **AI 加速修 bug = bug 流向更難追** → QA 的紀錄角色不是被取代，是**比以前更稀缺**

## 文章框架草擬

```
前言：那個讓我發呆三秒的場景
  → 我打開 JIRA 準備寫 bug report
  → 看到 RD 已經 commit「fix: ...」進去了
  → 那一刻我問自己：那我還要寫 ticket 嗎？

一、表面看：寫 ticket 變成行政負擔
  → 修了就修了，寫 ticket 像補作業
  → 直覺：可省則省

二、深一層：「修」跟「累積知識」是兩個動作
  → 修 bug = 個別問題解決，被 AI 加速
  → 累積 bug 知識 = pattern recognition，沒被加速
  → 兩件事被混在一起，所以才會覺得 ticket 多餘

三、為什麼累積知識特別重要（給組織的價值）
  → 新人 onboarding 的「我們常出什麼 bug」知識庫
  → QA 跟主管報告時的「分布長這樣」結構化證據
  → 找 root cause 模式（多次出現的 bug → 系統性問題）
  → 預防勝於治療：知道常出哪類，就能事前佈防

四、QA 的責任編輯角色——從擋稿到文獻管理
  → （呼應 ai-as-yes-man 的「責任編輯」隱喻）
  → 編輯不只是擋下不夠好的稿子，也整理文獻、留下索引
  → AI 加速時代，「整理 + 索引」這個職能比「擋稿」更不可替代

五、操作建議：light ticket 也是 ticket
  → ticket 不一定要 verbose
  → 最低限度：symptom + 修了 + 模式分類 tag
  → 重要的是有沒有「分類 tag」——這是未來找 pattern 的鉤子
  → 不要追求 ticket 完美，追求 ticket 存在

結語：修得越快，越該留紀錄
  → 在 AI 加速時代，QA 不是被取代，是**角色比重重新分配**——從「發現」往「記住」移
  → 沒人能告訴你『我們團隊最常出什麼類型 bug』時，那家公司其實沒有 QA
```

## 觸發條件（什麼時候從 notes/ 進 _drafts/）

- [ ] **再經歷 1-2 次「RD 修 bug 比 QA 報 ticket 還快」的場景**——案例越具體，文章越紮實
- [ ] **實際開始做「bug 分類 tag」實驗**——不論在 JIRA、Google Sheet、或私人筆記累積 4-6 週，**有資料才寫**（不要寫成「我打算做」這種空談）
- [ ] **yes-man-sequel 文章發出去後**——本篇是 sequel 的活體例證，發在 sequel 之後做為「**例證篇**」最自然
- [ ] **跟同事或主管實際聊過「bug 模式分類有沒有 value」**——他們的反應會是文章內容

## 紅線提醒

- ❌ **不要寫成「AI 威脅論」**——主張是 AI 加速**強化**了 QA 的紀錄價值，不是「AI 來搶飯碗」
- ❌ **不要寫成「QA 要爭取存在感」**——主張該是「**知識累積對組織的價值**」，不是「保住 QA 角色」
- ❌ **不要點名具體 RD / 修過什麼 bug**——RD 修 bug 快是好事，文章主角是「**知識累積這個動作**」，不是「快慢比較」
- ❌ **不要把 ticket 寫成「QA 的工具」**——ticket 是「**組織的記憶**」，這個 framing 更高，也更難被反駁
- ❌ **不要量化「我們團隊 80% bug 是 X 類型」**——除非真的有資料，否則粗估會破壞文章可信度
