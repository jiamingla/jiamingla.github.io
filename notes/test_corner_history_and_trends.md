## 趨勢分析

### 一、主題演進：從「會測試就好」到「整個工程組織都要會品質」

**2015–2017：建立基礎，手機為王**

早期幾乎清一色是 Mobile 測試，iOS（XCUITest、EarlGrey、Appium）與 Android（Espresso）占了大多數議題。這對應到台灣 App 創業潮，當時 KKBOX、LINE、CloudMosa、PicCollage、Honestbee 是最活躍的貢獻者。議題偏向「工具介紹」與「從手動轉自動化的第一步」，還有不少分享者坦承自己走了彎路（#14 的 Unit Test 心路歷程、#11 的放棄 Appium）。

**2018–2019：深化工程，CI/CD 與微服務崛起**

CI/CD Pipeline 整合（Jenkins vs GitLab CI、WWDC Test Plan）、微服務架構下的測試挑戰（Feature Test、docker-compose/k8s 環境建置）開始成為主流議題。HTC Viveport 那場（#22）是個里程碑，四個講題全部圍繞微服務測試，代表台灣 QA 圈在工程深度上的集體升級。同一時期也首度出現區塊鏈測試（#17、#20、#22）與跨國協作（#19 LINE 日本福岡 × Carousell 新加坡）。

**2020：疫情催化線上能力，金融業正式入場**

2020 年連辦五場，但都是在疫情壓力下撐過來的。國泰金控從 #28 開始連續出現，成為台灣金融業 QA 工程化最具代表性的案例。議題從「金融業到底在跑什麼測試」變成具體的微服務 API 自動化、區塊鏈品質把關、效能測試。CoolBitX 那場（#27）則是 crypto / fintech 賽道的首次現身。

**2021–2022：疫情後遺症與社群重建**

#29 延期、#30 線上改辦、#31 直播、#32 才恢復實體。這段時間議題量縮減，但 LINE 作為穩定場地方與講者供給來源發揮了關鍵作用。連 Load Testing 也首次以 Continuous Load Testing 的形式出現（#31），顯示可觀測性（Observability）開始進入測試圈視野。

**2023–2024：組織轉型、平台化、Chaos Engineering**

後疫情的復甦不只是回到原點，而是議題明顯往「組織與系統層次」升級。#33（Appier × KKBOX）的「品質是設計出來的不是測試出來的」反映了測試思維的根本轉移。#34 的 Change-Based Testing（只跑與本次變更相關的測試）代表效率工程的成熟。#35 的 Chaos Engineering 與 QA as Service 則把 QA 角色從驗收者變成系統穩健性的主動設計者。

**2025：Contract Testing 與基礎設施測試**

#36 Netskope 的三個議題——Pact 合約測試、Multi-simulator / Ngrok 整合、NPA infrastructure 分層測試——顯示測試工程正在往更底層、更跨系統邊界的方向走。Contract Testing 解決的是「微服務之間的溝通誰來負責驗證」這個在大規模系統中長期模糊的問題。

### 二、場地與社群生態的變化

**早期（#1–#10）**：場地高度集中在幾個固定場所（PicCollage、Mozilla Taiwan、北科大、AppWorks），反映社群規模小、場地贊助靠人脈。

**中期（#11–#25）**：LINE TW 成為最重要的固定場地，幾乎每隔幾場就回去辦一次。HTC、91APP、Carousell、COBINHOOD 等科技公司陸續作為場地方，社群規模擴大。

**近期（#26–#36）**：國泰金控（數數發中心）成為新的重要合作夥伴，連續出現三次（#28、#30、#35）；Appier、阿福管家、Netskope 等新面孔代入，顯示社群影響力已跨出早期的 Mobile App 圈子，延伸到 SaaS 和資安領域。

---

### 三、幾個值得關注的長期脈絡

**KKBOX 是最持久的知識貢獻者**，從 #2 一直到 #33，橫跨 8 年持續出現，議題也隨時間從 Xcode 自動化升級到效能測試架構，是社群的隱形骨幹。

**國泰金控是近年最重要的新進者**，從 #28 到 #35 共貢獻四場，且每次議題深度都在增加，代表金融業 QA 工程化的快速成熟。褚威（Tall Paul）個人更是跨 #28、#35 兩場分享，是少數在同一社群持續深化分享的講者。

**Jersey Su 是社群核心維繫者**，從 #4 的講者到 #34 的主辦方，再到 #36 仍在場，同時也出現在本次對話紀錄中回應社群詢問，說明 Test Corner 的延續依賴少數高度投入的核心人物在支撐。

---

### 四、#37 的可能方向

從對話紀錄來看，目前 #37 還沒有公司接洽主辦，Jersey 也在問是否有人想分享主題。結合趨勢判斷，以下幾個方向與 Test Corner 當前的脈絡最接：

- **AI 在測試流程中的實際落地**：#24 的 APE（AI Testing 猿）是唯一一次正式觸碰 AI Testing，距今已是 2019 年的事。現在 AI-assisted testing、LLM-based test case generation 已是業界熱門話題，但 Test Corner 完全缺席，是明顯的空白。
- **可觀測性（Observability）與測試的結合**：Continuous Load Testing（#31）是初步嘗試，但 distributed tracing、log-based testing、synthetic monitoring 等話題從未出現。
- **測試組織轉型與 QA 角色定位**：#33 的「品質是設計出來的」、#35 的「QA as Service」都在指向同一方向，這個話題在管理層面還有很多可以深挖。
- **平台工程（Platform Engineering）視角的測試**：測試環境管理、測試平台建設這條線從 #35 開始出現，但尚未有人從 Platform Engineering 的角度系統性分享。

## 五、個人對談與 #37 投稿策略（2026-06-17）

在私訊 Jersey (Test Corner 核心維繫者) 以及準備後續對談時，可採用以下策略：

### 1. 破冰與共鳴連結點（ACC 框架實踐）
*   **連結契機**：提及閱讀了 Jersey 寫的〈計畫趕得上變化〉一文，並分享自己如何在實務中（如登入服務損壞時）應用 James Whittaker 的 ACC 框架，寫下個人技術反思〈測試站登入壞了那天：我真的測完了嗎？〉。
*   **目的**：展現自己不僅是讀者，更是具備高度執行力與品質思維的實踐者，迅速建立同行的信任感。

### 2. 針對 #37 的三個候選分享主題（投稿提案）
結合 Test Corner 的工程深度趨勢，以及目前的 AI 落地與單兵 QA 痛點，可提出以下三個切角的提案：

*   **提案一：〈單兵 QA 的「影子自動化」革命：如何在沒有資源的泥沼中完成自我救贖？〉**
    - *核心主張*：不要盲目照搬大廠或顧問的重型 Top-down 自動化架構。分享在零文件、環境時常損壞的 Legacy 系統下，如何利用個人 10% 的空隙時間進行「影子自動化 (Shadow Automation)」。
    - *實戰做法*：主攻 API 與 DB 層（如資料預備、一鍵重置腳本），解決 80% 重複手動，並將工具包裝成「特洛伊木馬」給開發者使用建立同盟。
    - *聽眾共鳴*：直擊台灣中小型新創團隊唯一 QA 的生存痛點。

*   **提案二：〈當開發用 AI 提速：QA 如何用「黃金測試集」與「風險顯影」奪回品質主導權？〉**
    - *核心主張*：AI 提速導致代碼產出暴增與 Bug 增加，QA 人力維持原狀。QA 必須擺脫無盡的瑣碎 Bug 報告，主導定義必測主動線的「冒煙測試黃金集 (Golden Set)」。
    - *實戰做法*：分享如何克服 AI 案例冗長難讀的閱讀瓶頸；如何主動顯影「測不了的已知風險」（例如 Alpaca 以外券商無 Sandbox 的真實案例），將風險決策權透明化移交給 PM。
    - *聽眾共鳴*：直擊 AI 輔助開發下的品質管理轉型與責任邊界劃分。

*   **提案三：〈一頁式測試計畫的肌肉記憶：從登入服務損壞談起〉**
    - *核心主張*：當 Sprint 與環境快速變動，傳統測試計劃書流於形式。分享如何在 10-30 分鐘內套用 ACC 框架產出「一頁式測試計畫 (One Page Test Plan)」。
    - *實戰做法*：如何將技術測試計畫轉譯為 PM 聽得懂的「商業風險/痛點對齊語言」，在危機中第一時間定錨品質。

### 3. 對談提問與引導方向（結合 Coffee Chat）
*   **組織重視度診斷**：詢問前輩如何以「ESG 三級診斷框架（真投資、生存剛需、主管跟風試水溫）」評估一家公司對品質工程的實質投入，並評估「留在原地當影子工程師磨練溝通」與「果斷轉場」的時機。
*   **IC 領導力展現**：請教作為單兵（IC），在沒有主管職缺的環境下，除了帶實習生和辦讀書會，該如何在日常與履歷中累積並展現帶人的 Leadership 潛力。
*   **對外溝通界線**：請教進行跨部門與對外溝通時，如何踩穩「溝通樞紐」而非「客服」的界線，既能同步現況解決問題，又不將己方的流程漏洞全盤托出。
