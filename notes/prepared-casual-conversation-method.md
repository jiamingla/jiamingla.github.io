# 準備過的閒聊：PDT 的社交版（method 備忘）

**建立**：2026-06-12
**狀態**：💡 生鮮 method 記下、不知哪天用得上（可能變工具 / 文章 / 帶人方法）
**起源**：6/11 下班花五分鐘問即將離職的 QA 同事「下一步去哪 + 有沒有早點知道就好的事」、體驗到「準備過的閒聊」威力

---

## 核心 method

> **認真的閒聊，但又不是正式的對話。**（user 6/12 自己 surface 的 framing）

帶著準備好的問題、但用閒聊的溫度遞出去。

| Mode | 缺點 |
|---|---|
| 正式對話 | 對方防備、交淺言深的壓力 |
| 純閒聊 | 問不到深的、隨機 |
| **準備過的閒聊（中間態）** | **繞過兩邊缺點**——有閒聊的鬆、有準備的深 |

**只加了一個變量**：user 本來就會找人閒聊、昨天只是加了「**準備**」——結果完全不同。

---

## 為什麼這是 PDT 的社交版

- PDT 招牌：「測試的靈魂在提問」——對產品用問題找未知
- 準備過的閒聊：對**人**用同一套——帶著準備好的問題、打開對方沒預備的區域
- 6/11 活體證明：問即將離職同事「下一步去哪」，預設框是「換環境 vs 追新」二選一，對方真實答案（換部門、但不做 QA）**跳出預設框**——**好問題的價值不在預設選項裡、在它打開了沒預備的區域**

→ 「答案藏在沒問過的問題裡」這次測的不是產品、是一個人的職涯。

## 如果要 scale 成「每天五分鐘」習慣（先別急著建系統）

6/11 user 提到想「每天下班問五分鐘」。設計問題（**等 MVP 多跑幾次再決定**）：

1. **問誰**：不該每天問同一個人（會從交流變盤問）——輪替：同事 / PM / RD / 實習生
2. **問什麼**：從 23 面向挑題、改寫成閒聊版
3. **為什麼**：目的決定持不持久
   - 收集情報 → 功利、對方會感覺到
   - 練樞紐 → 對、但小心為練而練
   - **真好奇 + 跳出執行視角** → 這個能持久
4. **紅線**：別在紅燈時段（剛下班、累）把它設計成宏大系統；先 MVP、感受充電 vs 耗電、再決定 scale

## 即將離職同事的特例價值（6/11 活案例）

即將離職的人 = **站在你站不到的位置**（一隻腳在門外、回頭看組織更清楚）+ 心理最鬆最敢講真話。

最值得問的不是「未來規劃」（客套），是：
- **回望型**：「有沒有哪件事是你覺得『早點知道就好了』的?」（hard-won lesson 的善意傳承）
- 收尾姿態：「我還在繼續、想跟你偷學一點」——把「打聽」變成「珍惜你的經驗」

6/11 結果：對方換部門、確定不做 QA → **輕信號**（不急著歸因 / 別個人化）：
- 若「QA 在這組織看不到未來」→ 呼應「AI 時代 QA 何去何從」
- 若個人選擇 → 跟你無關
- **未來若又有 QA 離開 / 都往別部門 → 個案變 pattern、再看**
