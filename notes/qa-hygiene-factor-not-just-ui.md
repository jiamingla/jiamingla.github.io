# QA 不該只守激勵因子：從 Herzberg 看品質的守備範圍（骨架）

**建立**：2026-06-17
**狀態**：🌱 骨架——核心 thesis + Herzberg 脈絡 + SDET 延伸（沒 SRE 怎辦）
**起源（私人燃料、不寫進文章）**：半年前公司某高層認為「QA 就是做 UI 測試 + 一些 UX 測試」。user 當下不服、半年後用 Herzberg 講清楚為什麼。

## 核心 thesis：下檔不對稱——QA 被框在激勵因子角落，保健因子才是高風險

| | QA（UI/UX） | SRE / 可靠性 |
|---|---|---|
| Herzberg | **激勵因子**（做好=加分、用戶不特別謝） | **保健因子**（平常沒人注意、一炸是災難） |
| 做到完美的**上檔** | 體驗變好（有限） | 沒人發現（中性） |
| 出事的**下檔** | 小惱火（「UI 怪怪的」） | **災難**（當機、憤怒、流失、信任崩） |

→ **真正的引擎是「下檔不對稱」**：UI 差 = 微惱、可靠性炸 = 無上限災難。所以「品質工作擺在哪能擋最多傷害」= **保健因子那層**，而 QA 正被框在激勵因子角落。

## 🔧 SDET 延伸（6/17 user surface）：如果產品沒有 SRE，我一個 QA 能做什麼？

> 「這也讓我思考 SDET 那篇——如果產品沒有 SRE，那我一個 QA 能做點什麼？」

接 [SDET 篇](/post/sdet-role-reflection-in-ai-era/)的「影子 SDET/SRE」——但這裡更具體、更有 Herzberg 底氣：

**沒有 SRE 時，保健因子那層是「無主之地」**（連線十）——QA 不必變成全職 SRE，但可以用單兵手段**守住保健因子的最小防線**：

| 單兵 QA 能做的（守保健因子） | 對應既有資產 |
|---|---|
| **核心路徑冒煙測試**（登入/交易還活著） | [smoke 框架](smoke-test-golden-set-spec.md)——這就是在守保健因子！ |
| **核心功能紅綠燈**（第一時間宣告核心狀態） | [brand-value 提案](brand-value-and-smoke-test-proposal.md)「異常時誰接住」 |
| **監控/告警意識**（不是建 SRE 平台、是知道哪裡會炸、盯著） | 影子工程師（SDET 篇） |
| **把保健因子風險攤給決策者**（不自己硬扛） | 6/4「測不了的已知風險攤給 PM」 |

→ **關鍵論點**：單兵 QA 在沒 SRE 的產品裡，最高槓桿不是「測更多 UI」，是「**守住那條沒人守的保健因子底線**」。這正好把「QA 別只守激勵因子」落成 actionable——**沒 SRE = 保健因子無主 = 單兵 QA 的最高價值切入點**。

→ 連你的 endgame：新創第一位 QA / RD-Head-style QA = **守備範圍從激勵因子擴到保健因子的人**（[small-team](small-team-qa-content-and-startup-endgame.md)）。

## 🔑 客戶導向悖論：客戶沒要求我守保健因子，怎麼辦？（6/17 user 卡關核心）

> user：「如果整間公司的人都是我的客戶，那客戶沒提出要我把守備範圍往 SRE 走，該怎麼辦？」

**這是這篇的核心張力、也是必須正面回答的一題。**

### 悖論：保健因子的定義就是「沒人會主動要求」

保健因子平常隱形、價值藏在沒發生的事裡——所以**「等客戶開口才做」套在保健因子上會自我矛盾**：它保證你永遠困在激勵因子角落，**直到災難那天、客戶才用憤怒的方式「提出需求」**。客戶不是不需要，是**保健因子的需求天生說不出口**。

### 解一：客戶有明示需求 + 隱性需求——服務隱性需求正是 QA 招牌

| | 明示需求 | 隱性需求 |
|---|---|---|
| 例 | 「測這 feature / 做 UI 測試」 | 「核心別在我沒注意時炸掉」 |
| 誰服務 | 任何 QA | 資深 / 責任編輯 / PDT |

「只做客戶開口要的」= yes-man；「服務說不出口的需求」= QA 招牌（PDT 問沒問過的問題 / 責任編輯挑沒被要求挑的漏洞 / 無主之地）。**卡關 = 把自己的招牌忘了。**

### 解二（操作關鍵）：不必宣稱「往 SRE 走」、用既有授權守

- ❌「我要做 SRE 的事」= 侵 turf、要授權、攬不該扛
- ✅「**QA 本來就該守核心功能不壞**」= 既有授權內、沒人反對、**smoke 是載體**（無疑是 QA 工作、又剛好守保健因子）
- → 守的是保健因子、披的是「徹底的 QA」的皮。不需許可、不侵誰、價值自己被看見（尤其核心差點炸、smoke 接住那天）

## 主管既是客戶、又是利害關係人：他可能不希望我往 SRE 走，怎麼面對？（6/17 user surface）

**主管（Head）比一般客戶複雜**——他不只「有隱性需求」，還是**有自己利益的利害關係人**，可能主動不希望我擴張。

### 先 steelman：主管為什麼可能不希望（從對方角度）

| 主管的顧慮 | 本金 |
|---|---|
| **turf / 跨團隊摩擦** | QA 踩進 SRE 地盤 → 他要去擺平的跨團隊麻煩 |
| **頻寬 / 聚焦**（尤其旺季） | 想要我的頻寬在交付、不是自主擴張 scope |
| **當責模糊** | QA 做 SRE-ish 的事出包、誰扛？模糊 ownership = 他的頭痛（連 6/4 DTNO「給越多被反問越多」） |
| **不是他的 KPI** | 他被既有 scope 衡量、我做別的不幫他指標 |
| **他自己的 QA 模型** | 他可能跟那位高層一樣，覺得「QA = UI/UX」、「QA 做可靠性」不 compute |

### 怎麼面對（不對抗、不佈道）

1. **不框成「往 SRE 走」**——那觸發上面所有警報。框成「**QA 把核心功能可靠性也涵蓋進來**」（smoke 守登入/交易）——他無法反對「徹底的 QA」。**同一個載體、同時解掉操作面 + 主管疑慮**
2. **主管是盟友不是障礙**（6/15 Head 在保護你頻寬、幫你擋 PM）——用 advisor-style 把保健因子 gap 當「**觀察 / 選項**」surface 給他、讓他決定 framing 跟時機（連 6/9 讀懂 Head 原則、6/17 別把對方定格他也在成長）
3. **別佈道、尤其旺季**——他若現在不要（旺季 / turf 敏感），別硬推（連 6/8 少攬、別當佈道者）。保健因子的工作可以等淡季、或等一次真實 incident 讓需求自己浮現
4. **讓「接住的 incident」替你說話**——你已經做了 smoke（QA scope 內）。哪天核心差點炸、smoke 接住了，那一次 demonstration 勝過十次抽象爭取（被看見自然發生）

### 如果 surface 過、主管仍不要——這本身是 endgame 的 data

接受「不是我能控制的」（6/4）——**但這不是死路、是資訊**：
> 如果這間組織結構性地把 QA 框在 UI/UX、不讓它守保健因子，那這就是「**這裡適不適合我『守備範圍含保健因子』的志向**」的數據。

→ 直接連 [新創第一位 QA endgame](small-team-qa-content-and-startup-endgame.md)：**新創從白紙定義 scope、沒人把你框進 UI/UX**。主管的抗拒若持續，不只是障礙、是「**為什麼那個 endgame 吸引我**」的佐證（連 SDET 篇「換一間有土壤的公司、比在原地推動快」）。

## ⚠️ 政治提醒

- 「公司 SRE 不夠好」**對內危險**（傷 SRE 團隊關係）→ 框成「**保健因子那層結構性地守備不足 / 是塊無主之地**」、矛頭對結構不對人（同 bug-management 紀律）
- 不寫高層那句真實對話（見揭露決定）
- 不寫成「QA 比 SRE 重要」——是「QA 的守備範圍該含保健因子、別被框在激勵因子角落」
