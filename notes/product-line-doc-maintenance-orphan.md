# 維護一份「整條產品線介紹文件」：又一塊無主之地（骨架 + 去識別範例）

**建立**：2026-06-12
**狀態**：🌱 骨架——文章框架 + 去識別化範例文件本體
**起源**：user 自己默默做了一份「新進 QA 補充文件」（補 PRD 對新人不友善的部分）——這份文件**不屬於任何人的 KPI**，但對後面每個新人救命。想寫成「**如何維護一份整條產品線的介紹文件**」的方法論文章
**姊妹檔**：[bug-management-is-a-profession-pm-gap](bug-management-is-a-profession-pm-gap.md)（bug 管理是無主之地）、[pm-qa-interface-structured-workflow](pm-qa-interface-structured-workflow.md)（PM-QA 介面 schema）——三者都是「沒人認領的專業 / 文件」這條主軸

---

## 文章核心論點

### 1. 這是一塊典型的「無主之地」

| 角色 | 為什麼不會做這份文件 |
|---|---|
| **PM** | 寫 PRD（功能規格）是 KPI、但「給新人的導讀 / 術語補充 / 資料來源驗證法」不是 |
| **資深 RD** | 知道一切、但都在腦袋裡（tacit）、寫下來不是他的 KPI |
| **新進 QA** | 最需要這份文件、但剛來、沒能力寫 |
| **資深 QA（你）** | ✅ 唯一同時「知道痛點 + 有能力寫 + 在乎」的人——但這也不是你的 KPI |

→ 呼應 [線十 無主之地](writing-themes.md)：**沒人天然該收這個 inbox**。但跟 bug backlog 不同的是——**這塊地你已經主動接了**（做出了 v1.0 文件），所以這篇可以從「第一人稱、我已經做了」的正向角度寫，不是控訴。

### 2. 為什麼這份文件值得做（即使沒 KPI）

- **救新人**：PRD 是給「已經懂的人」看的、對新人不友善（術語沒收錄、帳號不知去哪要、資料來源不知怎麼比對）
- **救自己**：寫一次 = 之後每個新人問同樣問題時不用重講（連 [準備過的閒聊](prepared-casual-conversation-method.md)：與其每次口頭回答、不如沉澱成文件）
- **schema asset**：這份文件本身就是 [schema asset](ai-era-tw-qa-needs.md)（跨新人重用、不是一次性 report）——demonstrate「設計 > 執行」
- **接 Head 的 demonstrate**：帶新人、降低團隊認知負荷 = RD Head-style QA Head 的核心動作（連 [ithome 規劃](ithome-ironman-2026-planning.md) endgame）

### 3. 怎麼維護（方法論主菜）

從這份 v1.0 文件抽出的「可遷移結構」（不綁特定產品）：

1. **導讀層**：新人不用全背、給「先讀哪幾章 + 順序 + 各章目的」的優先級表
2. **術語層**：補 PRD 沒收錄的縮寫 / 黑話、每個術語標「**對 QA 的意義**」（不只定義、是測試角度）
3. **帳號 / 環境層**：手上有什麼帳號、去哪要、不同角色測試怎麼準備（含多帳號多裝置情境範例）
4. **資料來源驗證層**：每個數據來自哪、即時 vs 盤後、怎麼跟後端比對、異常時的判斷流程
5. **NFR 白話層**：把抽象指標翻成「實際上怎麼測」
6. **版本 + 維護機制**：標版本、標 Open Issue（未確定的需求不要先寫死測試案例）、標「有疑問問資深 QA」

→ **核心心法**：這份文件不是「複製 PRD」，是「**翻譯 PRD 成新人能用的形狀**」——跟 [POM](pom-refactor-from-runnable-to-maintainable-planning.md)「把 UI 細節翻譯成測試意圖」、[first-principle](software-testing-first-principle.md)「降低讀者不確定性」同一個 DNA。

## ⚠️ 揭露邊界 + caveat（user 自己提的）

- **「PM 其實可能有很多相關文件、只是我沒附上來」**——文章**不能暗示「PM 都沒做文件」**。framing 要誠實：「PM 有 PRD 等規格文件、但『給新人的導讀層』這塊通常沒人補」，矛頭對**文件類型的缺口**、不對 PM
- 這份補充文件是「**一個 QA 的補充**」、不是「**取代 PRD 的權威文件**」——寫的時候定位清楚（補充 vs 取代）
- **現公司產品線指紋重**（金融 / 投資內容 / 作者社群 / 特定資料源名稱）——去識別化要徹底（見下方範例已處理）
- 跟 [cross-app-locator](cross-app-locator-strategy.md) / [bug-triage 前公司案例] 同樣紀律：抽象成通用商業場景

## 〔範例〕新進 QA 補充文件（產品線導讀版）

**文件版本**：v1.0
**適用對象**：新進 QA
**搭配文件**：產品需求文件（PRD）
**文件用途**：補充 PRD 中對新進人員不夠友善的部分——術語解釋、測試帳號說明、資料來源驗證方法、第一次讀文件的閱讀建議。

> 範例 domain：一個「**內容訂閱型 App**」（創作者發布內容、會員付費訂閱、含即時數據顯示）。各團隊可替換成自己的產品。

### 1. 如何閱讀這份 PRD（給新人的導讀）

剛進來不用把整份文件背完，建議閱讀順序：

| 優先 | 章節 | 目的 |
|---|---|---|
| 1 | 術語與縮寫（PRD + 本文件） | 先建立共同語言 |
| 2 | 使用者角色定義 | 了解有哪些身份、測試要切換角色 |
| 3 | 角色權限矩陣 | 測試權限邊界最重要的參考 |
| 4 | 本文件「測試帳號」 | 知道手上有什麼帳號可用 |
| 5 | 各模組「概述」段 | 用白話理解每個模組是什麼 |
| 之後 | 各模組 AC / BR / EC | 設計測試案例時才逐條看 |

**常見編號格式**（範例）：

| 格式 | 意思 |
|---|---|
| `FR-XX-001` | 功能需求（模組縮寫-流水號） |
| `AC-001` | 驗收標準（測試案例設計依據） |
| `BR-001` | 業務規則（不可違反的商業邏輯） |
| `EC-001` | 邊界條件（異常 / 特殊情境） |
| `NFR-PERF-001` | 非功能需求（效能類） |
| `OI-001` | Open Issue（待釐清、**尚未確定前不要寫死測試案例**） |

### 2. 術語補充（PRD 沒收錄的、每個都標「對 QA 的意義」）

| 術語 | 說明 | 對 QA 的意義 |
|---|---|---|
| **WebView** | App 內嵌網頁的元件 | 測網頁載入 / 返回鍵 / 內部連結行為 |
| **DeepLink** | 從外部直接跳轉 App 內特定頁 | 測推播點擊跳轉、分享連結開對頁 |
| **DeepLink 短網址服務** | 第三方生成 DeepLink 的短網址（如 AppsFlyer OneLink 類服務） | 分享連結格式可能是短網址、測跳轉 |
| **wss（WebSocket Secure）** | 持續連線的雙向即時資料通道 | 即時資料來源用 wss；數據沒更新可能是連線斷、測重連機制 |
| **IAP（In-App Purchase）** | App Store 內購機制 | iOS 金流用沙箱帳號、不產生真實費用 |
| **APNs / FCM** | iOS / Android 推播服務 | 測推播需確認對應平台通知權限正常 |
| **Cold Start** | 冷啟動（完全關閉後重開、非背景恢復） | 效能 NFR 常要求冷啟動秒數、測法：全關→計時開→可互動為止 |
| **ANR** | Android 主執行緒卡住超過 5 秒的無回應錯誤 | 不同於 Crash、會彈「等待 / 關閉」框、發生時截圖 + 記步驟 |

### 3. 測試帳號與環境

> 環境 URL / 設定向資深 QA 詢問；以下為架構說明。

後端有多套環境、測試主要用**非正式環境**避免影響真實用戶；部分情境（如金流）可能需正式環境驗證、用前先確認。

| 帳號類型 | 用途 |
|---|---|
| 一般會員 | 測未付費的功能限制（付費牆、鎖定） |
| 付費會員 | 測付費功能 |
| 內容創作者 | 測創作者專屬功能（發布、即時互動、刪訊息等） |
| 管理員 | 測管理權限（刪他人訊息等） |

> ⚠️ 付費帳號是公司資源勿濫用；測試產生的資料（訊息 / 收藏）測完清理；創作者帳號在測試環境發布前確認不會誤觸推播給真實用戶。

**多角色測試準備範例**（測「管理員可刪他人訊息、一般會員不能」）：
```
1. 一般會員（裝置 A）→ 發一則測試訊息
2. 管理員（裝置 B）→ 長按該訊息 → 確認有「刪除」
3. 一般會員 → 長按自己訊息 → 確認只有「收回」（無「刪除」）
```

### 4. 資料來源說明與驗證方法（最關鍵的一節）

**為什麼 QA 要懂資料來源**：畫面數據都從後端讀取顯示。**不能只看畫面數字「看起來合理」、要跟來源比對**。

```
後端資料來源 → 後端 API → App 前端畫面
QA 驗證 = 確認「畫面顯示的」與「來源的」一致
```

四種典型資料來源（**已抽象化來源名稱**）：

| 來源（範例代號） | 性質 | 更新時機 | 測試重點 | 比對方式 |
|---|---|---|---|---|
| **來源 A（盤後靜態）** | 收盤後更新的靜態資料 | 交易日收盤後、非即時 | 盤後更新後畫面數值是否一致；盤中異常先確認是否尚未更新 | 用後台查詢工具以代碼查、與畫面比對 |
| **來源 B（混合）** | 部分即時、部分盤後 | 依欄位而異 | 先確認欄位屬即時還是盤後、再驗更新行為 | 請資深 QA 說明哪些欄位來自此源 |
| **來源 C（即時推送 / wss）** | WebSocket 長連線即時推送 | 即時、有變動主動推送 | 盤中是否每隔幾秒更新；背景→前景是否恢復；斷線重連是否重訂閱 | 用抓包工具（如 Charles）監看 wss；或後台即時查詢頁 |
| **試算表型來源** | 以試算表 API 讀取、人工維護的資料 | 手動更新後、有快取非即時 | 更新後畫面是否合理時間內反映；舊資料問題查快取刷新 | 開對應試算表與畫面比對 |

> 💡 部分資料來源帶「範圍參數」（例：控制只顯示個股、或個股 + 其他標的類型）——測試時注意參數設定影響顯示範圍。

**數值異常時的判斷流程**：
```
發現數值異常
  ├─ 先問：這欄位來源是哪個？
  ├─ 即時源（wss）→ 確認連線正常 → 關閉重開 App
  ├─ 盤後源 → 確認是否已過收盤 + 後端是否已更新
  └─ 試算表源 → 開試算表確認內容 + 確認 App 快取刷新
```

### 5. NFR 對 QA 的實際意義（把抽象指標翻成「怎麼測」）

| 類型 | 指標範例 | 實際怎麼測 |
|---|---|---|
| 效能 | 冷啟動 ≤ N 秒 | 全關→計時開→首頁可點為止、錄影逐格 or profiler |
| 效能 | 頁面切換 ≤ N 秒 | 快速切底部導覽、觀察卡頓、**以使用者感受為主** |
| 效能 | 滾動 FPS | 快速滾長列表觀察掉格、嚴重時用 GPU profiler |
| 相容 | 最低 OS 版本 | 在最低支援版本裝置跑主功能 Smoke Test |
| 相容 | 深色模式 | 切系統深色、確認對比度 / 文字可見 |
| 可靠 | Crash Rate | 遇 Crash 記步驟 + 裝置 + OS、查 Crashlytics |
| 可靠 | ANR（Android） | 彈「等待 / 關閉」即 ANR、記步驟、嚴重時抓 trace |

**哪些 NFR 新人不用自己測**：服務可用性 / 訊息送達率（後端監控）、安全性（需專業工具）、維護性（部署架構層）——這些由資深 QA / 後端負責。
