## ✅ 揭露決定（已定 2026-06-17）：護理師朋友的專業視角

**不寫個人健康史**。改用「**我聽待在身心科的護理師朋友分享**」——他們也很難驗證「一個人的症狀改善到可以稱為健康 / 好了」。

為什麼這個框架更好：

- **力量幾乎不減、甚至更強**：專業人員的第一線觀察，比個人經歷更有公信力、更像「業界普遍難題」而非「我的個案」
- 跟你過去引神經科學 / 統計學 / 航空業同個套路——**借專業領域的真實困境，照亮 QA 的同構困境**

## 📚 醫學研究依據（網路初查 2026-06-17，待跟護理師朋友核對）

查到的臨床事實，每條都對應 QA 的同構困境——這是文章「不是我瞎類比、是真的同構」的底氣：

| 醫學研究發現 | 對應 QA 的困境 |
|---|---|
| **病人自評「好轉」vs 醫生量表評估，不一致率 51-55%**（PHQ-9 51% / BDI-II 55%） | **RD/使用者覺得「修好了」、QA 驗證不這麼認為**——這是文章最強的實證錨點 |
| **Remission（症狀緩解）≠ Recovery（功能性康復）**——量表分數降了不代表能回到正常生活 | **功能 pass ≠ bug 真的消失 ≠ 系統健康**；測試綠燈 ≠ 品質真的好 |
| **「沒有 gold standard」**——缺乏絕對基準時，病人回報的結果難以詮釋 | **測試沒有 oracle**——很多時候「什麼叫對」本身就沒有絕對答案 |
| **Recovery 的嚴格定義：臨床緩解 + 社會功能 + 症狀至多『輕微』+ 持續至少 2 年** | **「bug 消失」需要時間維度的持續觀察**，不是單次 pass 就算數 |
| **為何要走完療程**：繼續服藥 relapse 18% vs 停藥 37%；緩解後建議至少再 6 個月；**3 次以上發作史 → relapse 風險更高** | **為何要做回歸測試 / 為何「修好一次」≠「永遠修好」**；偶發 bug 像高 relapse 風險的慢性病，要持續監控 |
| **測量導向照護（MBC）：不靠單一量表、要多種評估方法** | **多軸 schema / 多種驗證手段**（連 [pass-fail-not-enough](../source/_drafts/pass-fail-not-enough-for-ai-testing.md)） |
| **沒按時服藥 → 醫生失去驗證基準線**（變數失控） | **測試環境/條件不一致 → 無法歸因「修好了還是剛好沒重現」**（flaky 同源） |

**研究來源（待核對）**：
- 病人 vs 醫生評估不一致：[Comparison between self-administered depression questionnaires and patients' own views (PMC8108392)](https://pmc.ncbi.nlm.nih.gov/articles/PMC8108392/)、[Concordance between clinician-rated and patient reported (PMC8672917)](https://pmc.ncbi.nlm.nih.gov/articles/PMC8672917/)
- Remission ≠ functional recovery：[Frontiers (PMC9360322)](https://pmc.ncbi.nlm.nih.gov/articles/PMC9360322/)
- Recovery 嚴格定義 / 2 年：[umbrella review (Molecular Psychiatry s41380-023-02298-3)](https://www.nature.com/articles/s41380-023-02298-3)
- 走完療程 / relapse：[Discontinuation meta-analysis (Molecular Psychiatry s41380-020-0843-0)](https://www.nature.com/articles/s41380-020-0843-0)
- 缺 gold standard 詮釋難：[CGI in liaison psychiatry (PMC9059324)](https://www.ncbi.nlm.nih.gov/pmc/articles/PMC9059324/)

→ **不必全引**，挑 2-3 個最強的（首推「51-55% 不一致」+「remission ≠ recovery」+「為何走完療程」）。

## 核心論點（這篇的靈魂）

> **驗證「功能正常」容易、驗證「bug 存在」容易，但驗證「bug 已不存在」極度困難——這幾乎是認識論 / 哲學難題。**

證明存在 vs 證明不存在的不對稱性：
- **證明存在**：找到一個 crash / 破版 / 一次 LLM 幻覺 → 附 log + 截圖 → bug 是鐵錚錚的事實。**一個反例就夠**
- **證明不存在**：測 5 次沒發生 → 能保證第 6 次不發生嗎？這在邏輯上是「**證明全稱否定**」——需要窮盡，而軟體狀態空間不可窮盡

→ 連 [first-principle](software-testing-first-principle.md)「軟體永遠 partial、不可窮盡」：證明 bug 不存在 = 試圖窮盡一個不可窮盡的空間。

## 三層遞進結構（實體 → 傳統軟體 → LLM）

不可控性 / 證明難度逐層放大：

| 層次 | 系統性質 | 證明「修好了」的難度 |
|---|---|---|
| **實體（馬克杯）** | 物理極限明確、可窮盡 | 改材質 → 耐熱 / 容量可計算可測完。**容易** |
| **傳統軟體** | 決定論，但我們寫的 code 只佔 1-10%，其餘是開源套件 / 第三方 API / OS 交織的黑盒 | 改一段 code「應該是這裡吧」→ 盲人摸象。**難** |
| **LLM** | **機率論**——連開發者都無法解釋它怎麼推出某答案、沒有固定邏輯路徑、只有機率分佈 | 微調 prompt 後，要輸入多少排列組合才能宣稱「永遠不會再講錯」？樣本空間近乎無限。**幾乎不可能** |

**關鍵升級**：傳統軟體至少是「決定論」（同輸入理論上同輸出，證明難但有限）；LLM 是「機率論」，**把證明不存在的難題放大無數倍**——同一個 prompt 兩次跑可能不同結果，「修好了」連定義都動搖。

→ 連 [ai-automation-cost-structure](ai-automation-cost-structure.md)（AI non-deterministic）、[yes-man](/post/ai-as-yes-man-rd-pm-trust-calibration/)（信心校準）、[pass-fail-not-enough](../source/_drafts/pass-fail-not-enough-for-ai-testing.md)（信心是獨立維度）。

## 各類 bug 的「驗證修復」難度光譜（初版，可再跟 RD 補充）

**關鍵洞察（6/17 user surface）**：難度其實有**兩個獨立來源**，原本只想到第一個——

- **認識論難**：bug 不可窮盡 → 無法證明「不存在」（偶發 / 漏洞 / LLM）
- **成本難**：bug 可窮盡、deterministic、邏輯上「能證明修好」，**但驗證範圍 >> 修復範圍**，光時間就吃不消（config / 環境 / 整合）

→ **「修復範圍小」≠「驗證範圍小」是 QA 最常被低估的成本**。RD 改一行覺得「就這樣」，QA 要驗的是那一行碰到的整個面。

| bug 類型 | 例子 | 可窮盡？ | 驗證成本 | 為什麼這麼難驗 |
|---|---|---|---|---|
| **功能型** | 按鈕點了沒反應 | 可 | 低 | 修復範圍 = 功能本身，驗證該功能正常即可 |
| **配置 / 環境型**（小修、大爆炸半徑） | k8s 一行 config 設錯 → 整組服務壞掉；修復只改一行 | **可窮盡** | **低難度、但高成本** | **修復 = 一行；驗證範圍 = 那行 config 影響的所有服務**。就算能測完也極花時間——這是「成本難」的代表 |
| **整合 / 接縫型** | 兩模組各自對、接起來壞 | 部分 | 中 | 改一側要重驗整個互動，組合數會擴張 |
| **效能 / 負載型** | 只在高併發 / 大資料量下崩 | 難 | 高 | 要先重現「規模」才驗得到，平常測不出來 |
| **偶發 / 時序 / race 型** | 有時會、有時不會 | **不可窮盡** | 很高 | 測 N 次沒出現 ≠ 沒了——這是「認識論難」的核心 |
| **「說不上來就是不對」型** | UX 怪、感覺不對勁 | — | 很高 | 連「bug 是什麼」都模糊，更難定義「它消失了」 |
| **資安 / 邏輯漏洞型** | 越權、注入、繞過 | 對抗性、近無限 | 極高 | 證明漏洞不存在 = 對「所有惡意輸入」證明全稱否定 |
| **LLM / 機率型** | 幻覺、偏見、不穩定回答 | 機率、近無限 | 幾乎不可能 | 同 prompt 兩次跑可能不同，樣本空間近無限 |

→ 文章可用這張表當「**難度光譜**」主視覺：從「功能型（好驗）」一路滑到「LLM（幾乎不可能）」，中間清楚標出「成本難」與「認識論難」的分水嶺。

→ **RD 的不確定性是關鍵真實感**：很多時候 RD 也在系統裡盲人摸象、改完「覺得應該是這裡」、丟給 QA「你測測看」，QA 承受整個系統的最終未知風險。**去識別化即可**（不點名特定 RD / 事件 / 公司；k8s config 這類通用技術例子可保留，不算指紋）。

> 📌 **待跟 RD 請教 / 補充**：這 8 類是初版。可請 RD 補「他們眼中最難跟 QA 解釋『修好了沒』的 bug 類型」——尤其偶發型、整合型，RD 的視角會讓表更完整。

```
一、破題：無法被輕易驗證的「痊癒」
   - 護理師朋友視角：他們在身心科第一線，也很難判定「一個人症狀改善到
     可以稱為健康 / 好了」——病人主觀覺得好了、醫生卻要走完療程；
     病人沒按時服藥 → 醫生失去驗證基準線
   - 補一句實證：病人自評 vs 醫生量表的不一致率高達 51-55%（見研究依據）
   - 核心叩問：變數無法控制時，怎麼定義「完全恢復正常」？

二、從實體到虛擬：不斷擴大的黑盒子
   - 馬克杯（實體、可窮盡）→ 傳統軟體（我們的 code 只佔 1-10%）
   → LLM（連開發者都解釋不了、只有機率分佈）

三、QA 的哲學難題：證明「不存在」的不對稱
   - 證明存在：一個反例就夠
   - 證明不存在：要窮盡不可窮盡的空間
   - 傳統 RD 盲改「你測測看」 vs LLM 微調 prompt 後的無限樣本空間
   - 兩種 bug（功能型 vs 說不上來型）的驗證難度差

四、與不確定性共存：QA 的真正價值
   - 不追求 100% 無瑕，是建「風險控制網」
   - 重新定義「修好」= 把風險壓到商業 / 用戶可接受範圍
   - 呼應療程：醫生靠「療程覆蓋率 + 後續觀察」、QA 靠「測試覆蓋率 + 監控 log」

結語：與未知和平共處
   - QA 最高級的心態不是「保證沒 bug」，是
     「我已排除最致命的風險，剩下的我有能力隨時監控應對」
```

## 🗣️ 問朋友的口語草稿（用「準備過的閒聊」心法、不要像審問）

> 用法：一次問一個人、聽多於問、別急著接話把它變成我的演講。開場用「請教你的專業看法」最自然（對方被當同儕、不是被打聽）。用詞匿名化承諾先講，對方才放心。

### 給護理師朋友（身心科）

**開場**：
> 「欸，我最近在寫一篇東西，想把『軟體測試』跟你們身心科的工作做類比——因為我發現我們驗 bug 跟你們判斷一個人好了沒，好像有點像。想跟你確認幾個我外行的理解對不對？放心，我不會寫到你或醫院，純粹借你們的專業困境講軟體的道理。」

**核心幾題**（挑著問、不用全上）：
1. 你們判斷一個病人「好得差不多、可以減藥 / 結案」的時候，主要看什麼？是有量表分數，還是更多靠觀察、靠病人自己講？
2. 我看資料說「病人自己覺得好了」跟「量表測出來的」常常對不起來——你實際遇到是這樣嗎？比較常是「病人覺得好了但你們覺得還沒」，還是反過來？
3. 「症狀緩解」跟「真的康復、回到正常生活」是不是兩件事？你們怎麼分？
4. 為什麼病人覺得好了、還是要把療程走完？是怕復發嗎？有沒有「停太早結果又回來」的例子？
5. 如果病人沒乖乖吃藥，你們判斷「他到底好了沒」是不是就更難？（變數失控）
6. 講白一點——你們有沒有一個「絕對標準」能說「這人 100% 好了」？還是其實沒有這種東西？

→ 重點驗證：第 2 題（51-55% 不一致是否屬實）、第 3 題（remission≠recovery）、第 6 題（無 gold standard）。

### 給 RD 朋友

**開場**：
> 「欸我在寫一篇講『怎麼證明一個 bug 真的修好了』的東西，想聽你 RD 的視角——因為我覺得很多時候你們改完、其實也沒有 100% 把握吧？想請教幾個。」

**核心幾題**：
1. 你改完一個 bug、丟給 QA 的時候，哪種 bug 你自己其實也沒把握「真的修好了」？
2. 有沒有那種「改一行 config / 一行 code，但影響範圍超大」的經驗？那種你會怎麼跟 QA 講要測哪些？（驗證「成本難」的代表）
3. 偶發的、有時會有時不會的 bug，你改完怎麼判斷它真的沒了？還是其實也是賭？
4. 哪種 bug 最難跟 QA 解釋「我改了什麼、為什麼這樣就好了」？
5. 你覺得「修復的範圍」跟「該驗證的範圍」常常差很多嗎？

→ 重點補：第 1、3、4 題補「bug 難度光譜」的 RD 視角；第 2、5 題印證「修復範圍 ≠ 驗證範圍」。
