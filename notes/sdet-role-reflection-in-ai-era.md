## 📌 核心反思筆記

### 1. 網友論點背後的「認知盲區」
* **網友的論點**：「Java 大佬認為測試跟 code review 應該是工程師之間交換做，不應該有 SDET 專職。因為你能看懂別人程式碼還知道他那裡寫錯 = 你的功力比別人高深（那為什麼不做 RD？）」
* **盲區解析**：這段話把 SDET（Software Development Engineer in Test）的職能**過度簡化**為「幫 RD 抓 Bug」或「做 Code Review 的人」。如果 SDET 的日常只是幫忙看代碼、做人工測試，那確實由 RD 互相 Review 效率更高。但這就像認為「SRE 的工作只是幫 RD 部署和寫開機腳本」一樣，是沒有看過完整工程體系的局限思維。

### 2. SRE 的類比：從「幫忙做事」到「建構基礎設施」
沒有體驗過現代工程團隊的人，常會覺得「為什麼需要專職的 SRE？部署跟環境我們 RD 自己弄不就好了？」
* 當團隊規模擴大、架構變複雜時，RD 自己做部署會遇到標準不一、監控缺失、容災架構無法全域考量等問題。
* **SRE 的價值**不是幫忙部署，而是**建構一套「讓 RD 能夠安全、快速、無痛部署」的基礎平台**。
* 同理，**SDET 的價值**不是幫 RD 做 Code Review 或手動測試，而是**建構一套「讓研發流程能夠自我驗證、快速反饋」的測試基礎設施（Test Infrastructure）**。
  - 例如：設計 CI/CD 測試流水線、自動化測試框架的封裝、測試資料生成平台、環境自動隔離與還原工具、測試成效可視化看板等。
  - 這本質上是 **DPE (Developer Productivity Engineering)**，是開發工具給開發者用，而不是代替開發者開發。

### 3. 「能抓 Bug = 功力更高深」的思維謬誤
* **思維模型的不同**：
  - **RD 開發思維**：關注「如何實現這個功能」（Happy Path），容易陷入路徑依賴與認知盲區。
  - **測試與品質思維**：關注「系統在什麼樣的邊界、異常輸入、極端負載或異步環境下會崩潰」。
* **看得懂代碼不等於懂測試**：能看懂別人的程式碼，跟「能設計出高覆蓋率、低維護成本的測試架構」是兩回事。寫錯程式碼有時只是粗心或局部細節沒考慮到，而設計測試策略需要的是**全域的風險評估能力**。

### 4. AI 時代下的「職能流動」與初階工程師的現實無奈
雖然理想上「不看職稱，只看能否解決特定問題」，但身為基層或初階工程師，我們必須面對現實的**組織牆與無奈**：
* **基層的現實侷限（呼應敏捷三叔公的觀察）**：我們並非決策者或管理層，無法直接重塑公司的組織架構、KPI 考核方式或 HR 的招聘標籤。在實務上，「職稱（QA/RD/SDET）」依然牢牢定義了我們的考績與薪資天花板，不可能說拿掉就拿掉。
* **分工的歷史意義**：組織當初劃分 RD 與 QA 的界線，是為了透過專門化降低個人的認知負荷，並提高特定環節的交付速度。這種分工在過去有其效率上的必然性。
* **變革下的生存焦慮**：在 AI 降低了編碼與解讀代碼門檻的今天，如果組織的分工依然僵化不變，團隊效率就會被過多的跨職能溝通（摩擦力）拖垮。身為基層，我們如果只死守著「職稱定義的舊邊界」，而不跟著時代調整能力，最終確實會面臨被邊緣化的淘汰風險。
* **基層的「隱形突破」策略**：
  - 既然無法改變組織的「職稱」，我們能改變的是自己「為團隊解決問題的實質射程」。
  - 在既有的職稱框架下，利用 AI 的加持，悄悄跨出一步去補足團隊的痛點（例如：以 QA 之名，行建構測試基礎設施、優化團隊流程之實，做一個「影子 SDET/SRE」）。這不是去推翻體制，而是在體制內用「實質影響力」為自己搭建護城河。

## 🚀 未來發文的思考與寫作方向

這是一個非常適合挑戰業界主流成見的題材，可以寫成一篇引人深思的職涯思辨文：

*   **引子**：引用網友「Java 大佬認為不需要 SDET，因為能抓 Bug 的人功力更高深」的爭議論點。
*   **第一層論點 (SRE 的類比)**：
    - 說明這是一種「沒吃過豬肉，只看過豬走路」的認知偏誤。
    - 用 SRE 類比 SDET：你以為的 SDET 是在幫你擦屁股（手動測試、肉眼找 bug），但實際上的 SDET 是在幫團隊蓋自動化防禦網（Test Platform / Infra）。
*   **第二層論點 (開發與測試的思維拆解)**：
    - 釐清「看懂程式碼」與「具備品質工程思維」的本質差異。
    - 舉例：廚師會煮菜（RD），與食品衛生稽核員（QA）知道如何系統性防範食物中毒，是完全不同的專業技能。
*   **第三層論點 (AI 時代的基層真實挑戰與生存策略)**：
    - **直面現實的無奈**：承認基層非管理職「無法直接改變組織職稱與分工」的現實。
    - **組織分工的雙面刃**：分析當初劃分分工的合理性，以及它在 AI 衝擊下面臨效率摩擦時的必然調整。
    - **影子工程師的逆襲**：基層的生存之道在於「不破壞職稱，但默默擴大解決問題的守備範圍」。利用 AI 賦能，以既有職稱在邊界之外默默建置測試基礎設施與優化流程，將自己鍛造成組織中不可或缺的「影子 SDET/SRE」與「溝通/技術樞紐」。
