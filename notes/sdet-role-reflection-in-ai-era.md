## 📌 核心反思筆記

### 1. 網友論點背後的「認知盲區」
* **網友的論點**：「Java 大佬認為測試跟 code review 應該是工程師之間交換做，不應該有 SDET 專職。因為你能看懂別人程式碼還知道他那裡寫錯 = 你的功力比別人高深（那為什麼不做 RD？）」
* **盲區解析**：這段話把 SDET（Software Development Engineer in Test）的職能**過度簡化**為「幫 RD 抓 Bug」或「做 Code Review 的人」。如果 SDET 的日常只是幫忙看代碼、做人工測試，那確實由 RD 互相 Review 效率更高。但這就像認為「SRE 的工作只是幫 RD 部署和寫開機腳本」一樣，是沒有看過完整工程體系的局限思維。

### 2. SRE 的類比：從「幫忙做事」到「建構基礎設施」
沒有體驗過現代工程團隊的人，常會覺得「為什麼需要專職的 SRE？部署跟環境我們 RD 自己弄不就好了？」
* 當團隊規模擴大、架構變複雜時，RD 自己做部署會遇到標準不一、監控缺失、容災架構無法全域考量等問題。
* **SRE 的價值**不是幫忙部署，而是**建構一套「讓 RD 能夠安全、快速、無痛部署」的基礎平台**。
* 同理，**SDET 的價值**不是幫 RD 做 Code Review 或手動測試，而是**建構一套「讓研發流程能夠自我驗證、快速反饋」的測試基礎設施（Test Infrastructure）**。
  - 例如：設計 CI/CD 測試流水線、自動化測試框架的封裝、測試資料生成平台、環境自動隔離與還原工具、測試成效可視化看板等。
  - 這本質上是 **DPE (Developer Productivity Engineering)**，是開發工具給開發者用，而不是代替開發者開發。

### 3. 「能抓 Bug = 功力更高深」的思維謬誤
* **思維模型的不同**：
  - **RD 開發思維**：關注「如何實現這個功能」（Happy Path），容易陷入路徑依賴與認知盲區。
  - **測試與品質思維**：關注「系統在什麼樣的邊界、異常輸入、極端負載或異步環境下會崩潰」。
* **看得懂代碼不等於懂測試**：能看懂別人的程式碼，跟「能設計出高覆蓋率、低維護成本的測試架構」是兩回事。寫錯程式碼有時只是粗心或局部細節沒考慮到，而設計測試策略需要的是**全域的風險評估能力**。

### 4. AI 時代下的「職能流動」與初階工程師的現實無奈
雖然理想上「不看職稱，只看能否解決特定問題」，但身為基層或初階工程師，我們必須面對現實的**組織牆與無奈**：
* **基層的現實侷限（呼應敏捷三叔公的觀察）**：我們並非決策者或管理層，無法直接重塑公司的組織架構、KPI 考核方式或 HR 的招聘標籤。在實務上，「職稱（QA/RD/SDET）」依然牢牢定義了我們的考績與薪資天花板，不可能說拿掉就拿掉。
* **分工的歷史意義**：組織當初劃分 RD 與 QA 的界線，是為了透過專門化降低個人的認知負荷，並提高特定環節的交付速度。這種分工在過去有其效率上的必然性。
* **變革下的生存焦慮**：在 AI 降低了編碼與解讀代碼門檻的今天，如果組織的分工依然僵化不變，團隊效率就會被過多的跨職能溝通（摩擦力）拖垮。身為基層，我們如果只死守著「職稱定義的舊邊界」，而不跟著時代調整能力，最終確實會面臨被邊緣化的淘汰風險。
* **基層的「隱形突破」與換公司決策（ESG 三級診斷框架）**：
  - 如果真的想往 SDET 發展，最快的路其實不是在原地感化主管或部門，而是認清自己與公司的期望不一致，直接換到一間有此工程土壤的公司。這就像**尋找一間本質上重視 ESG 的公司，而不是去改造一間「做做 ESG 樣子」的公司一樣**。
  - **經典的 ESG 三級診斷**：1. 第一級（玩真的）：老闆親信/二代帶頭，直接成立編制健全的部門；2. 第二級（生存剛需）：被歐美大客戶要求，被動全力配合；3. 第三級（試水溫）：老闆看了書想做點樣子，成立一兩人的功能小組但不給實質預算。
  - **資深顧問的名言**：「如果你今天想在 ESG 領域發光發熱，你自然知道該去哪一類公司。」對比軟體測試：也是看公司將品質視為核心競爭力（第一級）、生存危機剛需（第二級）、還是主管只是看書跟風試水溫（第三級）。
  - **賣關子（去識別化）**：診斷自己目前在第幾級就對號入座即可，避開直接掛鉤現職。當留在原地時，以既有職稱在邊界之外默默做工具（影子工程師）是累積跳槽實力或在現行架構生存的最務實防禦手段。這也呼應 Joe 在[〈如果不想上火星，你就別去SpaceX〉](https://www.darencademy.com/article/view/id/3579)所述的「選擇自由與理念相符環境」邏輯。

### 5. 如果團隊「連 SRE 都沒有」的基層求生指南
如果團隊處於第三級成熟度，連基本的 SRE 都沒有，環境、資料庫與流水線隨時在起火，此時盲目建構高大上的自動化測試就等於「在流沙上蓋城堡」。基層工程師的影子生存策略應調整為：
* **別蓋城堡，改建防空洞（用 Mock 隔離環境不確定性）**：如果測試總是卡在環境與第三方 API 的不穩定，果斷使用 Mock（如 Node.js Express/Python FastAPI）建立局部的穩定依賴，完成 90% 的功能驗證，拒絕無止境的跟隨環境起舞。
* **扮演最微小的「影子 SRE」（探針與通報自動化）**：寫幾十行的 Python 監控腳本定時 Ping 測試伺服器，並透過 Slack/Teams Webhook 自動通報異常。不需大架構，用主動監控通報把環境故障的排查溝通摩擦力降到最低。
* **把「環境阻塞時間」顯性化，讓主管為「沒有 SRE」買單**：在測試日誌或 Sprint 報告中，拉出獨立指標「測試阻塞率（Block Time Rate）」（例如：5 天測試時間有 3 天因為環境掛掉而阻塞，實際阻塞率 60%）。用數據將環境欠債顯性化，把「測試慢」的黑鍋還給基礎設施缺失。
* **以戰養兵的跳槽心態**：在荒蕪之地默默實踐 Mock 與主動監控，將生存掙扎化為履歷上最亮的 SDET 實戰故事，存夠能量後，主動換到有合適土壤的公司。

## 🚀 未來發文的思考與寫作方向

這是一個非常適合挑戰業界主流成見的題材，可以寫成一篇引人深思的職涯思辨文：

*   **引子**：引用網友「Java 大佬認為不需要 SDET，因為能抓 Bug 的人功力更高深」的爭議論點。
*   **第一層論點 (SRE 的類比)**：
    - 說明這是一種「沒吃過豬肉，只看過豬走路」的認知偏誤。
    - 用 SRE 類比 SDET：你以為的 SDET 是在幫你擦屁股（手動測試、肉眼找 bug），但實際上的 SDET 是在幫團隊蓋自動化防禦網（Test Platform / Infra）。
*   **第二層論點 (開發與測試的思維拆解)**：
    - 釐清「看懂程式碼」與「具備品質工程思維」的本質差異。
    - 舉例：廚師會煮菜（RD），與食品衛生稽核員（QA）知道如何系統性防範食物中毒，是完全不同的專業技能。
*   **第三層論點 (AI 時代的基層真實挑戰與生存策略)**：
    - **直面現實的無奈**：承認基層非管理職「無法直接改變組織職稱與分工」的現實。
    - **組織分工的雙面刃**：分析當初劃分分工的合理性，以及它在 AI 衝擊下面臨效率摩擦時的必然調整。
    - **影子工程師的逆襲**：基層的生存之道在於「不破壞職稱，但默默擴大解決問題的守備範圍」。利用 AI 賦能，以既有職稱在邊界之外默默建置測試基礎設施與優化流程，將自己鍛造成組織中不可或缺的「影子 SDET/SRE」與「溝通/技術樞紐」。

## 🎵 點綴歌曲：Passenger〈Holes〉（2012）

**為什麼是這首**：`holes` = 漏洞 = bug。**QA 一輩子在找別人系統的 holes，但這首歌講的是人生的 holes**——職涯的洞、被淘汰焦慮的洞、組織牆留下的洞。這個雙關剛好坐落在第三層（基層無奈 → 影子工程師）的情感核心。

**三句最扣主題的歌詞 + 對應論點**：

| 歌詞 | 對應這篇的論點 |
|---|---|
| *"When you've got nothing, you've got nothing to lose"* | 基層沒有決策權、改不了職稱 / KPI——但「nothing to lose」也是解放 → 影子工程師策略：反正改變不了體制，那就在體制內默默擴大守備範圍 |
| *"sometimes it seems you gain less than you lose"* | AI 時代的 QA：學了 AI、跨了職能，但職稱天花板還在，gain 可能 less than lose——**誠實面對這個，不灌雞湯** |
| *"Where we've got holes, we've got holes / But we carry on"* | 結尾的韌性。不是悲觀，是**承認洞的存在後依然前行** = 影子工程師的逆襲 |

**建議擺法：頭尾包夾**（比單放一處有力）
- **開頭 hook**：用 `holes = 漏洞` 雙關切入——「我的工作是替別人的系統找漏洞（holes），但有時候我覺得自己的職涯也破了一個洞。」立刻把技術題拉進個人 stake
- **結尾 coda**：用 *"but we carry on"* 收——承認所有的洞（組織牆、職稱天花板、淘汰焦慮）之後，影子工程師依然 carry on。呼應 [not-just](/post/not-just-make-manual-script-automate/) 的歌詞 coda 模式
- 引用節制：各引 1-2 句 + 完整出處（Passenger〈Holes〉, 2012），避免大段轉貼版權風險

**調性提醒**：這首歌偏 melancholic-but-resilient。配這篇剛好——但**別讓「洞」的悲傷蓋過「影子工程師逆襲」的主動性**。歌負責情感共鳴、論述負責給出路，兩者分工。⚠️ 也對照 [寫作風險備忘 / 線六超載]：這篇本身偏線九/線六，情緒濃度要控制，別寫成「QA 好可憐」的自憐文——主軸是「**看清現實 + 主動搭護城河**」。
