最近在 Ministry of Testing (MoT) 論壇看到一個非常值得深思的討論：**「我們是否跑了比實際需求更多的回歸測試？」**

在自動化測試的領域，我們很容易陷入「數量迷思」。當腳本從 100 條增加到 1000 條時，我們可能覺得覆蓋率更高、更有安全感，但事實往往是：維護成本呈倍數增長，而發現關鍵問題的效率卻在下降。

我深深的體悟是：**刪除一個測試案例比建立一個測試案例，技術上容易，但心理上困難很多。**

### 1. 想刪，但誰敢刪？

- 你有沒有遇過一個從來沒失敗過的測試，後來發現它根本在測一個已經下架的功能？
- 你有沒有想刪某個測試卻不敢刪的經驗？
- 你團隊裡有沒有發生過「為了衝數量」寫出低價值腳本的情況？

這些場景的背後都指向同一個問題：**沒有人為這些測試「當責」。**

如果一個測試案例失敗了，卻沒人知道它當初為什麼被寫下來；如果一個測試永遠綠燈，卻沒人有權限決定它是否該被刪除——這個測試就不再是品質保障，而是**技術債**。

論壇建議每個測試都應由專人負責，這在大型專案中尤為重要。但現實是，在職場中提出「我們來精簡一下吧」的人，往往就是那個被指派去做的人。要想優化自己的工作，還是得需要一點勇氣。

### 2. 測試不是越多越好，而是要「有理可循」

論壇中有個觀點深得我心：**關鍵不在於減少數量，而是確保每個測試都「切實可行」，並且「有存在的理由」。**

在我的實務觀察中，很多無意義的回歸測試來自於：
* **過期的規格**：功能已經改版，但舊的測試腳本還在跑「過時的預期結果」。
* **重複的驗證**：在 UI 層級重複驗證了單元測試（Unit Test）早已覆蓋的邏輯。
* **為了指標而測試**：為了達成主管要求的「腳本數量」或「覆蓋率」績效，而產出了大量低價值的綠燈。

### 3. 實踐中的改變：從「數量」轉向「韌性」

作為一名初階但有追求的 QA，面對堆積如山的回歸測試，我們能做的一小點改變是：

**在新增任何一條自動化腳本前，先問自己四個問題：**
1. 如果這個測試失敗了，我能立刻知道是哪層邏輯出錯嗎？（可解釋性）
2. 這個功能點是否已經在其他層級（如 API 測試）驗證過了？（重複性）
3. 我是否有信心在 UI 變動時，快速修復它而不讓它變成 Flaky Test？（維護性）
4. 如果這個測試明天消失了，會有人注意到嗎？（存在感）

### 結語：斷捨離的代價，由誰來扛？

回歸測試不應該是 QA 的心理安慰劑，而應該是精準的導航雷達。與其跑 1000 個「可能有用」的測試，不如確保那 100 個「核心測試」每次都能精準地抓到 Bug。

但道理人人都懂，真正難的是：**誰來推動這件事？**

現在 2026 四月，距離我 2023 三月退伍已經悄悄過了三年。退伍前的那兩個月我在某個桃園陸軍營區，主要在幫忙輔導長做些政戰的差事。那個職位的行政瑣事並不少，輔導長常常加班加到周末，我有思考過為什麼沒有人去精簡這些基層的冗長行政流程，但最後的答案是——這不一定符合上頭的利益。不知道那位輔導長是否順利二十年告退去開大貨車了。

推動斷捨離不只在職場會遇到阻力。四年前，我待的教會每次開預算規劃會議，常常討論得很冗長、沒有結尾，或是聊天聊太久。後來有人推動改變，順利推進成我們這個世代比較想要的樣子——開會就只討論議題，一個小時內就結束了。但那段過程也造成了教會幾個青年人流失，對於十幾人的教會，每一個人的離開都是巨大的損失。

改變會有代價。精簡流程不一定會滿足上頭的利益，還可能會踩到既得利益者的腳，刪掉測試會讓習慣「看綠燈安心」的人不舒服。

身為一個基督徒，我期待能夠像約瑟那樣有智慧地處理手上的工作，常常思考怎麼樣聰明地工作。如果現有的大量回歸測試是大家的心理安慰劑，我還算期待自己能夠成為這個部門或是跨部門的心理安慰劑——不是靠數量給人安全感，而是靠判斷力。

具體該怎麼做？我可能會像大人學Podcast給的建議一樣，開始重視主管、其他部門他們可能會重視什麼利益，然後盡量將這些跟「精簡回歸測試」連結在一起吧。

你願不願意當那個站出來推動改變的人？還是你還在猶豫？我自己也還在猶豫，但至少開始想了。
