## 前言：那個熟悉的「來自動化吧」場景

> **這節要做的事**：用一個畫面把讀者拉進來。場景越具體越好。
>
> **建議寫法**：
> - 開頭一個情境：主管說「我們開始做自動化吧」 / Sprint 規劃時被指派「把這 200 條 case 轉成自動化」 / 某次回歸跑了兩天痛苦到極限
> - 接著的「自然反應」：拿手動 case 清單一條一條翻成 Selenium / Appium 腳本
> - 三個月後的現實：腳本 flaky、維護成本高、發現的 bug 沒幾個
> - 拋問題：**「我們是不是搞錯了什麼？」**
>
> **金句方向**：
> > 「我們以為自動化是『把手動測試的速度加快』，後來才發現——它根本是另一種測試。」

（寫作區）

## 一、為什麼業界都這樣做？

> **這節要做的事**：先替「直接轉換」這個做法說公道話——不是它愚蠢，是它有合理的吸引力。**這節要寫得溫和，才有資格在下一節挖坑**。
>
> **三個原因參考**：
> 1. **看起來最直觀**：手動案例已經寫好了，照搬就能跑——零思考成本
> 2. **ROI 容易計算**：「以前要兩天、現在自動跑」——主管一眼看懂
> 3. **不需要重新設計**：免去「該測什麼」的二次決策，把節省下來的時間放在「該寫腳本了」
>
> **小結方向**：這條路「上手快」是它的優點——但這個優點同時是它最大的陷阱（鋪墊下一節）

（寫作區）

---

## 二、為什麼這條路會崩盤？三個結構性原因

> **這節是論述核心**。三個原因要寫得具體、可驗證，不能只是抽象主張。

### 1. 手動案例是「點 X，看 Y」——但「看 Y」往往依賴人類視覺判斷

> 想想你寫過的手動案例：步驟很容易翻譯（「點按鈕 A」、「輸入 X」），但**驗證點往往是「畫面看起來對」、「彈窗不奇怪」、「動畫順暢」**——這些自動化做得到嗎？
>
> **🤔 Q1（替整篇文章找一個活體案例）**：
> 你公司目前**最 flaky 的自動化腳本**是哪一條？它的 assertion 是視覺型（「按鈕變色了」「畫面對齊了」）還是結構型（「API 回 200」「DB status 變了」）？
>
> 把這個案例的細節寫進來——讀者一秒理解、文章可信度 +10。

（寫作區）

### 2. 自動化做視覺判斷既慢又脆——assertion 只能挑結構性事實

> 即使你用了 image diff、視覺 regression 工具——這類驗證仍然是**最不穩定、最容易誤報、CI/CD 跑最慢**的部分。
>
> 而真正能被機器穩定 assert 的東西，其實是另一組：API response、DB row、cookie/storage、URL、事件 tag、log 內容⋯⋯
>
> **🤔 Q2（找一個對比案例）**：
> 你能舉一個「**手動花 30 分鐘判斷得很好、但自動化跑 10 秒就漏掉**」的案例嗎？
>
> 例如：UX 順不順、彈窗時機是否恰當、動畫過渡是否突兀、頁面載入時的視覺跳動⋯⋯這些手動測試員一眼看出來，但自動化腳本即使「全綠」也測不到。

（寫作區）

### 3. UI 變動頻率 >> 業務邏輯變動頻率

> 前端 UI 每兩週改版一次很正常，但底下的 API contract、DB schema、業務規則往往**幾個月才動一次**。
>
> 把腳本綁在 UI 上 = 把腳本綁在最易變的那一層 = 每次 UI 一動，腳本就 break。
>
> **結果**：維護腳本的時間 > 跑腳本省下的時間。**這就是 ROI 崩盤的數學原因**。

（寫作區）

## 三、本質差異：兩者的稀缺資源不一樣

> **這節要拋出整篇文章的核心框架**——前面是現象，這節給理論。
>
> **核心對比**：
>
> | 維度 | 手動測試 | 自動化測試 |
> |---|---|---|
> | 稀缺資源 | **人的判斷時間** | **腳本維護成本** |
> | 擅長 | 探索、判斷、視覺、邊界發現 | 重複、批次、確定性、結構驗證 |
> | 失敗模式 | 漏測、疲倦、不一致 | flaky、過期、維護債 |
> | 失敗的代價 | 一次性 | **複利累積** |
>
> **金句方向**：
> > 「不該用便宜資源（自動化的腳本時間）做需要稀缺資源（人類判斷）的事；
> > 也不該用稀缺資源（測試員的注意力）做機器能可靠完成的事。」
>
> **這節是論述的「轉折點」**——前面在說現象，這節在說「為什麼必然會這樣」。

（寫作區）

## 四、該怎麼分工

> **這節要給讀者帶得走的清單**。前三節是理論，這節是 actionable。

### 手動測試應該專責的

> - **探索式測試 (Exploratory Testing)**——PDT 的本職，找未知 bug
> - **邊界與異常路徑**——時間、數量、權限切換、網路中斷時的視覺反應
> - **UX 判斷**——順、卡、嚇人、視覺對不對、動畫合不合理
> - **重要功能的關鍵 path**——人盯著看比較放心（金流、登入、付款）
> - **新功能的第一次驗證**——還沒穩定到值得寫腳本

（寫作區）

### 自動化應該專責的

> **🤔 Q3（清單具體化）**：
> 「結構性資料驗證」具體包含什麼？我的預設清單：
>
> - **API response 驗證**（status code、payload 結構、欄位值）
> - **DB 狀態檢查**（row 存在 / 不存在、欄位值正確、count 正確）
> - **Cookie / Storage / URL** 的一致性
> - **事件追蹤 tag** 是否正確埋設（GA、AppsFlyer、自家 log）
> - **Smoke test**（每次 deploy 跑一輪確認核心 API 還活著）
> - **跨環境 / 跨裝置回歸**（自動化擅長批次跑）
>
> 你想到的還有什麼？或者哪幾項在你公司的場景特別重要？這份清單越具體，讀者越能對照自己的工作。

（寫作區）

## 五、怎麼從現狀切換（不是砍掉重寫）

> **這節要解決讀者讀完的「對，但我現在怎麼辦？」**焦慮。
>
> **核心訊息**：不是一次全砍重寫，是漸進清理。
>
> **三個「砍掉訊號」**：
> 1. **常 flaky** → 砍掉，重要的話改回手動
> 2. **assertion 大部分是視覺判斷** → 砍掉，改成 API 層測試
> 3. **已經被手動跑了一輪當保險** → 表示你自己都不信任，砍掉
>
> 騰出來的時間：建結構性驗證腳本（API、DB、tag）。
>
> **🤔 Q4（數據支撐）**：
> 你公司現有 UI 自動化腳本，如果按「該砍 / 該留 / 該轉成 API 測試」分類，**比例大概是多少**？
>
> 不用精確——「目測 6 成該砍、2 成該留、2 成該轉 API」這種粗估就很有說服力。

（寫作區）

## 結語：自動化不是手動的「快版本」，是不同物種

> **這節要給整篇文章一個記憶錨點**。
>
> **可能的收尾結構**：
>
> 1. **回到開頭**：那個「來自動化吧」的場景——下次再遇到，先別急著翻譯 case
> 2. **重申核心**：自動化跟手動不是同物種的快慢版，是**兩種不同的測試**
> 3. **拋一個給讀者的問題**：你公司現在的自動化腳本，有幾條是真正在驗證「人類做不到的事」？有幾條只是「手動 case 的廉價影印機」？
> 4. **呼應 PDT**（可選）：人類的稀缺資源是**判斷力**——自動化的核心不是「跑得快」，是**把該機器判斷的事，精準定義後交給機器**。
>
> **金句候選**：
> > 「該不該自動化」這題的真正答案，不是「值不值得」，是「**它本質上是不是結構性可驗證的？**」
>
> > 「自動化不是讓測試變快——是讓測試員的注意力，不用花在機器該做的事上。」

（寫作區）

## 不要寫進去的東西（紅線）

- ❌ 不要說「業界都做錯」這種大話——說「最常見的浪費」就夠了
- ❌ 不要點名任何具體工具的爛（Selenium 不爛、Appium 不爛——是用法錯了）
- ❌ 不要寫成「自動化沒用 / 退回手動」——文章主張是**分工**，不是退步
- ❌ 不要過度量化（「我估計 80% 的腳本該砍」這種沒數據的數字很危險）
- ❌ 不要在這篇放任何主管 / PM 的影子——這篇是純技術主張，不該夾雜組織問題
