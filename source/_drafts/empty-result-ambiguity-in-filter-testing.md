## 前言：那個「查無資料」的瞬間

你寫了一個自動化測試，跑「篩選某個條件」的功能。

腳本點下篩選、等載入完成、抓結果列表。然後 UI 顯示「查無資料」。

問題來了——**這個「空」是 pass 還是 fail？**

它可能是兩種完全不同的東西：

| 種類 | 描述 | 該回什麼 |
|---|---|---|
| ✅ 真空 | 篩選邏輯正確，資料源剛好沒有符合條件的項目（合理的查無資料） | `pass` |
| ❌ 偽空 | 篩選 bug、後端錯誤、UI 沒渲染 | `fail` |

兩種「空」在 UI 上**長得一模一樣**，自動化沒辦法直接分辨。

最直觀的處理是回 `沒成功也沒失敗`，把它丟給人工 review——但這等於放棄自動化的精神。每次 release 都要花人時、不同人會有不同判斷、release cycle 變慢。

這篇文章想拆的是——**遇到 empty 歧義，QA 有哪些路可以走？怎麼選？**

---

## 一、Empty 歧義為什麼難解

先講為什麼這個問題這麼煩人：

1. **單純看前端篩選後的結果無法判斷** — 結果是空集合，沒有東西可以驗證對錯
2. **沒有 ground truth（真實基準）** — 你需要一個獨立的真相來源來告訴你「空到底合不合理」
3. **後端跟前端常常各執一詞** — 後端說我回對了、前端說我顯示對了，bug 卻在中間某處
4. **歷史資料不重複** — 同樣的篩選條件昨天有十筆、今天可能就 0 筆，沒辦法用 snapshot（快照）比對

這個歧義在很多 domain 都會遇到：

- 訂房平台篩選「五星級 + 某地區 + 某價位」
- 電商搜尋「某品牌 + 某價位區間」
- 求職網站「某類型 + 某薪資範圍」
- 金融資訊 APP 篩選「漲跌幅 ≥ 某個百分比」

只要你的功能是「**對一個會變動的資料集做條件篩選**」，這個歧義就會找上你。

## 二、五個方案，從理想到實用

### 方案 1：後端直接交叉比對（理想，但成本高）

最乾淨的做法——直接打後端 API 抓「**符合此條件的所有資料**」，跟 UI 顯示比對：

```python
expected_count = api.count_items_matching(condition)
if expected_count == 0:
    return 'pass'   # 後端也沒有 → 真空
else:
    return 'fail'   # 後端有 N 筆但 UI 空 → 偽空（UI bug）
```

**優點**：100% 客觀、權威、能完全消除歧義

**缺點**：
- 需要把每支被測 APP 的後端 API、參數對應、欄位映射全部整理出來——這是長期基礎建設
- 內部 API 通常有權限、版本、欄位差異等問題，更別提外部 API 可能距離 QA 更遠更無法控管
- 短期內做不到時，這個方案只能當長期目標

### 方案 2：用「外部領域信號」做合理性 heuristic（啟發式判斷）

如果你的 APP 所在的領域有一個**外部信號**可以用來推測「empty 合不合理」，可以用它做廉價的 heuristic。

舉個具體例子（金融資訊 APP）：

| 篩選條件 | 外部信號（市場大盤方向） | empty 是否合理 |
|---|---|---|
| 漲幅 ≥ 5% | 大盤下跌 | ✅ 合理（市場跌的時候，漲幅高的個股自然少） |
| 跌幅 ≥ 5% | 大盤上漲 | ✅ 合理（同上反向） |
| 任何「≥ 0%」系列 | 任何 | ❌ 永遠不合理（任何時候都該有個股漲跌幅 ≥ 0%） |
| 其他組合 | — | 保守視為不合理 |

寫起來很短：

```python
def is_filter_empty_plausible(filter_condition, external_signal) -> bool:
    if condition_is_trivially_satisfiable(filter_condition):
        return False  # ≥ 0% 這種條件永遠不該空
    if signal_aligns_with_emptiness(external_signal, filter_condition):
        return True
    return False
```

**優點**：
- 用得到的訊號往往本來就在 UI 上（例如金融 APP 的市場大盤指標），不需要新 API
- 立即可用、成本極低

**缺點**：
- 邊界場景會 false pass（例如大盤漲只一點點 + 篩選條件嚴格，理論上邊緣個股仍可能存在）
- 只適用於**有方向性**的篩選條件（漲跌幅、上升下降、趨勢類）
- 對「無方向性」的條件（價格範圍、品牌、地點）完全沒幫助

### 方案 3：原本驗證就好（放棄解 empty 歧義）

UI 空時直接回 `fail`——預設「篩選後空」就是錯。

**優點**：簡單、能抓「整個資料源壞掉」這種極端情況
**缺點**：把所有真空 case 都誤殺成 fail。實務上不能用。

### 方案 4：`沒成功也沒失敗` + 人工 review（保守）

加上詳細訊息（篩選條件、外部信號、時間戳）丟給人工判斷。

**優點**：保守、不會 false pass / fail
**缺點**：違背自動化精神。每次 release 都要花人時，不同人會有不同判斷。

### 方案 5：用「同一頁面的無篩選全量」做 client-side cross-check（前端交叉驗證，巧妙）

對於**沒有方向性的數值條件**（價格、金額、市值範圍）——方案 2 的 heuristic 用不上，方案 1 的後端 API 又沒接好。怎麼辦？

換個角度：**用 APP 自己的「無篩選狀態」當 ground truth**，跟客戶端篩選結果比對：

```python
def verify_filter_consistent_with_unfiltered(filter_name, expected_codes_fn):
    actual_codes = set(get_current_result_ids())     # 1. 抓篩選後
    set_filter_to_none(filter_name); save()          # 2. 取消無篩選
    all_items = get_current_result_list()            # 3. 抓全部
    expected_codes = expected_codes_fn(all_items)    # 4. caller 自訂判斷邏輯
    return actual_codes == expected_codes            # 5. 集合相等
```

`expected_codes_fn` 接整份 `dict[id, info]`，caller（呼叫端）自己寫 predicate（判斷式，例如 `price <= 10`）。

**優點**：
- **不需要後端 API**——用 APP 自己驗 APP，邏輯閉環
- **雙向驗證**：能抓 false positive（多顯示）+ false negative（漏顯示）+ true empty（兩邊都空）
- 完全消除 empty 歧義
- 雙向相等比單向「actual ⊂ expected」更嚴謹

**缺點 / 注意事項**：
- 每個 case 多一次「取消篩選 + 抓完整列表」的開銷
- 副作用：執行後篩選會變成「無篩選」狀態，下個 case 要記得 reset
- 「無篩選」是同一頁所有資料，**不是整個資料源**——但對「篩選正確性」驗證已足夠

## 三、單向驗證 vs 雙向驗證——一個容易混淆的選擇

實作方案 5 時遇到一個重要的設計決策。先放一張對照表：

| 驗證方式 | 描述 | 抓得到的 bug |
|---|---|---|
| **單向驗證**（actual ⊂ expected） | 只檢查 actual 中每筆是否符合條件 | False positive（多顯示） |
| **雙向驗證**（actual == expected） | 檢查兩個集合是否完全一樣 | False positive + false negative（漏顯示） |

原本驗證設計是單向：「APP 顯示的要符合條件」。但方案 5 的實作天然是雙向的——它順便把驗證升級了。

**這裡藏一個 scope creep（範圍蔓延）警告**：解 empty 歧義 ≠ 雙向驗證。它們是**兩個獨立的決策**，常常被綁在一起做。雖然順便升級不一定壞，但設計時要意識到自己同時做了兩個決定。

我們最後接受了雙向升級，因為：

- 漏顯示的 bug 在原本的單向驗證下抓不到
- 雙向驗證的成本邊際很小（只是把 `⊂` 改成 `==`）

但下次設計類似測試時，**請先把這兩個決策分開思考再合併**。

## 四、為什麼最後是「方案 2 + 方案 5」並用

最後的選擇——

- **有方向性的篩選條件**（漲跌幅、增加減少、上升下降類） → **方案 2（外部信號 heuristic）**
- **無方向性的數值條件**（金額、範圍、容量類） → **方案 5（無篩選 cross-check）**
- **長期目標**（基礎建設到位後） → **方案 1（後端交叉比對）**

理由：

1. **方案 1 最權威，但被「對應每支 APP 後端參數」這個前置工程擋住**——短期做不到
2. **方案 2 適用於有方向性的篩選**——外部信號可以當 heuristic 信號，便宜可用
3. **方案 5 適用於無方向性的篩選**——外部信號用不上，但 APP 自己的「無篩選狀態」可以當 ground truth
4. **方案 3 太邊緣**（資料源整個壞才會抓到）
5. **方案 4 是現狀**，被改善的就是它
6. 三個方案**不衝突**——未來方案 1 上線後，方案 2/5 還可以當第二道防線（API 失敗時 fallback）

## 五、副題：PageObject 該不該住 verify_* 方法？

實作方案 5 時遇到一個 PageObject 設計問題：`verify_volume_filter(threshold)` 這種「直接給 pass/fail 結論」的方法，**該住在 PageObject 裡嗎？**

兩種立場：

| 立場 | 主張 | 理由 |
|---|---|---|
| **純 POM 派**（嚴格） | Page 只該封裝「頁面操作 + 狀態查詢」，不負責測試邏輯 | POM 描述「頁面長什麼樣」，不是「測試怎麼判斷對錯」——後者屬於 test layer（測試層） |
| **實用派**（寬鬆） | POM 可以提供 verification helpers（驗證輔助方法）是合理的「商業邏輯封裝」 | 避免每個 case 重複輪子（DRY），test script 保持宣告式 |

我們選擇實用派，理由：

1. **既有 convention（慣例）** — 專案早已有 `verify_*` 系列方法
2. **case script 規模考量** — 80+ 個 case 不可能每個都重複 10 行 verification 邏輯
3. **目前還夠** — PageObject 的方法數還沒多到難維護

但有一條**判斷紅線**：

> **PageObject 不該知道「100」「10 元」這種特定 case 的 magic number（魔術數字／寫死的常數）**。但「給我 threshold，我幫你驗整個流程」是 OK 的——這是把「**怎麼驗**」標準化，不是把「**驗什麼**」寫死。

具體：

| 可接受 ✅ | 不該放 ❌ |
|---|---|
| `verify_volume_filter(threshold)` | `verify_volume_above_100()` |
| `verify_price_filter(condition, threshold)` | `verify_price_le_10()` |
| `verify_filter_consistent_with_unfiltered(name, predicate_fn)` | `verify_c050_passes()` |

**未來重構觸發點**（任一條件成立就重新評估）：

- PageObject 方法數超過 50 個
- `verify_*` 方法佔 PageObject 比例超過 40%
- 出現「同一個 verify 方法被超過 5 個其他方法依賴」的警訊
- 多個 caller 需要不同的「驗證策略」（strict vs loose）但用同一個 PageObject

任一條件成立 → 考慮 extract 成獨立的 `Verifier` 類別。

## 六、學到的事（meta-lessons）

排除技術細節，這次決策真正的價值在這幾條 meta-lesson：

### 1. Scope creep 要警覺

我以為在解「empty 歧義」（方案 5），結果順便也做了「雙向驗證升級」。看似都對的兩件事，會讓決策難以追溯。**下次出現「順便做了 X」，先停下來問自己：這是同一個決策嗎？**

### 2. 假設要驗證

原本假設「filter bug 通常是後端條件下錯」，後來才意識到——**這個 APP 的篩選是 client-side 邏輯**。bug 模式因此完全不同：不是 SQL where clause 錯，是 UI 控制元件 mapping 錯、欄位解析錯、跨閾值的邊界錯。

**設計測試前，先確認系統實際運作方式**。否則你會用對的方法解錯誤的問題。

### 3. 工具 vs 認知是兩個瓶頸

組合爆炸（多種篩選 × 多閾值 = 數千組）這個瓶頸**有工具解**：等價類劃分、邊界值、pairwise testing（配對測試）。

但「**哪些組合該測**」這個瓶頸**沒有工具能解**——它需要 QA 對風險、對使用者、對歷史 bug 模式的判斷。

工具能擴大產能，但**選擇是 QA 的職責**，這部分還沒有 AI 可以代替。

### 4. Feature flag（功能開關）是策略空間

一開始不確定要不要做雙向驗證，覺得「設計空間留下來」比立即實作或永久砍掉都更好。**用 feature flag / config 預留切換能力，是測試框架設計裡很值的投資**。

### 5. YAGNI（先別做還用不到的事）對 cache 的克制

timing 數據出來前以為要加 cache（怕跑得慢）。等實際跑了才確認哪些調用真正成為瓶頸——大多數情況**沒有瓶頸**。

**沒有測量過的優化都是早優化**。

## 七、結語

「篩選結果為空」這種看起來最不起眼的測試 case，**其實是測試設計的好試金石**。

它逼你回答：

- 你的 ground truth 從哪裡來？
- 你的驗證是單向還是雙向？
- 你的 PageObject 邊界在哪？
- 你解的是「empty 歧義」這個問題，還是順便升級了驗證強度？

每一題都不是「跑案例」這個行為的一部分——它們是**測試設計的判斷題**。

> 寫測試的核心姿態，從來不是「我把所有 case 都跑了」，而是「**這個結果為什麼可信？我憑什麼相信我的綠燈？**」

當你下次盯著一個自動化跑出來的「pass」，問自己這個問題。如果答得出來、答得清楚——這份綠燈才是真的綠燈。
