> 這是我準備參加鐵人賽的 buffer 稿之一，先發再改。

「App 壞了」大概是測試工作裡最容易被誤判的一句話。畫面一片空白、測試找不到元素、跑一跑就當掉——你很容易脫口而出「這版有問題」。但很多時候，壞的根本不是 App，是別的東西頂替了罪名。

QA 真正的價值，往往不在「回報壞了」，而在**準確地指出：到底是哪一層壞了。** 分享三個我實際踩過、最後發現「不是 App 的錯」的案例，以及它們背後的診斷分層。

---

## 案例一：整片黑畫面，卻不是 App 崩潰

有一個列表很重、畫面繪製吃資源的 App，在模擬器上打開後整片黑，大約二十秒後跳「應用程式無回應」。直覺結論：這版 APK 壞了。

但一條條看 log，事實完全相反：

- 後端 API 回 **200**，資料正常送達 → 不是網路 / 後端問題
- **沒有** FATAL EXCEPTION、沒有 crash 堆疊 → 程式碼沒有真的崩潰
- 記憶體充足 → 不是記憶體不足（OOM）
- 唯一異常：一堆 **GPU 繪製相關的錯誤**（`EGL_BAD_MATCH` 之類）

拼起來：後端 OK + 沒有例外 + 不是 OOM + 只有 GL 繪製錯 → 這是**模擬器的 GPU 繪製撐不住**，不是 APK 有問題。換一台**真手機**（有真正的 GPU）跑，畫面完整、一切正常。

**判準**：繪製吃資源的畫面（複雜列表、大量動畫），在用軟體模擬 GPU 的環境裡特別容易踩雷。這種「黑畫面」是環境問題，不是產品缺陷。

---

## 案例二：偶爾找不到元素，卻不是當機

另一個 App，連續跑一整輪測試時，某個步驟三不五時失敗一次，訊息是「找不到頁面元素」。但單獨跑那一步，永遠重現不出來。

關鍵線索：

- App 的行程還活著（不是 crash）
- 失敗當下的截圖，畫面是一個**客服 / 活動網頁**，不是 App 本身

追 log 才發現：是 App 自己在某個頁面觸發了行銷推播，彈出一個覆蓋整個畫面的網頁。測試不是「找不到元素」，是**元素被一層它不預期的東西蓋住了**。

更有意思的是：為什麼**只在連續跑**時發生？因為推播的投放條件是「使用者累積走過幾個頁面」——自動化連跑得太快，把那個累積條件湊滿了，於是推播在半路 fire。單獨跑一步，湊不滿，自然不重現。

**判準**：「單獨跑不重現、連續跑才壞」本身就是線索——它指向**累積狀態**，而不是隨機。（這也是我另一篇想寫的主題：flaky 不是隨機，是還沒被解讀的訊號。）

## 案例三：本地能跑，CI 不能——不是腳本錯

同一份腳本，在我筆電上能跑完登入，搬到共用的 CI 機器上就卡在登入頁。很容易懷疑「腳本寫壞了」，但兩邊腳本一字不差。

根因是**登入頁跑在不同的瀏覽器容器**：我本機的環境讓 App 走內嵌網頁（測試工具能操作），CI 機器的環境讓它走另一種外部瀏覽器分頁（測試工具碰不到）。差異純粹來自兩台機器的瀏覽器環境不同，跟腳本無關。

**判準**：本地能跑、CI 不能，第一個要問的不是「我腳本哪裡錯」，是「**這台機器跟我本機，哪裡不一樣**」。

---

## 把它收成一張診斷分層

遇到「壞了」，先別急著下結論，照這個順序問一輪：

| 層 | 要問的問題 |
|---|---|
| **後端 / 資料層** | API 回幾？資料有沒有送到？ |
| **App 邏輯層** | 有沒有 FATAL EXCEPTION / crash 堆疊？行程還活著嗎？ |
| **資源 / 環境層** | 是不是 OOM？有沒有 GPU / 繪製錯誤？換一台（尤其真手機）重現嗎？ |
| **覆蓋層** | 畫面上是不是有別的東西（推播、系統對話框、客服頁）蓋住了？ |

最快、最有效的一招，其實是最後那句：**換一台跑跑看**——尤其換到真手機、真 GPU。環境變因一換，環境問題當場現形，你就不會白白把時間花在「修一個根本沒壞的 App」上。

---

「App 壞了」是結論，不是觀察。觀察是「畫面黑了」「找不到元素」「登入卡住」。從觀察到結論之間，隔著好幾層可能性——而**願意一層層排除、而不是急著把罪名安給最顯眼的那個嫌疑犯**，大概就是測試這份工作，最像偵探的地方。
