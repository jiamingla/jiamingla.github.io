## 預設讀者（2026-05-13 確定）

**寫給 QA 看**，而且**預設讀者的測試結果只有 `pass` 跟 `fail` 兩種狀態**——這是業界大多數 QA 團隊的現況。

不能假設讀者已經有 `notest` / `skipped` / `inconclusive` 的詞彙。我自己團隊有 `notest`，但它在文章裡會降級成**反例案例**（「我們團隊試著加了第三狀態，但仍然不夠」），而不是論述的主軸。

這個讀者假設改變了整篇文章的弧線——**從「我們的 notest 為什麼亂」變成「你的 pass/fail 為什麼很快會不夠」**，半徑變廣，立論變強。

## 寫作風險備忘（重要）

**走 B 路線（公開＋去個人化）**——不寫個人自我揭露版本。理由是同公司不同產品的同事會讀到這篇，個人化的「我用 notest 包裝不想動手」會被對號入座，影響職場觀感。

去個人化處理：
- 「應測但測不想」這種用法 → 框架為「我觀察到 QA 圈普遍存在」，不寫成自己的承認
- 同公司其他團隊的 schema 對照 → 框架為「自發秩序」（schema 從來不是中央決定），**不**寫成「他們比我們好」
- 自己團隊的 schema 問題 → 中性指出，避免貶低調性

如果某天願意寫誠實版，再開一份 `notest-motivation-honesty-DRAWER.md` 抽屜文。

## 緣起（觸發點，不直接當文章 hook）

2026-05-13，主管對同事提到他觀察到的現象：

> 「AI 跑測試的結果有很多 `notest`，到底是 AI **跳過沒測**，還是它**信心不夠不敢下判斷**？」

我意識到三件事：

1. 這跟我手上正在寫的 [empty 歧義](/post/empty-result-ambiguity-in-filter-testing/) **是同一個母題**（「中間態被當成終態」）的兩個變奏
2. **我自己團隊有 `notest`，但業界大部分沒有**——而我自己用 `notest` 也已經混了兩種完全不同的情境（「正要去測試」+「決定不測」）
3. 主管問的表面是「可見性問題」，但底層其實是「**結果 schema 設計問題**」

之後跑去問同公司負責不同產品的 QA 同事，發現他們團隊的 schema 切得比我們乾淨——詳見第三節案例分析。

## 文章主結構（草擬）

### 一、開場：你的測試結果欄位長什麼樣？

**不從 `notest` 切入**——大多數 QA 沒有這個詞彙。直接從讀者熟悉的兩狀態世界出發。

可能的開頭：

> 你的測試結果現在長什麼樣？  
> 大多數 QA 團隊是這樣——綠色的 `pass`、紅色的 `fail`，再加上 timestamp。  
> 你可能已經用這個 schema 十年了。它很乾淨、好讀、能上 dashboard。  
> 但當「執行測試的人」從你變成 AI，這個兩欄世界**會立刻不夠用**。

### 二、為什麼 pass / fail 對 AI 不夠

核心對比：

| 測試執行者 | 行為特性 | 結果該長什麼樣 |
|---|---|---|
| 人寫的測試腳本 | **deterministic**（給定輸入，輸出唯一） | `pass / fail` 兩狀態就夠 |
| AI 執行測試 | **non-deterministic**（會猶豫、會繞路、會出錯） | 需要表達「不確定」「半途中斷」「該人工複查」 |

**關鍵洞察**：信心度是 AI 測試的內建屬性，不是 bonus 資訊。

- 人寫的腳本世界裡，信心永遠是 100%（不然測試腳本本身就是壞的）
- AI 世界裡，每一筆結果都帶一個獨立的信心維度

這就是為什麼 `pass / fail` 二元欄位**結構性**地不夠用——不是名字不好，是缺一整個維度。

### 三、第一直覺解法：加第三個狀態——為什麼也不夠

讀者讀到這裡會自然想：「那加個 `notest` / `skipped` / `inconclusive` 不就好了？」

這節就是案例分析：**我自己團隊就走了這條路**——我們加了 `notest`。看起來補了缺，其實只是把問題往後推。

#### 3.1 我們團隊的 `notest` 至少塞了四種東西

1. **主動跳過**（by design） — AI 判斷此 case 不該跑
2. **低信心不下判斷** — AI 跑了、不確定，丟給人工
3. **執行失敗** — 還沒進到判斷就壞了（環境、依賴）
4. **沒有判斷規則** — AI 根本不知道該怎麼驗

#### 3.2 同公司不同產品也分得不一樣——schema 是自發秩序，不是中央規範

訪問同公司負責另一個產品的 QA 同事，他們的 schema 切得更乾淨：

| 情境 | 我們團隊 | 他們團隊 |
|---|---|---|
| 還沒輪到測 | `notest` | **空著**（pending） |
| 不打算測 / 不需要測 | `notest` | **N/A** |
| 想測但缺條件 / 缺方法 | `notest` | **notest** |

他們對 `notest` 有一個清楚的工作定義：

> 「應該要測，但是沒有測試。」

**注意這個觀察**：同一家公司、同一個職位、做不同產品——schema 卻長得不一樣。這代表測試結果欄位**從來不是中央決定下來的**，是 QA 個人 / 小組逐步累積出來的**自發秩序**。

這個發現本身就有意義——**你的測試結果欄位不是先驗的真理**，是某個人在某個時點做的設計決定。沒人說它不能重做。

#### 3.3 但即便如此乾淨的定義，仍藏著被忽略的維度

「應該要測，但是沒有測試」這個定義看起來清楚，**但它自己也藏了兩種完全不同的東西**：

| 性質 | 例子 | 觀察到的職場現象 |
|---|---|---|
| **應測但測不能**（ought but **can't**） | 沒有跌停股可測、沒有測試方法 | 外部限制——值得被看見、值得補測試環境 |
| **應測但測不想**（ought but **won't**） | 這次價值不夠、步驟太麻煩 | 內部選擇——容易被同一個欄位包裝 |

我觀察到 QA 圈裡這種用法相當普遍——把「不太想下去測」的案例也標進 `notest` 或同等欄位，因為它在欄位上**長得跟「無法測試」一模一樣**。動機完全不同，但欄位無法分辨。

舉一個常見場景幫助理解：

> 某個 case 標 `notest`，理由欄寫「資料量太少不重要」。這背後可能是「真的不值得測」，也可能是「測起來太麻煩，我用『不重要』包裝它」。**欄位本身分不出這兩種**。

這才是 schema 為什麼會被當成緩衝工具的結構性原因：**當「應測但測不能」跟「應測但測不想」共享同一個格子，動機就沒有了揭露的義務**。

#### 3.4 我自己一個人用 `notest` 也已經混了兩種狀態

| 我的用法 | 性質 | 終態還是中間態？ |
|---|---|---|
| 「正要去測試」 | 時間性（pending）—— 之後會變成 pass / fail | 中間態 |
| 「決定不測」 | 決策性（skipped）—— 終局決定 | 終態 |

> **這是文章最自我反省的一段**：我寫部落格倡導「問題驅動測試」，卻對自己每天看到的 `notest` 欄位從沒問過為什麼有它、為什麼這樣用。**直到主管問同事，我才開始問**。

#### 3.5 教訓

**單一新狀態無法承擔「不是 pass 也不是 fail」的所有可能**。再加幾個狀態（`skipped` / `inconclusive` / `pending`）也只是延遲爆炸——核心問題是「**結果不是一維的**」。

更深一層：**欄位的設計會被使用習慣反向塑形**。當欄位有承載「主觀選擇」的空間，它就會被那樣使用。schema 不是中性的記錄工具，它**會影響使用者願不願意誠實**。

### 四、正解：把測試結果結構化成多軸

提出四軸 schema：

| 欄位 | 取值 | 說明 |
|---|---|---|
| **execution** | `done` / `aborted` / `not_started` | 跑了沒、有沒有跑完 |
| **verdict** | `pass` / `fail` / `inconclusive` / `skipped_by_design` | 跑了的話結論是什麼 |
| **confidence** | `high` / `medium` / `low` | 對 verdict 的信心 |
| **action** | `accept` / `human_review` / `rerun` / `fix_env` / `update_spec` | AI 對下一步的建議 |

再加一個自由文本欄位 **`evidence`**：記錄「為什麼下這個 verdict」+「不確定來自哪裡」。

論述重點**不是欄位名字**——是**思考方式**：
> 跑了沒、結論是什麼、結論多可信、下一步做什麼——**這四件事各自獨立**。塞進同一欄就會丟資訊。

把舊的 `notest` 四種拆到新 schema：

| 舊 `notest` 場景 | 新 schema 表達 |
|---|---|
| 主動跳過 | `execution=done, verdict=skipped_by_design` |
| 低信心不下判斷 | `verdict=inconclusive, confidence=high, action=human_review` |
| 執行失敗 | `execution=aborted, action=fix_env` |
| 沒判斷規則 | `verdict=inconclusive, confidence=high, action=update_spec` |
| 我自己的「正要去測試」 | `execution=not_started` |
| 我自己的「決定跳過」 | `verdict=skipped_by_design` |

每一筆都能被精準分流——主管讀 dashboard 也能立刻分辨。

### 四.5、為什麼多軸 schema 真正解開的不是技術問題，是責任問題

讀到這裡很容易讓人以為——多軸 schema 是個**技術正確性**問題：分流更精準、dashboard 更可讀、分類更乾淨。

但這還不是它最有力的論點。

多軸 schema 真正解開的，是 QA 在 AI 測試時代的**簽名困境**：

| 資訊密度 | QA 的可選動作 | 心理結果 |
|---|---|---|
| **稀薄**（只有 `pass / fail / notest`） | 只能整批接受 / 整批拒絕 | **不敢簽**——簽下去等於空白支票 |
| **豐富**（多軸 + evidence） | 可以精準同意一部分、挑戰一部分 | **敢簽**——知道自己同意了什麼 |

換句話說——**schema 的豐富度，是 QA 願意當「責任承擔者」的前提**。沒有 evidence trail，QA 簽下去等於空白支票；有了 evidence trail，QA 簽下去是**對特定主張的背書**。

這個觀察反過來會翻轉一個常見焦慮：

> 「AI 取代 QA」的常見焦慮是「AI 越強，QA 越被取代」。  
> 但實際上更可能發生的是——**AI 越能輸出詳細資訊，QA 越敢承擔責任**。  
> AI 不是 QA 的競爭者，AI 是 QA 的**舉證材料供應商**。

寫測試結果欄位的 schema，看起來是技術選擇，**但它決定了 QA 在組織裡敢不敢開口、敢不敢蓋章、敢不敢拿自己的職業信用去背書一個 release**。

詳細的當責討論在另一篇：[如果主管想用 AI 取代我，我會給他看的四份「當責清單」](/post/ai-replacement-accountability-checklist/)——這篇是它的 schema 層補丁。

### 五、信心是「屬性」不是「狀態」（核心 differentiator）

> 這節是文章跟其他「AI 測試結果」討論最大的差異化。大部分人講這題會卡在 `pass / fail / skip` 的範疇內，但**真正的設計洞察在這一層**。

反例 vs 正例：

| 反例（壞設計） | 正例（好設計） |
|---|---|
| `notest`（把低信心當成獨立狀態） | `verdict=fail, confidence=low`（fail 仍然是 fail，只是不確定） |
| `inconclusive`（一刀切「不知道」） | `verdict=inconclusive, confidence=high`（「我**很確定**我無法判斷」也是有用資訊） |

「**我很確定我無法判斷**」跟「**我可能猜錯了 fail**」是兩種完全不同的訊號：

- 前者 → oracle 缺失，要**補規格**
- 後者 → AI 能力邊界，要**訓練 / 換模型**

把信心拉成獨立維度，這兩個訊號才分得開。

### 六、QA 該怎麼開始：不用一次補齊四軸

不需要一次跳到完整四軸 schema。文章最後給讀者一個**可以明天就動手**的最小步驟梯子：

1. **第一步（最小變動）**：在 `pass / fail` 之外加一個 `confidence` 欄位（high / medium / low）。光這一步就能分流出：
   - `fail-low-confidence` → 可能是 flaky，先 rerun
   - `pass-low-confidence` → 可能是**假綠燈**，值得人工抽查
2. **第二步**：把「跑了沒」拆出來——區分 `aborted`（環境壞，沒跑完）與 `done`。修環境的工作就被識別出來
3. **第三步**：補上 `action` 欄位——讓 AI 不只報結果，還報下一步建議

**重要的態度**：這篇不是要讀者明天就重做 schema，是要他**意識到自己的 `pass / fail` 世界正在被 AI 撐破，且這個破口有結構性的解法**。

### 七、結語：接回母題

把文章接回 [empty-result-ambiguity-in-filter-testing](/post/empty-result-ambiguity-in-filter-testing/) 的母題：

> 篩選測試的「空」、AI 測試的「notest」——都是同一個品質工程命題的兩面：**當系統允許「中間態」存在，它就必須同時承擔回答「這個中間態到底是什麼」的責任**。

差別在於：
- empty 歧義發生在**資料層**——測試對象的結果
- notest 歧義發生在**判斷層**——測試本身的結果

兩個都是「**資訊在欄位邊界消失**」。一個是「沒資料 vs 篩選壞」，一個是「沒測 vs 不敢判斷」。**寫測試的人有責任不讓這種坍縮發生**。
