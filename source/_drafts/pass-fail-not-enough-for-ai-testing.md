你的測試結果現在長什麼樣？

大多數 QA 團隊是這樣——綠色的 `pass`、紅色的 `fail`，再加上一個 timestamp。你可能已經用這個 schema 十年了。它很乾淨、好讀、能上 dashboard，過去也從來沒出過問題。

但當「執行測試的人」從你變成 AI，這個兩欄世界**會立刻不夠用**。

而且不夠用的方式，不是「再加一格就好」那麼簡單。

---

## 一、為什麼 pass / fail 對 AI 不夠

先看一個對比：

| 測試執行者 | 行為特性 | 結果該長什麼樣 |
|---|---|---|
| 人寫的測試腳本 | **deterministic**（給定輸入，輸出唯一） | `pass / fail` 兩狀態就夠 |
| AI 執行測試 | **non-deterministic**（會猶豫、會繞路、會出錯） | 需要表達「不確定」「半途中斷」「該人工複查」 |

關鍵在這句：**信心度是 AI 測試的內建屬性，不是 bonus 資訊。**

- 在人寫的腳本世界裡，信心永遠是 100%——不然這支測試腳本本身就是壞的。
- 在 AI 世界裡，每一筆結果都帶一個獨立的信心維度：它可能 70% 確定這個 `pass`、95% 確定那個 `fail`。

這就是為什麼 `pass / fail` 二元欄位**結構性**地不夠用——不是名字取得不好，是它缺了一整個維度。

---

## 二、第一個直覺解法：加第三個狀態——為什麼也不夠

讀到這裡，你大概會自然想：「那加一個 `skipped` / `inconclusive` / `待測` 不就好了？」

這正是很多團隊走過的路。我也看過——而且我自己就走過。看起來補了缺，其實只是把問題往後推。

### 那個「多出來的第三狀態」至少塞了四種東西

不管你的團隊把它叫 `skipped`、`待測`、`N/A` 還是別的——當我認真去看標進這格的案例，至少有四種完全不同的情境混在一起：

1. **主動跳過**（by design）—— 判斷此 case 不該跑
2. **低信心不下判斷** —— 跑了、不確定，丟給人工
3. **執行失敗** —— 還沒進到判斷就壞了（環境、依賴）
4. **沒有判斷規則** —— 根本不知道該怎麼驗

四種動機、四種後續處理，但在 dashboard 上長得一模一樣。

### schema 是自發秩序，不是中央規範

有趣的是，我見過同一個職位、不同團隊的人，這格切得完全不一樣。有一組的切法乾淨得多：

| 情境 | 一種常見切法 | 另一種更乾淨的切法 |
|---|---|---|
| 還沒輪到測 | 全部塞第三狀態 | **空著**（pending） |
| 不打算測 / 不需要測 | 全部塞第三狀態 | **N/A** |
| 想測但缺條件 / 缺方法 | 全部塞第三狀態 | 才標那個「應測未測」的狀態 |

後者對那個狀態有一個清楚的工作定義：

> 「應該要測，但是沒有測試。」

注意這件事：**同一個職位、做不同產品，schema 卻長得不一樣**。這代表測試結果欄位**從來不是中央決定下來的**，是 QA 個人 / 小組逐步累積出來的**自發秩序**。

這個發現本身就有意義——**你的測試結果欄位不是先驗的真理**，是某個人在某個時點做的設計決定。沒人說它不能重做。

### 但即便這麼乾淨的定義，仍藏著被忽略的維度

「應該要測，但是沒有測試」這個定義看起來夠清楚了，**但它自己也藏了兩種完全不同的東西**：

| 性質 | 例子 | 職場現象 |
|---|---|---|
| **應測但測不能**（ought but **can't**） | 沒有測試環境、沒有測試方法 | 外部限制——值得被看見、值得補資源 |
| **應測但測不想**（ought but **won't**） | 這次價值不夠、步驟太麻煩 | 內部選擇——容易被同一個欄位包裝 |

我觀察到 QA 圈裡這種用法相當普遍——把「不太想下去測」的案例，也標進那個「未測」欄位，因為它在欄位上**長得跟「無法測試」一模一樣**。動機完全不同，但欄位分不出來。

舉一個常見場景：

> 某個 case 標成未測，理由欄寫「資料量太少不重要」。這背後可能是「真的不值得測」，也可能是「測起來太麻煩，我用『不重要』包裝它」。**欄位本身分不出這兩種。**

這才是這類欄位為什麼會被當成緩衝工具的結構性原因：**當「應測但測不能」跟「應測但測不想」共享同一個格子，動機就沒有了揭露的義務。**

說句對自己誠實的話：我寫部落格倡導「問題驅動測試」，卻對自己每天填的那個欄位，從沒問過為什麼有它、為什麼這樣用。我自己用它，其實也早就混了兩種狀態——「正要去測」（時間性，之後會變 pass/fail，是**中間態**）跟「決定不測」（決策性，是**終態**）。**直到有人當著我的面問起，我才開始問自己。**

### 教訓

**單一新狀態，無法承擔「不是 pass 也不是 fail」的所有可能。** 再加幾個（`skipped` / `inconclusive` / `pending`）也只是延遲爆炸——核心問題是「**結果不是一維的**」。

更深一層：**欄位的設計會被使用習慣反向塑形。** 當一個欄位有承載「主觀選擇」的空間，它就會被那樣使用。schema 不是中性的記錄工具，它**會影響使用者願不願意誠實**。

## 三、業界框架早就分得比你細：先學它們（中間解）

讀完上面的混亂，自然會問：「那業界主流框架怎麼處理？」

答案是——**主流框架早就把這幾件事分開了。** 而且要注意：**自動化測試框架跟手動測試管理框架，給的狀態空間還不一樣。**

### 自動化測試框架（給腳本用，狀態相對精簡）

| 框架 | 對「未測」的拆分 | 多分出的細節 |
|---|---|---|
| **pytest** | `passed` / `failed` / `skipped` / `xfail` / `xpassed` / `error` | `skipped`（主動跳過）與 `error`（執行失敗）**分開**；`xfail` 標「預期會失敗」 |
| **JUnit / TestNG** | `passed` / `failed` / `skipped` / `error` | 把「主動跳過」跟「執行失敗」拆開 |
| **Cypress / Playwright** | `passed` / `failed` / `skipped` / `pending` / **`flaky`** | 多了 `pending`（還沒跑）與 `flaky`（多次 run 結果不一） |
| **Allure 報告** | 額外的 **`broken`** | 把「**測試自己壞了**」跟「**系統有 bug**」分開 |
| **NUnit / MSTest** | 額外的 **`inconclusive`** | 「不能判斷」獨立成一格——但業界普遍**不鼓勵用**（理由見後面信心那節） |

### 手動測試管理框架（給人填寫，狀態更豐富）

| 框架 | 預設狀態 | 比自動化多出的關鍵狀態 |
|---|---|---|
| **JIRA Xray** | `PASS` / `FAIL` / `TODO` / `EXECUTING` / `ABORTED` / **`BLOCKED`** | `BLOCKED`（被外部擋住）、`EXECUTING`（測試進行中） |
| **Zephyr** | `PASS` / `FAIL` / `WIP` / `BLOCKED` / `UNEXECUTED` / **`DESCOPED`** | `WIP`（進行中）、`DESCOPED`（這次不該測） |
| **TestRail** | `Passed` / `Failed` / `Blocked` / **`Retest`** / `Untested` | `Retest`（要求重測）獨立成狀態 |
| **HP ALM** | `No Run` / `Not Completed` / `Passed` / `Failed` / `Blocked` / `N/A` | `Not Completed`（中途中斷）+ `N/A`（不適用） |

幾個共通點：

- **幾乎全部都有 `BLOCKED`**——「我有試但被擋住」是手動測試的核心狀態，跟 `fail` 完全分開。
- **大多有「進行中」**（`EXECUTING` / `WIP` / `Not Completed`）——人測一個 case 可能跨好幾天，要有狀態表達中途。
- **部分有 `Retest` / `DESCOPED` / `N/A`**——表達「這個結果暫時不算數」或「這個 case 已經不該存在」。

**為什麼手動框架的狀態比自動化多？** 因為——

> 腳本被要求 deterministic——同樣輸入要同樣輸出，所以狀態空間可以精簡。
> 人執行測試本來就 non-deterministic——同一個 case 今天跑明天跑，環境變、被人插隊——需要更多狀態表達真實情況。

這個「手動跟自動化根本是兩種不同物種」的觀點，我在[〈不要只是把手動測試腳本自動化〉](/post/not-just-make-manual-script-automate/)裡談得更完整——兩者的稀缺資源不同（人是判斷時間、腳本是維護成本）、擅長的事不同、連失敗模式都不同。**狀態空間的差異，只是這個「不同物種」事實在 schema 上的倒影。**

### 兩種框架加起來告訴你的事

1. 「不是 pass 也不是 fail」**確實需要分流**——主流框架分到 4-10 種。
2. 「主動跳過」vs「執行失敗」vs「外部 BLOCKED」**永遠分開**——這是業界共識。
3. 「測試自己壞了」vs「系統壞了」是 Allure / Xray 帶起來的細節，**對 AI 測試特別重要**。
4. 手動框架早就承認執行者非 deterministic——給了更多狀態表達真實複雜性。

所以你現在的 `pass / fail / 第三狀態` 三格，跟業界距離至少 4-7 種狀態。**升級到 pytest-style 或 Xray-style 的多狀態，是合理的中間解**——不用重做框架，只是把現有混合欄位拆細。

但這還是不夠——下一節說為什麼。

## 四、即便用上業界所有狀態，AI 測試仍缺一整個維度——多軸 schema

注意一件事：**業界框架的這些狀態，本質上都是「橫向分類」**——把「不是 pass 也不是 fail」切成更多更細的格子。

但 AI 測試需要的不只是橫向分類——它需要**一整個新的縱向維度：信心度**。

前面提過手動跟自動化是兩種不同物種。**AI 執行測試，是闖進來的第三種物種**——而且牠帶來一個前兩種都沒有的東西。看這個三層對照：

| 執行者 | 狀態空間特性 |
|---|---|
| **自動化腳本** | 5-6 種狀態就夠——因為腳本要 deterministic，「不確定」根本不該存在（不確定的腳本就是壞腳本） |
| **人工執行** | 8-10 種狀態——因為人會卡關、被擋住、跨天執行，但人通常**直接給最終 verdict**，很少在欄位上標「我也不確定」 |
| **AI 執行** | 上面 **PLUS** 一個獨立維度：「這次跑得有多紮實 / 多可信」——同樣標 pass，AI 可能這次讀 code 推測、下次真的開瀏覽器跑、再下次只 mock 了 API。**橫向分類分不出這個維度** |

腳本不會「不確定」、人很少標「不確定」——但 **AI 內建的就是「不同程度的把握」**。而業界至今沒有任何測試框架的 schema 把「信心」當一級欄位。

這就是為什麼還要再升一階：**多軸 schema**——把橫向 axis（狀態分類）跟縱向 axis（信心、執行紮實度）**分開**。

一個四軸的範例：

| 欄位 | 取值 | 說明 |
|---|---|---|
| **execution** | `done` / `aborted` / `not_started` | 跑了沒、有沒有跑完 |
| **verdict** | `pass` / `fail` / `inconclusive` / `skipped_by_design` | 跑了的話結論是什麼 |
| **confidence** | `high` / `medium` / `low` | 對 verdict 的信心 |
| **action** | `accept` / `human_review` / `rerun` / `fix_env` / `update_spec` | 對下一步的建議 |

再加一個自由文本欄位 **`evidence`**：記錄「為什麼下這個 verdict」+「不確定來自哪裡」。

重點**不是欄位名字**，是**思考方式**：

> 跑了沒、結論是什麼、結論多可信、下一步做什麼——**這四件事各自獨立。** 塞進同一欄就會丟資訊。

把前面那「四種混在一起的東西」拆到新 schema：

| 舊的混合場景 | 新 schema 表達 |
|---|---|
| 主動跳過 | `execution=done, verdict=skipped_by_design` |
| 低信心不下判斷 | `verdict=inconclusive, confidence=high, action=human_review` |
| 執行失敗 | `execution=aborted, action=fix_env` |
| 沒判斷規則 | `verdict=inconclusive, confidence=high, action=update_spec` |
| 「正要去測」 | `execution=not_started` |
| 「決定跳過」 | `verdict=skipped_by_design` |

每一筆都能被精準分流——主管讀 dashboard 也能立刻分辨。

## 五、多軸 schema 真正解開的，不是技術問題，是責任問題

讀到這裡很容易以為——多軸 schema 是個**技術正確性**問題：分流更精準、dashboard 更可讀。

但這還不是它最有力的論點。

多軸 schema 真正解開的，是 QA 在 AI 測試時代的**簽名困境**：

| 資訊密度 | QA 的可選動作 | 心理結果 |
|---|---|---|
| **稀薄**（只有 `pass / fail / 未測`） | 只能整批接受 / 整批拒絕 | **不敢簽**——簽下去等於空白支票 |
| **豐富**（多軸 + evidence） | 可以精準同意一部分、挑戰一部分 | **敢簽**——知道自己同意了什麼 |

換句話說——**schema 的豐富度，是 QA 願意當「責任承擔者」的前提。** 沒有 evidence trail，QA 簽下去等於空白支票；有了 evidence trail，QA 簽下去是**對特定主張的背書**。

這個觀察反過來會翻轉一個常見焦慮：

> 「AI 取代 QA」的常見焦慮是「AI 越強，QA 越被取代」。
> 但實際上更可能發生的是——**AI 越能輸出詳細資訊，QA 越敢承擔責任**。
> AI 不是 QA 的競爭者，AI 是 QA 的**舉證材料供應商**。

寫測試結果欄位的 schema，看起來是技術選擇，**但它決定了 QA 在組織裡敢不敢開口、敢不敢蓋章、敢不敢拿自己的職業信用去背書一個 release**。

（更完整的當責討論，在另一篇：[如果主管想用 AI 取代我，我會給他看的四份「當責清單」](/post/ai-replacement-accountability-checklist/)——這篇是它的 schema 層補丁。）

## 六、信心是「屬性」，不是「狀態」

這節是這篇跟其他「AI 測試結果」討論最大的差異。大部分人講這題會卡在 `pass / fail / skip` 的範疇內，但真正的設計洞察在這一層。

對比一下壞設計與好設計：

| 壞設計 | 好設計 |
|---|---|
| 把低信心當成一個獨立狀態 | `verdict=fail, confidence=low`（fail 仍然是 fail，只是不確定） |
| `inconclusive` 一刀切「不知道」 | `verdict=inconclusive, confidence=high`（「我**很確定**我無法判斷」也是有用資訊） |

「**我很確定我無法判斷**」跟「**我可能猜錯了這個 fail**」是兩種完全不同的訊號：

- 前者 → oracle 缺失，要**補規格**
- 後者 → AI 能力邊界，要**訓練 / 換模型**

把信心拉成獨立維度，這兩個訊號才分得開。如果把信心硬塞回「狀態」那一欄，它們就永遠混在一起。

## 七、QA 該怎麼開始：四步梯子

你不需要一次跳到完整四軸 schema。給你一個可以**明天就動手**的最小梯子：

1. **第一步（最輕，不改框架）**：先升級到 pytest-style 的多狀態——把 `pass / fail / 未測` 改成 `pass / fail / skipped / error`（或再加 `pending`、`broken`）。沒改欄位結構，只是把混合欄位拆細。光這步就解掉「跳過 vs 執行壞」混為一談的最大痛點，且業界框架早就驗證可行。
2. **第二步（加維度）**：在多狀態之外加一個 `confidence` 欄位。光這一步就能分流出：
   - `fail` + 低信心 → 可能是 flaky，先 rerun
   - `pass` + 低信心 → 可能是**假綠燈**，值得人工抽查
3. **第三步（拆執行）**：把「跑了沒」拆細——區分 `aborted`（環境壞，沒跑完）與 `done`。修環境的工作就被識別出來。
4. **第四步（加建議）**：補上 `action` 欄位——讓 AI 不只報結果，還報下一步建議。

重要的態度：這篇不是要你明天就重做 schema，是要你**意識到自己的 `pass / fail` 世界正在被 AI 撐破，而且這個破口有結構性的解法**。

## 結語

我寫過另一篇談[篩選後沒結果的歧義](/post/empty-result-ambiguity-in-filter-testing/)——篩選測試的「空」，跟 AI 測試的「未測」，其實是同一個品質工程命題的兩面：

> **當系統允許「中間態」存在，它就必須同時承擔回答「這個中間態到底是什麼」的責任。**

差別只在發生的層次：

- 篩選的「空」發生在**資料層**——測試對象的結果。
- 「未測」的歧義發生在**判斷層**——測試本身的結果。

兩個都是「**資訊在欄位邊界消失**」。一個是「沒資料 vs 篩選壞了」，一個是「沒測 vs 不敢判斷」。

寫測試的人，有責任不讓這種坍縮發生。
