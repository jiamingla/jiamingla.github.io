# Tools Roadmap — 把 PDT 實體化成工具，而不只是文章

**起源**：2026-05-04 對話。寫部落格是一條路，但不是唯一路徑。對於一個初階 QA、在公司九個月、自評「沒做出明顯文化變革」——**做工具比寫文章更直接被主管看見**。

**核心思路**：先工具、後文章。工具用著用著會自動產生比硬擠出來的文章更紮實的素材。部落格成為副產品，不是主產品。

---

## 為什麼這條路徑可能比寫部落格更值得（給未來的自己）

從 2026-05-04 的對話延伸出的判斷：

1. **解決「沒做明顯文化推動」的自評焦慮** — 主管不會去讀你的部落格，但會看到你做的工具
2. **主管的腦袋語言** — 工具是「在做事」，部落格是「在寫感想」
3. **硬資料累積** — 工具有使用紀錄，未來談績效時是 evidence
4. **自動產生部落格素材** — 用工具的過程本身會產生紮實的故事
5. **不需要對外曝光** — 你還沒準備好推廣部落格；工具不一定要 public，內部用就有價值
6. **跟「擴大測試視野」這條線會合** — spec-maintenance-gray-zone § 八-2 已經寫到，主動跨進文件灰色地帶會擴大 QA 的測試範圍。工具是這個動作的加速器

---

## 路徑一：QA 自我 Review Agent（**先做這個**）

### 用途

給它你寫的草稿——bug report、測試計畫、回歸範圍判斷——它用 PDT 風格反問：

- 「你考慮過 X 了嗎？」
- 「失敗時畫面長什麼樣？」
- 「這個改動有沒有動到 Y？」
- 「上線後第一週要看什麼指標？」
- 「邊界值落在哪裡？」
- 「組合空間有沒有爆炸的可能？」

### 為什麼先做這個

- **範圍小、ROI 立即** — 一個 Claude Project + 一個系統 prompt 就能起步
- **下班就能用、上班也能用** — 個人助理，不需要任何人同意才能上線
- **第一週就會有體感** — 用幾次就知道好不好用、要怎麼調
- **不對外** — 完全在自己掌控內，沒有「主管看到」的顧慮

### 系統 Prompt 建構材料（已有）

- `problem-driven-testing-intro.md`（PDT 招牌主張，提問模板的源頭）
- `spec-maintenance-gray-zone.md`（spec / 規格 / 文件落差的提問脈絡）
- `ai-replacement-accountability-checklist.md`（當責四清單——對的問題、人際信任、覺察不對勁、簽名負責）
- `rethinking-regression-test-value.md`（哪些該砍、哪些該留的判斷紀律）
- 部分 `black-box-limitations-and-strategy.md`（黑箱與灰箱的視角分層）

### 第一版可以怎麼起手

1. 開一個 Claude Project
2. 把上述文章貼進 Project knowledge
3. 系統 prompt 設定為「PDT 提問教練」角色，明確要求：
   - 不直接給答案，用提問逼使用者思考漏洞
   - 涵蓋功能本身、依賴、邊界、失敗時情境、組合爆炸
   - 提問要具體可回答，不要「請考慮所有可能的風險」這種空話
4. 用兩三個真實 bug report 餵它，看回饋是否有用
5. 根據實際使用調整 prompt（這個迴圈會持續好幾週）

### 預期使用情境

- 每天結束前，把當天寫的 bug report / 測試紀錄丟給它檢查
- Sprint 規劃時，把 PM 給的 spec 丟給它生 PDT 提問清單
- 跟主管 1:1 前，把要報告的內容丟給它預先審視
- 寫部落格草稿時，丟給它看「我有沒有漏掉哪個角度」

## 路徑二：PDT Agent（晚一點再做）

### 用途

把 self-review agent 放大成團隊版。對象從「我自己的草稿」擴大到「PM 給的 spec」「RD 提交的 PR description」。

餵它一份規格 → 輸出一張「PDT 提問清單」。

這是 spec-maintenance-gray-zone 文章裡那段「在會議室問 PM 的問題」具象化的工具版。

### 為什麼晚一點做

- 需要 self-review agent 跑兩到四週、累積出「真正有效的 PDT 提問範本」
- 對外發布之前，需要更穩的 prompt 工程與案例庫
- 需要先驗證「PDT 提問」這個產品概念在團隊裡有人會用，不是只有自己覺得好

### 對外的價值

- 內部小工具 → 給主管 demo（這比寫部落格直接得多）
- README 放 GitHub → 個人技術品牌
- 部落格題材 → 「我做了一個 PDT Agent，學到 Y」（先工具、後文章的副產品）

## 路線圖

| 階段 | 動作 | 預估時間 |
|---|---|---|
| Week 1 | 開 Claude Project，灌進現有文章，寫 v1 系統 prompt | 1-2 小時 |
| Week 2-4 | 每天用，調 prompt，記錄哪些提問真的有抓到漏洞 | 持續用即可 |
| Month 2 | 累積有效提問範本，評估是否值得做 PDT Agent | 半天 review |
| Month 3+ | 如果 self-review agent 真的好用，開始做 PDT Agent | 1-2 週工作量 |

## 對部落格的反向影響

工具如果做出來、且真的有用，**會反過來餵養這幾條部落格內容線**：

- **PDT 系列** — 「我把 PDT 變成工具之後發現的事」
- **AI 取代系列** — 「我用 AI 工具讓自己更難被取代」（具體案例）
- **自動化決策線**（boredom-is-a-signal 那條）— 「QA 工具自動化的下一步：自我 review」

但這些都是副產品。**主目標是：把 PDT 從一個方法論，變成一個能在工作裡被使用的東西**。

---

## 提醒給未來的自己

- 不要急著把工具開源 / 對外曝光（推廣顧慮還在）
- 不要急著寫成部落格文章（寫了會稀釋實做的時間）
- 工具好不好用，**唯一的判斷標準是「我自己每天有沒有開來用」**——其他都是次要
- 如果一週後沒繼續用，代表 prompt 設計有問題，回去調，不要硬撐
