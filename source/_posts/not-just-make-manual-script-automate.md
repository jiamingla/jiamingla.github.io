## 前言：那個熟悉的「來自動化吧」場景

你一定經歷過、或者正在經歷這個讓人心跳加速卻又隱隱不安的場景：

某天 Sprint 規劃會議上，主管或專案負責人看著那份累積了兩百多條、每次回歸測試都要花掉一個人整整兩天的試算表案例，深吸了一口氣說：
> 「每次都要手動跑回歸太慢了，我們開始做自動化吧！這期先把這兩百條手動測試案例全部轉成自動化腳本。」

你聽完熱血沸騰，心想終於可以擺脫無聊重複的點擊工作了。於是，你領了任務，開始像打字機一樣，把手動案例清單上的步驟，一行一行翻譯成 Selenium、Appium 或 Playwright 腳本：
- *Step 1: 點擊登入按鈕 -> `driver.find_element(By.ID, "login-btn").click()`*
- *Step 2: 輸入帳密 -> `send_keys(...)`*
- *Step 3: 驗證登入成功畫面 -> `assert ...`*

三個月過去了，你的自動化案例庫看起來蔚為壯觀。然而，現實的耳光很快就甩了過來：
- CI/CD 執行自動化測試時，十次有八次在亮紅燈，但每次點進去檢查，發現程式根本沒壞，只是環境延遲、按鈕沒加載出來、或是前端改了個 Class 命名或換了個小組件，所以看起來一模一樣，但id不一樣影響了關鍵路徑，整組歹去。
- 為了讓這些腳本維持運作，你每天有超過一半的時間都在修改那些碎掉的 Locator、調整 `time.sleep` 或是應付 Flaky 測試。
- 最讓人氣餒的是，這兩百條自動化測試跑幾千次，卻幾乎沒幫你抓到過什麼有價值的 Bug。

你站在被維護債淹沒的泥潭裡，看著每天依然被塞進來的線上 Ticket，不禁懷疑：**「我們是不是搞錯了什麼？」**

我們犯的第一個也是最致命的錯誤，就是誤以為：**自動化測試是手動測試的「加快版本」**。我們以為只要把步驟翻譯成代碼，就能跑得又快又省力。

但事實上，手動測試與自動化測試，本質上是**兩種完全不同的物種**。直接搬運，就是崩盤的開始。

---

## 一、為什麼業界都這樣做？

既然「直接轉換」這條路在三個月後通常會崩盤，為什麼不論是大廠還是小團隊，大家在啟動自動化時的直覺依然是「照搬手動案例」？

因為這個做法在初期具有極強的吸引力：

1. **看起來最直觀、最不用動腦**：
   手動測試案例是現成的，步驟寫得清清楚楚（「點這裡、輸入那裡、確認畫面長這樣」）。你只需要充當程式碼翻譯官，把人類語言對齊成 API 調用，幾乎不需要重新設計測試邏輯，上手門檻極低。
2. **ROI（投資報酬率）看起來好算又漂亮**：
   在向主管匯報時，數字非常好呈現：「我們有 200 個案例，手動需要 16 小時；改用自動化後，30 分鐘就能跑完。」這種簡化的數學公式，主管最愛聽，專案預算和時間也最容易要到。
3. **不用做二次決策**：
   在專案進度緊湊時，團隊沒有時間停下來重新評估「到底哪些東西適合自動化」、「哪些東西只適合人測」。直接把既有的東西「影印」一遍，是阻力最小的防禦性選擇。

這條路在第一週會讓你覺得進展神速，上手快是它最大的優點——但這個優點同時也是它最大的陷阱。因為它跳過了軟體工程最核心的一步：**因地制宜的設計**。

## 二、為什麼這條路會崩盤？三個結構性原因

當你把手動測試的步驟直接硬塞給機器時，專案的生命週期就已經進入倒數計時。這背後有三個無法被技術克服的結構性原因：

### 1. 手動案例是「點 X，看 Y」，但「看 Y」高度依賴人類視覺與隱性判斷

手動測試案例的驗證點往往寫得很簡單，例如：「驗證登入成功」或「驗證廣告橫幅正常顯示」。

但在真實的操作中，人類測試員在確認「廣告橫幅正常顯示」時，大腦進行的是極為複雜的綜合判定：
- 橫幅的文字有沒有超出邊框？
- 動態折價券的圖案有沒有擋住關閉按鈕？
- 跑馬燈字體在不同解析度下有沒有疊在一起？

如果要把這些「畫面看起來對」的細節全部用 Selenium 代碼寫出來，你需要寫幾十行極端複雜的座標計算、樣式檢查（CSS value）與可見性判定。這類驗證在機器世界裡**極端脆弱且容易誤報**。一旦前端更換了設計風格，腳本就會全滅。

### 2. 自動化做視覺判斷既慢又脆——Assertion 只能挑結構性事實

即使引入了圖像對比（Image Diff）或視覺回歸測試工具，這類驗證依然是 CI/CD 流程中跑得最慢、最容易因為一像素的渲染差異而噴紅燈的部分。

真正能被機器高效、穩定驗證的，從來不是視覺呈現，而是**「結構性事實」**。

這裡有一個經典的對比案例：

> **「骨架屏（Skeleton Loading）的視覺跳動」**
> 
> 在測試一個內容列表頁面時，頁面加載時會先顯示骨架屏，接著資料載入後替換成真實內容。在這個過程中，如果發生了嚴重的視覺跳動（Layout Shift）或是閃爍，手動測試員一眼就能看出「這體驗極差，加載體驗壞了」。
> 
> 但如果換成 UI 自動化腳本，它在 `wait_for_element` 等到列表出現後，驗證了 `list.count == 10`。此時即使剛才加載時畫面上演了慘不忍睹的視覺崩潰，腳本依然會亮起**毫無意義的綠燈**。

機器擅長驗證的是結構與資料：API 的 status code、Database 欄位值、LocalStorage 的 Token、或者是埋點事件的 Tag 是否正確發送。它不擅長、也不該去扮演人類的眼睛。

### 3. UI 變動頻率 通常顯著地高於 業務邏輯變動頻率

前端 UI 幾乎是整個軟體架構中最不穩定的地方。兩週改版一次按鈕顏色、調整一次欄位排版、或重構一次 CSS Class，在敏捷開發中是日常茶飯事。

但底下的 API 協議、資料庫 Schema、以及核心的業務邏輯（例如「付費會員才能讀取這篇文章」），往往好幾個月甚至幾年都不會大幅變動。

如果你把自動化測試腳本緊緊綁在 UI 元件上，就等於是把你的城堡蓋在隨時會位移的沙灘上。**每次 UI 一動，你的腳本就 break，這就是自動化專案維護成本暴增、ROI 最終崩盤的數學原因。**

## 三、本質差異：兩者的稀缺資源不一樣

要治本，我們必須認清兩者在資源模型上的根本不同。我常這樣比喻：

> **手動測試，就像是在森林中慢步前行。** 你一邊向前走，一邊用肉眼掃描整片森林，感受空氣、光影與環境的異樣，並在層層堆疊的落葉堆裡，尋找躲在暗處的那隻特定小蟲子。你身處林中，看到的是整片「林」。
> 
> **自動化測試，則是設定好一條固定軌道，然後開著高鐵快速飛馳。** 高鐵能在 10 秒內跑完你慢步走兩小時的路程，但它每次執行時，都只能機械式地驗證軌道旁邊「特定的那一棵樹」上有沒有長蟲。對於軌道十公尺外正在發生的森林大火，高鐵上的機器人完全一無所知。

這就是自動化測試最常遇到的困境：**「見樹不見林」**。我們不能因為高鐵開得快，就預設它能幫我們看完這片森林。

敏捷三叔公 David Ko 在[〈關於測試自動化你該小心的地方〉](https://agile3uncles.com/2024/09/04/the-things-you-should-know-about-testauto/)裡有一個更直接的講法：自動化測試做的其實是「**checking**」——驗證已知場景的預期結果，**它無法替你找出未知的 bug**。換句話說，高鐵軌道是你鋪的，它只會檢查你鋪過的地方。

在資源模型上，兩者有著本質的不同：

| 維度 | 手動測試 | 自動化測試 |
| :--- | :--- | :--- |
| **稀缺資源** | **人的判斷時間**（時間上限固定，會疲倦，且一次只能做一件事） | **腳本維護成本**（代碼量越大，維護債越重） |
| **擅長領域** | 探索、直覺判斷、視覺檢驗、發現未知的 Bug | 重複執行、批次跑、確定性流程、結構資料驗證 |
| **失敗模式** | 漏測、專注力下降、執行不一致 | Flaky 誤報、過期失效、維護債累積 |
| **失敗代價** | 一次性（這次沒測到，下次注意就好） | **複利累積**（一個 flaky 測試會持續污染 CI 直到被修好） |

我們不該用便宜的資源（機器的執行時間），去做需要稀缺資源（人類的判斷力）的事；也不該用稀缺的人類注意力，去重複做機器能極其可靠完成的事。

## 四、該怎麼分工？

當我們看清了兩者的資源模型後，自動化與手動的邊界自然就清晰了。不要嘗試把手動案例「翻譯」成自動化，而是要重新規劃兩者的職責：

### 🛠️ 手動測試（與探索式測試）應該專責：
1. **探索式測試（Exploratory Testing）**：
   帶著懷疑與直覺去探尋未知領域，尋找那些「沒有寫在規格書裡的設計漏洞」（這也是[《問題驅動測試》](/post/problem-driven-testing-intro/)的核心舞台）。
2. **邊界條件與異常路徑的體驗**：
   例如：當弱網狀態下使用者瘋狂點擊按鈕，畫面的遮罩有沒有適時阻擋？這類涉及人類心理與操作極限的情境。
3. **UX 與排版美學**：
   字體有沒有跑版、動畫過渡是否流暢自然、有沒有讓人感到挫折的彈窗設計。
4. **新功能發布的初次探索**：
   功能還沒穩定到值得寫腳本的程度，手動測試是最快獲得品質反饋的方式。

### 🤖 自動化測試應該專責：
1. **API 層的一致性與合約驗證**：
   驗證 Payload 欄位結構、Status Code、Error Response 是否對齊規格。這是穩定度最高、執行最快的防線。
2. **底層結構性 facts 驗證**：
   例如：用戶在 UI 點擊追蹤作者後，自動化腳本去驗證 API 是否回傳成功、Database 的關注表（Follow Table）是否新增了對應的 Row、以及前端埋點（GA/AppsFlyer）是否發送了正確的事件 Tag。
3. **Smoke Test（冒煙測試）**：
   在每次部署後，快速跑一遍主幹流程（Login -> Search -> Buy -> Logout），確保系統最核心的 API 還活著。
4. **批次與環境回歸**：
   自動化非常擅長在多種不同的語系配置或資料庫組合下，批次跑相同的數據組合驗證。

## 五、怎麼從現狀切換？（不是全砍重寫，而是漸進式清理）

如果你的 codebase 裡已經躺了兩百條痛苦無比的 UI 自動化腳本，不要慌，你不需要全部砍掉重來。請用這三個「砍掉訊號」進行漸進式垃圾清理：

1. **常常 Flaky 很脆弱，修了常常壞或根本修不好的**：
   如果一個案例在過去一個月內誤報了 3 次以上，且花費你大量時間，請**果斷砍掉它**，若是核心功能，改退回手動測試。
2. **Assertion 幾乎全是視覺判斷的**：
   如果腳本都在比對「按鈕是不是綠色」、「彈窗有沒有跳出」，請**砍掉它**，改寫成 API 整合測試。
3. **已經被手動跑了一輪「當保險」的**：
   如果你在 Release 前依然會不自覺地用手動去複測那條自動化案例（因為你底層根本不信任腳本的綠燈），這說明這條自動化已經失去價值。請**毫不猶豫地砍掉它**。

根據我的實務經驗，在一個沒有經過設計、只是盲目照搬手動案例的自動化專案中：
- 至少有 **60%** 的 UI 腳本是應該被無情刪除的。
- 有 **20%** 應該被下沉、改寫為執行更快且更穩定的 API 測試。
- 只有剩餘 **20%** 最核心、最關鍵的主幹流程（如登入、結帳），才值得保留在 UI E2E 測試層。

把被維護債吃掉的時間騰出來，去寫底層結構性的驗證（如驗證 DB、API、事件 Tag），你的自動化綠燈才會重新獲得團隊的信任。

## 結語：自動化不是手動的「快版本」，是不同的測試

下次你的團隊再次準備大張旗鼓地「把所有手動案例自動化」時，請把這篇文章貼到群組裡，然後問大家這兩個問題：
1. 我們現在寫的自動化腳本，是在做「機器擅長的事」，還是在扮演一個「廉價且不穩定的手動影印機」？
2. 我們有沒有為了盲目追求 Dashboard 上的綠燈，而主動坍縮了我們對系統品質的感知頻寬？

（順帶一提：敏捷三叔公甚至把「**測試全部通過**」「**UI 測試從不出錯**」列為該警覺的壞信號——全綠的 Dashboard 有時不是品質好，是測試已經失去偵測能力。）

寫到「能跑」，只是自動化腳本的第一步；學會**「在對的位置，設計對的分工」**，才是品質工程的紀律。

最後說個小祕密——這篇文章最初的標題，其實藏了一首歌：

> 「你從不知道我想做的不只是朋友
> 還想有那麼一點點自私的佔有」
> ——黃小琥〈不只是朋友〉（詞：言兒／曲：伍思凱）

你從不知道，我想做的不只是把手動測試自動化。對這份工作，我也想要那麼一點點「自私的佔有」——不只是當一台翻譯步驟的打字機，而是成為設計分工的人。

這篇文章是我們探討 UI 自動化測試設計美學的第一篇。在接下來的篇章中，我們將陸續聊聊：
- **系列（二）**：綠燈下的盲區：自動化測試中的「資訊坍縮」與隱性品質防護的流失。
- **系列（三）**：放過 UI 元素吧！解耦 UI 動作與「資料狀態斷言」的自動化實戰。
- **系列（四）**：自動化測試案例的半衰期：為什麼你該有刪除測試的「垃圾回收（GC）」機制？
- **系列（五）**：AI 時代的自動化測試：我們不再寫步驟，我們寫 Page API 與觀測指標。

敬請期待。
