## 一、無法被輕易驗證的「痊癒」

前陣子和一位在身心科病房工作的護理師朋友聊天，我問了她一個在軟體工程師看來可能有點奇怪，但一直以來都讓我很好奇的問題：

> 「在你們病房裡，要怎麼百分之百確定一個病人的憂鬱症已經『痊癒』，可以安全停藥或出院了？」

她聽完笑了笑，搖搖頭說：「這幾乎是臨床上最難的事情。因為在精神醫學裡，很多時候沒有所謂的『黃金標準（Gold Standard）』。」

她告訴我，很多時候病人主觀覺得「我好了，我現在心情很棒」，但對照臨床醫生使用的評估量表（如 PHQ-9 或 BDI-II），兩者的評估不一致率高達 51% 到 55%。有些人只是處於短期的「症狀緩解（Remission）」，但距離真正能回到社會、正常生活的「功能性康復（Recovery）」還有一大段距離。

「而且最怕的是病人自己覺得好了就擅自停藥。」她補充，「臨床研究顯示，緩解期過早停藥的復發率（Relapse rate）是 37%，而乖乖走完療程的只有 18%。如果是有三次以上發作史的高風險患者，一旦病人不按時服藥，我們就會完全失去驗證他到底『好起來了，還是只是剛好這週心情好』的基準線。」

聽完她的分享，我突然愣住了。

這不就跟我們在做軟體測試時，試圖驗證一個 Bug 已經「修好了」面臨一模一樣的認識論困境嗎？

## 二、從實體到虛擬：不斷擴大的黑盒子

在物理世界中，驗證「修復」通常是直觀且可窮盡的。

如果一個馬克杯的把手斷了，你用接著劑黏回去，或者直接重新開模換成更耐磨的材質。這個馬克杯的物理邊界是清晰的，它的承重極限、耐熱度都可以通過明確的公式計算，並在實驗室裡完成 100% 的窮盡測試。

然而，當我們進入軟體的世界，這個黑盒子的邊界便開始無限向外擴張，驗證的難度也呈現指數級上升。我們可以將系統的複雜度與驗證難度劃分為三個層次：

| 層次 | 系統性質 | 證明「修好了」的難度 |
|---|---|---|
| **第一層：實體系統（如馬克杯）** | 物理極限明確，邊界可窮盡。 | **容易**。改物理材質，耐熱/容量可被算乾淨。 |
| **第二層：傳統軟體系統** | 邏輯決定論（Deterministic），但我們自己寫的 Code 通常只佔系統整體的 1% 到 10%，其餘是由開源套件、資料庫、作業系統與第三方 API 交織而成的龐大黑盒。 | **難**。RD 修改了一行程式碼說「應該修好了，你測測看」，QA 在驗證時本質上是在盲人摸象。 |
| **第三層：AI 與 LLM 系統** | **機率論（Probabilistic）**。系統沒有固定的邏輯路徑，連開發者也無法精確解釋模型是如何產出某個特定答案的。 | **幾乎不可能**。微調了 Prompt 或調整了參數後，我們要輸入多少排列組合的測試資料，才敢宣稱它「永遠不會再產生幻覺（Hallucination）」？ |

在傳統軟體中，雖然我們面對的是龐大的第三方接縫，但底層邏輯好歹是「決定論」——同樣的輸入在同樣的環境下，理論上應該得到同樣的輸出。

但到了 AI 時代，LLM 的隨機性與機率特徵，把「證明 Bug 已經不存在」的哲學難題放大了無數倍。當你微調 Prompt 解決了某個特定偏見，在沒有絕對基準線的狀況下，下一次執行時它是否會以另一種形式復發？這讓測試人員對「修復」的定義產生了根本性的動搖。

## 三、QA 的哲學難題：證明「不存在」的不對稱性

為什麼驗證 Bug 消失這麼難？因為這在邏輯學上是一個「**證明全稱否定（Universal Negative）**」的認識論難題。

*   **證明 Bug 存在（存在量化）**：極其簡單。你只需要找到一個反例——一個 Crash 紀錄、一個跑版的截圖、或者一次 LLM 的胡言亂語。只要有一個反例存在，Bug 存在就是鐵錚錚的事實。
*   **證明 Bug 不存在（全稱否定）**：極其困難。你測試了 5 次、50 次沒有重現，並不代表第 51 次在特定的網路延遲、高併發負載或記憶體碎片化下不會再次發生。軟體的狀態空間是近乎無限的，你試圖用有限的測試次數，去證明一個無限空間裡的否定命題。

這就像身心科醫生無法用一次 10 分鐘的問診量表，就斷定病人的憂鬱症「永遠不會復發」一樣。

在實務中，這種「驗證難度」其實來自兩個截然不同的維度：「**認識論難**」與「**成本難**」。

*   **認識論難（Epistemological Difficulty）**：系統狀態空間不可窮盡，我們在理論上無法獲得 100% 的信心。例如偶發的 Race Condition 或 LLM 幻覺，你測不出 Bug，不代表它真的不存在。
*   **成本難（Cost Difficulty）**：Bug 在邏輯上是決定論的、可以被完全驗證，但**驗證的範圍遠遠大於修復的範圍**。

「修復範圍小，不代表驗證範圍小」，這是 QA 的時間與精力最常被開發團隊低估的隱形成本。

| Bug 類型 | 實務範例 | 可窮盡？ | 驗證成本 | 為什麼這麼難驗？ |
|---|---|---|---|---|
| **功能型** | 按鈕點擊後沒有跳轉 | 是 | 低 | 修復與驗證範圍一致，測過該路徑即可。 |
| **配置 / 環境型** | 修改 k8s 的一行 `config.yaml` 參數 | 是 | **高成本** | **修復只需 5 秒改一行；驗證卻需要跑過受該配置影響的整組微服務**。這就是典型的「成本難」。 |
| **整合 / 接縫型** | 兩個模組各自單測正確，接起來壞掉 | 部分 | 中 | 改動一側需要重新驗證兩者之間的所有互動邊界。 |
| **效能 / 負載型** | 系統在高併發或巨量資料下崩潰 | 否 | 高 | 必須在測試環境複製「規模與時間變數」才驗得到。 |
| **偶發 / Race Condition** | 多執行緒在特定時序下會鎖死 | 否 | 極高 | 測 100 次沒發生不代表沒有——典型的「認識論難」。 |
| **資安 / 邏輯漏洞** | 繞過驗證進行越權存取 | 否 | 極高 | 證明漏洞不存在，等於要對「所有惡意輸入」證明全稱否定。 |
| **LLM / 機率型** | 模型在特定語境下吐出偏見或幻覺 | 否 | 幾乎不可能 | 樣本空間無限，同一個 Prompt 執行兩次可能結果相異。 |

當開發人員修改了一個 Race Condition 的 Bug，對著 QA 說：「我改了那行程式碼，應該修好了，你測測看。」這句話背後隱含的，其實是將整個系統的「最終不確定性風險」，轉嫁到了測試人員的身上。

回頭看我之前在[〈太多「無意義」的回歸測試嗎？〉](/post/rethinking-regression-test-value/)裡討論的成本模型。在某些決定論的場景下，要花多少時間與測試案例才能詳細證明一個 Bug 已經完全不存在，數學上其實是「算得出來」的（例如，窮盡 N 個配置變數與連動模組的排列組合）。但現實是，我們的工作不是待在象牙塔裡研究理論的數學家，而是必須在有限時間與資源內交付成果的工程師。當我們算出「徹底證明 Bug 不存在」需要耗費數天，而 Sprint 的交付窗口只有兩小時的時候，工程師的職責就不是追求數學上的絕對嚴謹，而是要在時間限制下做出最務實的配置抉擇。

## 四、與不確定性共存的品質藝術

既然在哲學上我們無法證明一個 Bug 已經不存在，那身為品質守門員，我們難道只能陷入無盡的測試焦慮嗎？

我們可以再次向醫學界借鏡。

精神醫學在面對「無法輕易驗證痊癒」的困境時，並非放棄治療，而是推動「**測量導向照護（Measurement-Based Care, MBC）**」。他們不依賴單一量表，而是透過多維度的指標評估、劃分清晰的復發風險等級，並引導病人建立長期的支持系統。

這正是品質工程的本質：**測試的目的從來不是追求絕對的 100% 無瑕，而是建構一套「風險控制網」。**

這也呼應了軟體測試經典著作《Lessons Learned in Software Testing》中的第 185 條原則（Lesson 185）：**「『足夠的測試』意指『為我的客戶提供足夠的資訊，以利他們做出明智的決策』（'Enough testing' means 'enough information for my clients to make good decisions'）。」** 該書指出，測試本質上是一個蒐集資訊的過程，由於在現實中要抓出所有 Bug 需要無限的測試成本，因此「何時停止測試」的判斷標準，不在於是否達到了哲學上的完美，而在於我們是否已經為決策者（如 PM、業務或技術主管）提供了足夠的風險評估依據，讓他們有能力在不確定性中決定是否發布產品。

台灣測試社群前輩 David Ko（柯仁傑）在[〈什麼時候測試可以停止？〉](https://kojenchieh.pixnet.net/blog/posts/9075406966)一文中，同樣呼應了 Lesson 185 的精神。他將實務上常見的測試停止指標歸納為五大類：高優先級 Bug 皆已解決、Bug 發現率趨於收斂（若發現率仍高，即使當下無新 Bug 也意味著仍有潛在問題未被發掘）、預算耗盡、時程結束，以及專案風險已降至可接受的極限。David 強調，測試既然是資訊收集的過程，當你面臨「是否能結束測試」的猶豫時，本質上該問的是：『我是否已經充分掌握了專案的品質與風險狀況？』如果是，團隊便能依據這些客觀資訊，在時間與成本的邊界內做出停止測試的決策。

我們需要將「Bug 修好了」這件事重新 reframe：

> **「修好」並不是讓程式碼達到哲學上的完美無暇，而是將系統的不確定性風險，降到商業營運與用戶體驗可以接受的範圍。**

身心科醫生靠的是「療程覆蓋率」與「後續的出院追蹤」；而品質工程師靠的則是「**測試覆蓋率的配置**」與「**線上的可觀測性（Observability & Monitor）**」。

1.  **區分測試配置的本金**：對於可以規模化驗證的決定論 Bug，我們使用自動化回歸測試來兜底；對於不可規模化、依賴經驗的偶發 Bug，我們使用核心的「冒煙測試（Smoke Test）」在交付第一時間進行風險定錨。
2.  **建立線上的「負熵防線」**：承認測試環境的局限性，主動在線上環境部署監控與 Log 警報。當 Bug 在線上以極低的機率復發時，系統能夠第一時間「顯影」並優雅地降級，而不是讓災難無限放大。

## 結語：與其追求無瑕的幻覺，不如著手建立風險網

這就像 Passenger 在那首〈Holes〉裡所唱的：

> *"Where we've got holes, we've got holes / But we carry on"*
> （當我們身上有了洞，它就成了洞；但我們依然前行。）

QA 的工作是一輩子在替別人的系統尋找漏洞（Holes）。但做久了你會發現，不論是組織的流程、測試的環境，還是我們自己的職涯，都充滿了無法被完全填滿的洞。

與其追求「把所有洞補滿」這種不切實際的遠大夢想，身為工程師，我們更需要的是一個明確的下一步行動。**明天回到辦公室，你可以直接開始做這三件事，來重新配置你的品質守護網：**

### 1. 重新校準你的溝通腳本（溝通降噪）
當開發人員對你說：「我改了那行程式碼，應該修好了，你測測看。」
*   **過去的你**：默默埋頭重測所有相關路徑，最後說「我測過了，沒問題」。(把最終的不確定性風險獨自扛下)
*   **明天的你可以說**：*「我已經驗證了核心的使用路徑與關鍵邊界，這次修復在決定性路徑上是正常的。至於那些在測試環境下測不到的偶發性極限狀況，我們是否已經在線上建立了警報？如果發生了，我們有什麼復原機制？」*
將風險從「測試人員的保證」轉移為「團隊對不確定性的共識」。

### 2. 重新配置你的測試「本金」
盤點你手上的測試案例，不要再把時間平均分配給所有的 Bug。
*   對於**決定論、功能型**的 Bug（如按鈕沒跳轉）：編寫自動化回歸測試，將其定錨為你的安全底線。
*   對於**認識論難、隨機或機率型**的 Bug（如 Race Condition、LLM 幻覺）：果斷停止無止境的手動重試。只做一次基礎冒煙驗證，並將省下來的時間，拿去設計線上異常流量或錯誤日誌的顯影機制。

### 3. 動手建立你的第一道線上「顯影液」
不要等基礎建設團隊架設好完美的 Observability 平台才開始。
*   **明天就動手**：去日誌系統（如 Sentry, CloudWatch, Datadog）裡，為你剛測試完的這項功能，建立一個專屬的錯誤過濾器或警報。
*   設定一個簡單的 Slack 或是 Teams Webhook：當該功能的錯誤率在線上超過 1% 時，第一時間通知測試與開發團隊。這能讓原本可能潛伏數週的線上復發，在 5 分鐘內被偵測並迅速 Rollback。

我們無法消滅所有的黑盒子，也不可能寫出 100% 完美的測試。但當你不再試圖向所有人保證「這版絕對沒有 Bug」，而是能誠實且自信地宣告：*「我已經排除了致命風險，並且線上顯影與回滾機制都已準備就緒，能隨時在災情擴大前迅速反應」*——那一刻，你才真正跨出了從「執行測試的人」到「品質負責人（Quality Owner）」的關鍵一步。
