「能看懂代碼並抓出 Bug，為什麼不去做 RD？」——聊聊專職 SDET 的定位迷思與 AI 時代下的影子工程師

“We’ve got holes, we’ve got holes
In our hearts, in our lives
But we carry on.”
—— Passenger, 〈Holes〉(2012)

身為一個專注於軟體品質的工程師，我的日常工作就是為別人的系統找出各種「漏洞（Holes）」。但有時候，看著社群論壇與職場上的各種爭論，我會隱隱覺得：我們這些以「尋找漏洞」為生的人，自己的職涯與定位，是不是也正被時代與環境悄悄砸出一個洞？

最近在網路上看到一段相當有意思的討論，一位資深的 Java 開發者提到：

「測試跟 Code Review 應該是工程師之間交換做，不應該有 SDET（Software Development Engineer in Test）這種專門職位。因為你能看懂別人程式碼，還知道他哪裡寫錯，代表你的功力比原作者高深。既然你功力更高，為什麼不直接去做 RD，而要做一個專職的測試？」

這句話在社群上引發了熱烈討論。對於很多待過中小型團隊、或是沒有與現代化軟體測試工程協作過的開發者來說，這段邏輯聽起來合情合理。但對於在現代軟體工程體系中打滾過的工程師而言，這段話其實精準地暴露出兩個極大的認知盲區。

今天，我想聊聊關於 SDET 職能定位的迷思，並透過我們熟知的 SRE 角色進行類比。同時，身為一個無法直接改變公司組織架構的基層工程師，我們在 AI 變革下該如何看待自己的「職稱」，以及如何默默打造自己的職涯防禦率。

一、 大佬口中的「SDET 不需要論」：背後的雙重認知偏誤

這段認為「能找出 bug 代表功力更深，所以不需要專職 SDET」的論點，本質上犯了兩個盲區：

1. 將 SDET 的職能過度簡化

這句話把 SDET 的工作想像成「幫 RD 看代碼找錯」或「當專職的 Code Reviewer」。

如果一個團隊中的 SDET / QA 每天的工作只是坐在電腦前，等 RD 把代碼寫完，然後一行行去抓語法錯誤、或者做純人工的手動點擊測試，那確實由 RD 彼此交換做 Code Review 的效率會高得多。因為程式碼是 RD 自己寫的，脈絡最清楚。

但這就像認為「SRE（Site Reliability Engineer）的工作只是幫 RD 部署程式、或是寫幾行開機腳本」一樣，是極度狹隘的認知。

2. 混淆了「開發思維」與「品質思維」

「看得懂別人的程式碼，知道哪裡寫錯」跟「有能力設計出高覆蓋率、低維護成本的測試架構」，完全是兩碼子事。

  • RD 開發思維：關注的是**「如何實現功能」**。大腦處於建立邏輯、串接 API、讓 Happy Path 順利跑通的狀態。開發者很容易陷入路徑依賴（Path Dependency）的盲區——因為腦中預設了使用者會照著特定的流程操作，往往會忽略邊界外的異常。
  • QA/SDET 品質思維：關注的是**「系統在什麼邊界、極端輸入、惡意負載或非同步環境下會崩潰」**。

舉個生活中的例子：**大廚（RD）會煮出一桌精緻的法式料理，與食品衛生稽核員（QA）**知道如何建立一套系統化的流程來防範廚房食物中毒，是兩種完全不同的專業技術。稽核員不一定要比大廚更會煎牛排，但他必須知道在哪些溫度區間、哪些儲存條件下容易滋生細菌。

能看出大廚今天沒洗手（程式低級 Bug），不代表你能規劃出全系統的食安防線（測試策略）。

二、 SRE 的類比：從「打雜幫忙」到「建構測試基礎設施」

要理解 SDET 在團隊中的真實價值，我們可以用 SRE 來做個非常貼切的類比。

在十多年前 DevOps 概念還沒普及時，很多傳統開發者也會質疑：「為什麼要專職的 SRE 或運維工程師？部署跟架設伺服器我們 RD 自己寫腳本不就好了？」

但當系統規模擴大、流量成長、微服務架構興起時，RD 開始發現力不從心：每個人寫的部署腳本標準不一、監控指標亂七八糟、伺服器掛掉時沒人知道是誰的問題。

這時，SRE 的核心價值浮現了：SRE 從來不是「幫 RD 去部署程式的打雜工」，而是「建構一套安全、快速、自動化的部署平台與監控基礎設施，讓 RD 可以無痛地自我服務（Self-Service）」。

同理，SDET 的核心價值，是建構「測試基礎設施（Test Infrastructure）」與推動「工程效能（Developer Productivity Engineering, DPE）」。

維度 SRE (Site Reliability Engineer) SDET (Software Development Engineer in Test)
外界的誤解 幫 RD 手動部署、寫開機腳本 幫 RD 做 Code Review、手動點點測試
真實核心價值 建構運維與部署基礎設施 建構自動化測試與品質基礎設施
具體交付物 CI/CD 流水線、Kubernetes 集群、監控看板、容災架構 自動化測試框架封裝、測試資料生成平台、環境自動還原工具、測試成效看板
終極目標 讓 RD 能夠快速、安全且無痛地把程式發布上線 讓 RD 能夠快速、安全地獲取測試反饋，自己為品質把關

SDET 不是代替 RD 寫測試，而是為 RD 鋪路。他們開發工具、封裝框架，讓 RD 在寫完程式時，能在一分鐘內拿到可信賴的測試反饋，而不是每天花時間去調整不穩定的 UI Locator 或應付 Flaky Test。

這是一門專門的「軟體工程」，它服務的對象是開發者，解決的是整個研發管線的摩擦力。

三、 基層的現實無奈：當初分工的意義，與 AI 帶來的變革

看到這裡，你可能會嘆口氣：「你說的都對，但那是在大廠或現代化團隊。我們只是一般的基層工程師，公司根本沒有 SDET 這種職缺，HR 招人也是 RD 歸 RD、QA 歸 QA，我根本改不了公司的組織與職稱。」

這確實是實務上最無奈、也最真實的狀況。

我們不是管理層，很難單憑一己之力去推翻當前的分工體制。其實，組織當初會劃分「RD 寫功能、QA 點測試」的界線，有其歷史背景與效率考量——透過專門化的分工來降低個人的認知負荷，讓每個人專注在自己的防區。

然而，隨著 AI 時代的變革，這種極度僵化的分工防區正在面臨前所未有的挑戰：

  • AI 讓編碼成本暴跌：現在 RD 可以用 AI 在幾秒鐘內產出大量的程式碼與單元測試。
  • 溝通與等待成為瓶頸：如果 RD 寫出代碼後，依然需要「丟給 QA 手動測試 -> QA 提 bug -> RD 再修復」的傳統循環，這中間的跨職能溝通與等待成本，會遠大於 AI 幫你省下的時間。

身為非管理層的我們，如果只因為職稱是 QA 或 RD，就畫地自限，拒絕利用 AI 去理解底層代碼或建構測試工具，我們很快就會在這場高頻寬的開發革命中被邊緣化。

四、 影子工程師的逆襲：不破壞職稱的生存策略

既然我們無法改變名牌上的「職稱」，我們能改變的是自己為團隊解決問題的實質守備範圍。

在 AI 時代下，一個聰明的基層工程師，應該採取**「影子工程師（Shadow Engineer）」**的生存策略：

  1. 以 QA 之名，行 SDET/SRE 之實：
    不用等到公司設立 SDET 的職位。如果你發現每次回歸測試都很耗時、或者部署環境總是不穩定，你可以利用 AI 的輔助，悄悄跨出一步去寫自動化腳本、優化 CI 流水線、或是封裝共用的定位器邏輯。你不需要推翻體制，你只需要默默地用程式碼幫團隊降低摩擦力。
  2. 用「實質影響力」建立護城河：
    當你扮演了團隊中跨職能的溝通樞紐，解決了別人不想碰的「測試環境難用」或「資料準備繁瑣」的痛點，不論你的職稱叫 QA 還是 RD，你在平行同事（Peers）心中的實質地位，早已超出了那張職稱的名牌。
  3. 主動拓寬技術與業務的邊界：
    AI 降低了解讀複雜代碼的門檻，這給了測試工程師極大的紅利。我們不再受限於「黑箱測試」，而是能夠進一步參與系統設計的討論，從更上游去預防品質漏洞。

結語

“Sometimes it seems you gain less than you lose
But we carry on.”

在 AI 逐漸模糊所有技術界線的今天，市場與組織對「職稱」的定義或許會遲到、或許會產生陣痛與無奈，甚至有時會讓我們覺得，學了這麼多新東西，回頭看著組織的圍牆與考績天花板，好像 gain less than we lose。

但這首歌的最後給了一句很有韌性的回答：But we carry on。

當我們不再糾結於名牌上到底印的是 QA、SDET 還是 RD，而是把目光專注在「如何用更高效的工程手段去解決團隊的痛點」，這就是基層工程師在變革中最好的自我修練。我們默默鋪路、蓋好防禦網，然後帶著屬於自己的實質影響力，繼續前行。
