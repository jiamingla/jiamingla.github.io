你的團隊 Bug 預設給誰？業界六種主流 Bug 分派（Triage）模式對照

前言：你的團隊收到一個 bug，預設是誰的？

你的團隊收到一個新 bug 的時候，預設 assignee 是誰？

是 PM？Tech Lead？QA？還是「未分派」，丟在一個沒人認領的佇列裡？

你可能從來沒問過這個問題——它就是這樣，預設是繼承來的。前一個人怎麼做，你就怎麼做，沒人停下來討論過。

但這個悄悄繼承來的「預設」，靜悄悄地決定了三件事：

  • 你的 backlog 會長成什麼形狀
  • 誰會先 burn out
  • 哪些 bug 永遠不會被修

這篇文章想做的事很單純：讓你先認出自己的團隊在哪一種模式，看到它的代價，然後知道——原來還有別的選擇。

先補一個名詞：這篇講的 bug triage，中文常說「bug 分派」或「bug 分流」，指的是一個 bug 進來之後——「由誰來看、判斷它有多嚴重、決定交給誰處理」這一連串動作。底下為了行文，「分派」和「triage」會交替使用，講的是同一件事。

我待過的兩間公司，流程像得驚人

先講我自己的經驗，因為它正好是這篇文章的起點。

我待過兩間公司，bug 的流程結構像得驚人——都是同一套組合拳：

  1. 按模組分派：bug 會依它屬於哪一塊，轉給對應的模組 owner（這是後面會講的 F）
  2. 每週開會推動：固定一個會議，把卡住的、有爭議的 bug 攤開來決定（這是 E）。而且兩間都是 cross-functional team（PM、RD、QA 同在一個團隊），所以這個會不是跨部門的大委員會，而是同團隊不同職能坐下來談

我一度以為這就是「業界常態」——bug 本來就該按模組分、每週開會推啊，不然要怎樣？

直到我認真做了一輪 survey 才發現：這只是六種主流模式裡的一種組合（F + E），而六種各有各的代價。 我以為的「常態」，其實是一個有取捨的選擇——只是當時身在其中，看不出來。

這個發現本身就值得寫成一篇。下面是我整理出來的六種。

一、六種主流的預設指派模式

模式 預設給誰 典型場景 優點 代價
A. PM 預設 Product Manager B2C 消費型、product-led 組織 商業優先級判斷準 缺技術深度，severity 估不準、難 dedup
B. Tech Lead 預設 RD lead / 模組 owner 工程主導的 startup、infra team 修復難度估得準 商業權重失衡，使用者體驗的 bug 容易被低估
C. QA 預設 QA team / QA lead 金融、醫療、高合規 enterprise dedup 與 severity 校準一致 QA 變成 default janitor（預設清潔工）
D. 未分派佇列 任何人 pick up Open source、客服中心 負載自然平均 無主之地，bug 默默老化
E. Triage 委員會 每週開會決定 跨部門大委員會（微軟 / Google 經典）；也有 cross-functional team 內部的輕量版 ownership 顯式、爭議當場解 會議開銷大、bug 要等到下次會議
F. 模組 owner / CODEOWNER 程式區域的 owner 大型 codebase domain expertise 對位 跨模組的 bug 卡在邊界、互推

幾個值得展開的觀察：

A（PM 預設）不是壞模式，它只是有盲區。 product-led 的組織把 bug 丟給 PM 是合理的——PM 最知道哪個問題會影響轉換率、哪個可以晚點修。問題在於 PM 通常缺技術視角：兩個看起來不一樣的 bug 可能是同一個根因，PM 不一定看得出來；一個 severity 看起來不高的 bug，背後可能是會擴散的系統性問題。

C（QA 預設）常見於金融、醫療這類高合規環境。 它的好處很實在：QA 看過最多 bug，dedup 和 severity 校準的一致性最高。但它的代價也最隱形——QA 會從「驗證品質的人」慢慢變成「分派 bug 的人」。 時間被吃掉，而且因為做得很順，沒人覺得這是問題。這是後面〈如果你是那個被預設的 QA〉那節要回來談的。

D（未分派佇列）是最誠實也最危險的。 它不假裝有人負責——bug 就放在那，誰有空誰撿。負載很平均，但代價是：沒人撿的 bug 就會一直沒人撿。 它會老化、會被新的 bug 蓋過去，最後變成那個「我們知道但一直沒處理」的長長清單。

E 和 F 通常不是單獨存在，而是搭在別的模式上。 我待過的兩間，其實就是 F + E 的組合：按模組分派（F）+ 團隊內的跨職能會議推動（E 的輕量版——cross-functional team 內部，而不是跨部門的大委員會）。這也帶出一個重點——

真實世界很少有「純」模式。 大部分團隊都是兩三種的混合。所以與其問「我們是哪一種」，不如問「我們的混合，把成本壓在誰身上」。

二、七種「處理 backlog」的做法——全是 workaround

認出指派模式之後，還有第二個問題：已經積起來的 backlog 怎麼辦？

業界發展出至少七種做法。我把它們列出來，但先講結論：這七種沒有一個真正解決問題，它們全都是 workaround。

做法 邏輯 局限
Bug bash 釋出前全團隊集中獵 bug，有時帶獎勵 是釋出前的 surge capacity，不解既有 backlog
Bug bankruptcy 定期把超過 N 天未處理的 bug 全部 close 真 bug 被一起殺掉，製造「乾淨了」的假信號
Zero bug policy bug 一進來 24–48 小時內必須 triage 需要有人專職盯，成本高
只看 severity 只處理 S1 / S2，S3 / S4 永遠在 backlog paper cuts 累積，體驗一點一點折損
Fix-it Friday / 修 bug 衝刺 每隔幾個 sprint 撥時間清 backlog 治標，不改變 bug 流進來的流程
SLA 分級 客戶報的有回應 SLA，內部報的沒有 二級制度，內部發現的 bug 被餓死
Embedded QA / shift-left QA 跟開發團隊同坐，開發中即時 triage 治本，但需要組織結構改造

這裡面我想特別點一個 bug bankruptcy 的隱形代價——它的痛不只是「假信號」這麼技術性。

決定不修「一個」bug，對 PM 來說很容易下手：權衡一下，這個影響不大，不修。但一次宣告幾百個 bug bankruptcy，對 QA 是完全不同的一回事。那幾百個，是你一個一個記錄、重現、追蹤、寫清楚重現步驟的東西。它們一次被清空，會有一種很真實的挫敗感——不是「資料被刪了」的可惜，而是「我做這些到底有沒有意義」的動搖。

所以 bug bankruptcy 的「假信號」是雙向的：它對上騙了管理層「我們乾淨了」，對下則悄悄消耗了 QA 對「記錄 bug 這件事本身」的信任。一個會反覆宣告破產的團隊，最後會養出「反正記了也會被清，那就別太認真記」的心態——這比那幾百個 bug 本身更貴。

看出共同點了嗎？除了最後一個（embedded QA）真的動到流程，前六種都是在「事情已經爆了之後」想辦法收拾。 業界到今天，對「backlog 該怎麼結構性治理」其實沒有共識。

三、為什麼沒有銀彈？

如果六種模式並存、七種做法都是 workaround，那代表一件事：這個問題沒有銀彈。 為什麼？

因為一個 bug backlog，本質上是四種角色、四種語言混在一起的語料：

  • 客服寫的是情緒和體驗：「使用者很生氣，說 app 一直閃退」
  • RD 寫的是 stack trace 和技術細節
  • PM 寫的是影響範圍和商業權重
  • QA 寫的是重現步驟和測試紀錄

很少有人能同時把這四種語言都讀得流利。 而每一種指派模式，都會在它最不擅長的那種語言上失手：

  • A（PM）讀不流利技術語言
  • B（Tech Lead）讀不流利客戶語言
  • C（QA）讀得最全，但被當成翻譯機累垮
  • D（未分派）乾脆沒人讀

所以你換模式，只是把「失手的位置」從一個角色移到另一個角色。真正的瓶頸不是「派給誰」，而是「沒有人天生能讀懂全部四種語言」。

這也是為什麼我越來越相信：這個問題的結構性出路，可能不在「換一個更好的人來分派」，而在「換一種能幫忙翻譯這四種語言的工具」。但那是另一篇的主題了。

四、如果你是那個被預設的 QA

如果你看完發現——你的團隊就是 C，而且那個 default janitor 就是你，那這節是給你的。

你大概有三條路：

  1. 抵抗：把自己的角色重新講清楚——「我是驗證品質的人，不是 bug 的預設收件匣」。這條最難，因為它需要主管的支持，而且要對抗一個「一直都這樣」的慣性。
  2. 改革：推動模式切換，比如把純 C 變成 C + E（加入委員會分擔決策）。中型團隊有機會，大公司很難。
  3. 妥協 + 自救：接受這個角色，但想辦法用工具減負——把重複的 dedup、分類、初步 severity 判斷，交給能幫你讀那四種語言的工具。

沒有哪條路是「正確答案」。但有一件事是確定的：先認出你在哪個模式，你才有得選。 不認出來，你只會覺得「我就是比較忙」「我就是比較負責」——把一個組織設計的問題，誤當成自己的個人問題。

結語

業界沒有一個 bug triage 模式是完美的。六種並存、七種 workaround，本身就是證據。

但選擇哪一種模式，從來不是隨機的——它反映的是你的組織用什麼語言在思考品質。 把 bug 預設給 PM 的組織，用商業語言思考；預設給 QA 的組織，用驗證語言思考；放著不分派的組織，可能根本還沒開始思考。

你不一定能馬上改變你的團隊在哪個模式。但光是知道自己在哪，就已經能做一件很重要的事：

停止把組織的問題，當成自己的問題。
